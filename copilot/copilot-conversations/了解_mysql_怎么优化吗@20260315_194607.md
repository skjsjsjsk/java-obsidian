**ai**: 根据当前笔记 [[MySQL]] 的内容，MySQL 优化主要可以从以下几个维度进行：

### 1. 索引优化
这是优化的核心：
- **建立合适的索引**：
    - 使用**联合索引**覆盖多个查询条件，遵循**最左前缀原则**。
    - 利用**覆盖索引**（索引包含了查询需要的所有字段），避免**回表**操作，减少 I/O 开销。
    - 适度进行**冗余设计**（反范式），将高频查询的字段全部包含在索引中。
- **避免索引失效**：
    - 避免 `!=` 操作符。
    - 避免在 `WHERE` 子句中对列进行函数运算，否则会导致索引失效。

### 2. SQL 语句与查询优化
- **避免使用 `SELECT *`**：只查询需要的字段，减少网络传输和内存消耗。
- **深分页优化**：
    - **延迟关联**：先在索引树快速定位到主键 ID，再根据 ID 去关联查询完整数据。
    - **书签法**：记录上一页最后一条 ID 的值，查询时使用 `WHERE id > last_id LIMIT 10`。

### 3. 分析工具的使用
通过工具定位慢 SQL：
- **开启慢查询日志**：捕获执行时间超过 `long_query_time` 的 SQL。
- **使用 `show processlist`**：查看当前正在执行的线程，找到执行时间长的 SQL。
- **使用 `EXPLAIN`**：分析 SQL 的执行计划，重点关注以下字段：
    - **type**：访问类型，效率从高到低为 `system > const > eq_ref > ref > range > index > ALL`。一般达到 `ref`、`eq_ref` 或 `range` 即可。
    - **key**：实际使用的索引。
    - **rows**：预估扫描的行数。
    - **Extra**：
        - 出现 `Using filesort`：说明无法利用索引顺序，需额外排序。
        - 出现 `Using temporary`：使用了临时表。
        - 这两种情况通常意味着需要优化 SQL 或索引。

在 InnoDB 的 **RR（可重复读）** 隔离级别下，正是通过这两种机制的配合，在很大程度上解决了幻读问题。具体原理如下：

### 1. MVCC 解决快照读的幻读
- **机制**：MVCC（多版本并发控制）利用 **Read View（读视图）** 和 **Undo Log** 来实现。
- **RR 级别的特性**：在 RR 隔离级别下，事务内**第一次**执行快照读（普通 `SELECT`）时会生成一个 Read View，并在整个事务期间复用这个 Read View。
- **原理**：由于 Read View 始终保持一致，后续的查询只能看到 Read View 生成时已经提交的数据版本，看不到之后其他事务插入的新数据。因此，避免了快照读情况下的幻读。
- *结合你的笔记*：笔记中提到 `undo log: 回滚和MVCC`，Undo Log 中存储了数据的历史版本，配合 Read View 实现了版本链的读取。

### 2. Next-Key Lock 解决当前读的幻读
- **机制**：当前读（`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`）读取的是最新数据，需要加锁。
- **Next-Key Lock**：它是由 **Record Lock（行锁）** 和 **Gap Lock（间隙锁）** 组成的。
    - **Record Lock**：锁住索引记录本身。
    - **Gap Lock**：锁住索引记录之间的间隙（不包含记录本身），或者是第一条记录之前、最后一条记录之后的间隙。
- **原理**：通过锁住间隙，其他事务无法在这个范围内插入新记录。即使你想插入的数据在物理上不存在，也会被 Gap Lock 阻挡。因此，避免了当前读情况下的幻读。

### 总结
- **快照读**：依靠 **MVCC**（一致性读视图）。
- **当前读**：依靠 **Next-Key Lock**（临键锁，包含间隙锁）。

这套组合拳使得 InnoDB 的 RR 级别能够很好地解决幻读问题，这也是它比 RC 级别更强一致性的原因。
