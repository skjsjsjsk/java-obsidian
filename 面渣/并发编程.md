- 进程和线程的区别: 进程是操作系统中分配资源的最小单位;  线程是进程中的独立单元,多个线程共享一个进程的资源(如内存),线程有自己独立的栈和寄存器
- 线程的创建方式:  1. 继承Thread类,重写run()方法,  2. 实现Runnable接口,重新run()方法,  3. 实现Callable接口, 重写call()方法 (这个可以获取到返回值)  , 然后创建 FutureTask 对象包装 Callable 实现类，再将 FutureTask 传给 Thread 对象启动
- 调⽤ start ⽅法时会执⾏ run ⽅法，那怎么不直接调⽤ run⽅法: 1. 直接执行run方法是同步执行当前代码,不创建新线程, 达不到多线程并发的目的  2. 先调用start方法会通知JVM底层的线程调度机制创建一个新的线程, 然后线程就绪, 等待调度执行,执行时执行run()方法中的代码(自动调用run方法)
- 线程的状态: 6种:  NEW(新建),  RUNNABLE(就绪),  BLOCKED(等待获取锁),  WATIING(等待通知),  TIMED-WAITING(超时等待(eg: sleep()之后就是此状态)),  TERMINATED(终止(执行完毕)) 
- sleep和wait的区别: 
	- sleep属于Thread类, wait属于Object类;  
	- sleep使用后不释放锁,但是wait调用后释放锁;  
	- sleep可以在任何地方被调用,让线程从RUNNABLE状态变成TIMED-WAITING状态,  wait只能在同步代码块中调用, 因为它调用的前提是该线程有锁, 它会让该线程从RUNNABLE变成WAITIGN;  
	- sleep会在睡眠时间到达后自动苏醒, wait需要使用`notify/ notifyAll` 来唤醒
- 线程安全实现: 
	- 悲观锁(互斥同步):`synchronized`关键字加锁和`ReentrantLock`类(通过调用`lock()`和`unlock()`)
	- 乐观锁(非阻塞同步):CAS(比较并交换: 要更新的值:V(现在的值), 预期值: A(一开始的值), 新值: B, 如果V=A, 则A更新为B, 否则不变, 因为是多线程, 一开始, V=5, A=5, B=6, 线程1要将V修改为6, 此时V=A, 那么V=6, 然后线程2也要修改V=7, 此时, V=6, A=5, B=7, 但是V != A, 所以不变)和原子类(通过CAS + `volatile`关键字实现)
	- 无同步方案(不使用锁): 不可变的对象(string, final修饰的等等) 和 `ThreadLocal` 线程隔离
	- 并发工具类:ConcurrentHashMap(CAS + synchronized) 和 CopyOnWriteArrayList(写时复制)

	- 多线程环境下 ++ 操作不是线程安全的,因为它没有原子性, ++ 有三步 1. 读取变量的值  2. 对该值 +1 操作, 3. 将新值赋给原值,  这就会导致有两个或多个线程同时读取到该值,然后进行 +1 操作
- ThreadLocal: 
	- 是什么: **ThreadLocal 是一个用于创建线程局部变量的类**。它通过为每个使用该变量的线程提供一个 **独立的变量副本**，从而实现了**线程隔离**。也就是说，每个线程只能访问自己 ThreadLocal 中的值，互不干扰。
	- 如何实现的: 并不是`ThreadLocal` 内部维护了一个 Map，键是线程，值是数据。而是 `Thread` 类内部维护了一个 `ThreadLocalMap`。
		- `Thread` 类中有一个成员变量 `threadLocals`（类型是 `ThreadLocalMap`）。
		- `ThreadLocalMap` 的 **Key** 是当前的 `ThreadLocal` 对象实例。
		- `ThreadLocalMap` 的 **Value** 是我们调用 `set()` 方法存入的值。
	- 调用 `userContext.get()`方法时:
		1. 获取当前线程 `Thread.currentThread()`。
		2. 拿到当前线程内部的 `threadLocals` Map (ThreadLocalMap)。
		3. 以 `userContext` (ThreadLocal实例) 为 Key，从 Map 中取出 Value。
	
	- 内存泄漏: Key是弱引用，Value是强引用, 当外部不在持有对ThreadLocal的引用时,key(弱引用被回收)会被GC回收,设置为null,而value因为它是强引用不变,如果该线程池运行时间过长,会导致这种情况越来越多,占用内存,导致OOM
		- 为什么value是强引用: Entry 对象本身还在 `ThreadLocalMap` 的数组里，而 Entry 里的 Value 依然是**强引用**。同时引用链未断, 引用链依然完整：`Thread` -> `ThreadLocalMap` -> `Entry` -> `Value`。
		- 如何解决: 调用`remove()`方法, 手动将 Value 置为 `null`，断开强引用链
	- 应用场景: 
		- 在 Web 应⽤中，可以使⽤ ThreadLocal 存储⽤户会话信息，这样每个线程在处理⽤户请求时都能⽅便地访问当前⽤户的会话信息。  
		- 在数据库操作中，可以使⽤ ThreadLocal 存储数据库连接对象，每个线程有⾃⼰独⽴的数据库连接，从⽽避免了多线程竞争同⼀数据库连接的问题。  
		- 在格式化操作中，例如⽇期格式化，可以使⽤ ThreadLocal 存储 SimpleDateFormat (非线程安全的)实例，避免多线程共享同⼀实例导致的线程安全问题
- 什么是JMM(Java内存模型): 是JVM规范中的一个抽象模型, 用来描述多线程环境里面共享变量的可见性, 共享变量存储在主内存中, 各个线程有自己的私人本地内存, 存储主内存中共享变量的副本, 当线程修改本地副本后, 将通过**Store/Write** 刷新至主内存, 其他线程下要读取时通过 **Read/Load** 同步最新值从主内存
- volatile 原理: 保证多线程环境下的可见性与有序性, 不能保证原子性
	- 可见性: 
		- 当线程对 volatile 变量进⾏写操作时，JVM 会在这个变量写⼊之后插⼊⼀个**StoreLoad**屏障指令，这个指令会强制将本地内存中的变量值刷新到主内存中       
		- 当线程读取 volatile 变量时，JVM 会插入读屏障。它会强制将自己的工作内存缓存失效，重新从主内存读取。
	- 有序性(禁止指令重排): 
		- **StoreStore 屏障**：保证在 volatile 写**之前**的普通写操作，一定先于 volatile 写完成（禁止“写写”重排）。
		- **StoreLoad 屏障**：禁止 volatile 写操作与它**之后**的任何读写操作重排（这是开销最大的屏障，因为它会刷新缓存）。
		- **LoadLoad 屏障**：禁止 volatile 读与它**之后**的普通读操作重排。
		- **LoadStore 屏障**：禁止 volatile 读与它**之后**的普通写操作重排。
	- 原子性(不保证): 例如 `count++` 操作，它实际上包含三步：`读取 -> 修改 -> 写回`。即使 volatile 保证了读取和写回的可见性，但两个线程同时读取到旧值 0，分别加 1 后写回 1，最终结果还是 1，而不是预期的 2。这种**竞态条件**必须由 `synchronized` 或 `AtomicInteger` 来解决。
- synchronized 的锁升级: 
	- 无锁到偏向锁: 
		- 当一个线程首次访问代码块时, 如果此时对象处于无锁并且偏向锁未被禁用的状态, JVM 会将该对象头的锁标记改为偏向锁状态, 并修改对象头的 Mark Word 为当前线程 ID 
		- 如果另一个线程尝试获取该偏向锁, 会检查获取该偏向锁的线程是否活跃, 如果不活跃可以把该锁给它, 否则撤销偏向锁, 升级为轻量级锁
	- 偏向锁到轻量级锁: 
		- 线程会在自己的堆栈中创建一个叫 **Lock Record（锁记录）** 的空间。该线程会尝试使用CAS来将对象的Mark Word 替换为指向自己 Lock Record 的指针。成功则获取锁, 否则一直自旋
	- 轻量级锁到重量级锁: 
		- 当自旋多次失败，或者有线程在等待队列中等待相同的轻量级锁时，轻量级锁会升级为重量级锁。
		- 此时JVM将在操作系统层面创建一个互斥锁---Mutex
		- 然后所有尝试获取锁的线程都将加入阻塞队列, 知道锁释放
- synchronized 和 ReentrantLock 的区别了解吗: 
	- synchronized 由 JVM 内部的 Monitor 机制实现; 而ReentrantLock 基于 AQS 实现
	- synchronized 可以自动加锁和解锁，ReentrantLock 需要手动 `lock()` 和 `unlock()`。
	- ReentrantLock还支持可中断锁, 公平锁等等, 还能通过lockInterruptibly响应中断, synchronized做不到
- ReentrantLock实现: 
	- 基于AQS实现
	- AQS 维护了一个 `volatile int state` 变量。当 `state` 从0变成1时, 该线程成功获取锁, 当线程重入时, 只会增加state的值, 这就是为什么一个线程多次调用`lock()`不会死锁, 释放锁时，`state` 减 1，直到减为 0 才表示锁完全释放。
	- 线程尝试获取锁时, 会进行CAS尝试将state从0到1
	- CLH队列(双向列表): 当获取锁失败时, 该线程会被包装成一个Node节点 进入一个双向链表(等待队列)尾部阻塞挂起, 当持有锁的线程释放锁（`state` 减为 0）时，才会唤醒队列中的下一个节点。
	- 默认为非公平锁, 通过在构造函数传入`true` 开启公平锁, 公平锁会在获取锁前会调用 `hasQueuedPredecessors()` 检查是否有前驱节点
- CAS: 
	- 原理: CAS(比较并交换: 要更新的值:V(现在的值), 预期值: A(一开始的值), 新值: B, 如果V=A, 则A更新为B, 否则不变, 因为是多线程, 一开始, V=5, A=5, B=6, 线程1要将V修改为6, 此时V=A, 那么V=6, 然后线程2也要修改V=7, 此时, V=6, A=5, B=7, 但是V != A, 所以不变) 
	- 问题: ABA使⽤版本号(`AtomicStampedReference`)/时间戳的⽅式来解决 ABA 问题
- 死锁: 
	- 四个条件: 
		1. 互斥: 资源不能共享
		2. 持有并等待：持有资源同时等待其他资源
		3. 不可抢占：资源不能被强制释放
		4. 循环等待: 线程直接形成闭环, 线程 A 等待线程 B 持有的资源，线程 B 等待线程 C 持有的资源, 直到线程 N ⼜等待线程 A 持有的资源。
	- 如何解决: 
		1. 按固定顺序申请资源
		2. 无法获取资源时先释放已持有资源再重试
		3. 编码时使用`ReentrantLock`的`tryLock()`代替`lock()`设置超时
- 悲观锁和乐观锁: 
	- 悲观锁: 悲观锁认为每次访问资源都会发生冲突, 因此会加锁在操作前; 有`synchronized` 或 `ReentrantLock`; 冲突发生, 未获得锁的线程会阻塞挂起, 导致上下文切换
	- 乐观锁: 乐观锁认为冲突不总会发生, 因此操作前不加锁, 会在更新数据时检查数据是否发生变化, 发生了则重试, 有**CAS**; 冲突发生, 线程未获得锁会不断自旋重试
- ConcurrentHashMap: 
	- 实现原理: 
		- JDK7使用分段锁, 中它将数据分成多个Segment（段）, 每个段继承ReentrantLock独立加锁, 默认并发度是16
		- JDK8使用CAS + synchronized来保证线程安全, 优先使用CAS无锁操作插入, 发生哈希冲突则使用synchronized锁住桶的头节点, 当链表长度 > 8 且数组长度 > 64 时，转换为红黑树，查询复杂度从 O(n) 降至 O(log n)。读操作完全无锁, 因为value使用volatile修饰, 保证永远读取到最新值
- 线程池: 
	- 什么是线程池: 就是用来管理和复用线程的工具, 减少线程的创建与销毁的开销.   ThreadPoolExecutor是线程池的核心实现, 它通过核心线程数, 最大线程数, 任务队列(阻塞队列)和拒绝策略来控制线程
	- 工作流程:
		1. 如果当前线程数 **小于** corePoolSize, 就创建核心线程来执行任务
		2. 当前线程数 **大于等于** corePoolSize, 将任务放到任务队列中等待
		3. 任务队列已满并且当前线程数 **小于等于** maxmumPoolSize时创建非核心线程来执行任务
		4. 任务队列已满并且当前线程数 **大于等于** maxmumPoolSize时执行拒绝策略
		5. 当非核心线程空闲时间 **大于** keepAliveTime, 就会被回收, 知道剩下corePoolSize个线程
	- 关于非核心线程: 只有当核心线程都在忙，且**任务队列也满了**之后，线程池才会创建非核心线程来救急.
	- 线程池的参数: corePoolSize(核心线程数), maxmumPoolSize(最大线程数), keepAliveTime(非黑犀牛线程的空闲存活时间), workQueue(任务队列), unit(存活时间的单位), threadFactory(线程工厂(用来创建线程的)), handler(拒绝策略)
	- 拒绝策略有哪些: 
		1. `AbortPolicy`（默认）：**直接抛异常**（`RejectedExecutionException`），阻止系统正常运行。
		2. `CallerRunsPolicy`：**哪里来回哪里去**。由调用者线程（提交任务的线程）自己去执行该任务。这种策略会降低提交速度，从而给线程池喘息的机会。
		3. `DiscardPolicy`：**直接丢弃**任务，不做任何处理，不抛异常。风险是任务丢失。
		4. `DiscardOldestPolicy`：**丢弃队列里最老的一个任务**，然后尝试再次提交当前任务。

	- 一般情况有以下两种:
		- **CPU 密集型任务**：`N + 1`（N 为 CPU 核心数）。尽量减少线程上下文切换。

- ThreadPoolExecutor里的execute()方法, 任务提交后是如何工作的: 任务提交后走三步判断
	- 首先看当前的工作线程数是否小于核心线程数, 如果小于, 那么就创建一个新线程执行任务, 不走队列
	- 第二步, 核心线程数满了, 新任务就会被放到任务队列里面, 等核心线程空闲了来取
	- 第三步, 如果任务队列也满了, 那么就创建非核心线程来执行任务, 直到到达最大线程数, 到了最大线程数后, 再有新任务就会触发拒绝策略
		- **其中在任务入队列之后有一个双重检查**, 先看如果线程池在这个间隙被关闭了, 那么任务移除并拒绝. 然后是如果线程池正常但是工作线程数为0, 那么就会创建一个非核心线程来消费队列. 
- 核心线程的创建时机: 默认是懒加载创建的, 在线程池刚被new出来的时候, 里面没有线程, 当任务进来, 就会创建一个新线程去执行, 直到到达核心线程数.     但是这样的并不适用于在应用刚启动就有大量请求的场景, 这样会到前面请求时间偏高.     ThreadPoolExecutor提供了一个方法(`prestartAllCoreThreads()`), 这个方法会在应用**启动阶段**预创建所有的核心线程(通过循环调用addWorker(null, true)，传null表示线程没有初始任务，创建出来直接去队列取活)

- 对于ThreadPoolExecutor, 它的默认的执行顺序是：核心线程 → 队列 → 非核心线程. 这样使得在Web应用场景下不太合理, 为了解决这个问题, 我们可以自定义一个队列TaskQueue, 继承LinkedBlockingQueue，重写offer()方法.  (在offer()方法里面让当前线程数还没到达最大值时返回false, 这样在execute()方法里workQueue.offer()返回false就会调addWorker(command, false)创建非核心线程)
