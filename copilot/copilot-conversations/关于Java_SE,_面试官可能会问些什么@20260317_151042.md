**ai**: 好的，聚焦在 **Java 基础** 部分，这是面试的敲门砖，通常也是面试刚开始的热身环节。虽然看似简单，但面试官经常会通过追问底层原理（比如 `String` 的不可变原理、自动拆装箱的实现）来考察你的基础扎实程度。

以下是为 `[[Java SE]]` 笔记准备的“Java 基础”详细考点大纲：

### 1. 基本数据类型与包装类
- **八种基本类型**
  - 熟记：`byte` (1字节), `short` (2字节), `int` (4字节), `long` (8字节), `float` (4字节), `double` (8字节), `char` (2字节), `boolean` (1字节)。
  - `int` 的取值范围是多少？（$-2^{31}$ 到 $2^{31}-1$）
- **基本类型 vs 包装类**
  - **区别**：包装类是对象，存在于堆中；基本类型是值，存在于栈中。
  - **自动拆装箱**：Java 5 引入，编译器自动调用 `Integer.valueOf()` 和 `intValue()`。
  - **面试陷阱**：
    ```java
    Integer a = 100;
    Integer b = 100;
    Integer c = 200;
    Integer d = 200;
    System.out.println(a == b); // true
    System.out.println(c == d); // false
    ```
  - **解析**：`Integer` 缓存机制默认缓存 **-128 到 127** 之间的对象。超出该范围会创建新对象。`Byte`, `Short`, `Long` 也有缓存，`Float`, `Double` 没有。

### 2. String 类 - **必问**
- **不可变性**
  - 为什么不可变？（`final` 修饰类，`char[]` (JDK 9 改为 `byte[]`) 数组私有）。
  - 不可变的好处？（字符串常量池安全、线程安全、hashCode 缓存）。
- **字符串常量池**
  - 存放位置？JDK 1.7 在永久代，1.7+ 移到了堆中。
  - `String s = "a"` vs `String s = new String("a")` 的区别？（前者直接去池找，后者先在堆创建对象，再去池找）。
  - `intern()` 方法的作用？（将字符串放入常量池并返回引用）。
- **拼接操作**
  - `String` vs `StringBuilder` vs `StringBuffer`。
    - `String`：不可变，每次拼接都创建新对象，效率低。
    - `StringBuilder`：可变，线程不安全，效率高。
    - `StringBuffer`：可变，线程安全（方法加 `synchronized`），效率略低。
  - 编译期优化：`String s = "a" + "b"` 在编译阶段会直接优化为 `"ab"`。

### 3. 面向对象特性 (OOP)
- **三大特征**
  - **封装**：隐藏属性，提供 `public` 方法访问。
  - **继承**：子类拥有父类非 `private` 的属性和方法。注意：Java 是单继承。
  - **多态**：同一个行为具有多个不同表现形式。必要条件：继承、重写、父类引用指向子类对象。
- **重写 vs 重载**
  - **重载**：同一个类中，方法名相同，参数列表不同（个数、类型、顺序）。与返回值无关。
  - **重写**：子类重新定义父类的方法。方法名、参数列表必须相同，访问权限不能更严，返回值类型可以是父类返回值的子类型（协变返回）。
- **抽象类 vs 接口**
  - **抽象类**：单继承，体现 "is-a" 关系，可以有构造方法、普通方法、各种成员变量。
  - **接口**：多实现，体现 "has-a" 或 "can-do" 能力。JDK 8 前全是抽象方法；JDK 8 支持 `default` 和 `static` 方法；JDK 9 支持 `private` 方法。
  - **设计原则**：优先使用接口，尽量减少抽象类的使用。

### 4. 关键字
- **`static`**
  - 修饰变量（类变量）、方法（类方法）、代码块（静态代码块，类加载时执行）、内部类（静态内部类）。
  - **注意**：`static` 方法中不能使用 `this` 或 `super`，只能访问 `static` 成员。
- **`final`**
  - 修饰类：不能被继承（如 `String`, `Integer`）。
  - 修饰方法：不能被重写。
  - 修饰变量：基本类型值不能改，引用类型引用地址不能改。
- **`this` vs `super`**
  - `this`：当前对象的引用。
  - `super`：父类对象的引用。
  - 子类构造器默认会调用父类的无参构造器（`super()`）。
- **`transient`**
  - 修饰变量，该变量不参与序列化过程。

### 5. Object 类通用方法
- **`equals()` vs `==`**
  - `==`：基本类型比数值，引用类型比内存地址。
  - `equals()`：默认也是比地址（即 `==`），但通常被重写（如 `String`）来比内容。
  - **约定**：重写 `equals` 时必须重写 `hashCode()`。
- **`hashCode()`**
  - 作用：为了在哈希表（HashMap, HashSet）中快速定位对象。
  - **规则**：两个对象 `equals` 返回 `true`，则 `hashCode` 必须相同；反之不一定。
- **`clone()`**
  - **浅拷贝**：复制引用，修改拷贝对象会影响原对象。
  - **深拷贝**：复制对象及引用的对象（需要重写 `clone` 方法或使用序列化）。

### 6. 异常处理
- **体系结构**
  - `Throwable` -> `Error` (严重错误，如 OOM) / `Exception`。
  - `Exception` -> `Checked Exception` (编译时异常，必须处理) / `RuntimeException` (运行时异常，不强制处理)。
- **关键点**
  - `try-catch-finally` 执行顺序：`finally` 无论如何都会执行（除非退出 JVM）。
  - `try` 块里有 `return`，也会先执行 `finally`。
  - `try-with-resources` (JDK 7)：自动关闭资源（实现了 `AutoCloseable` 接口）。

### 7. Java 核心 API
- **反射**
  - 功能：在运行时动态获取类信息、创建对象、调用方法。
  - 应用：框架开发（Spring 的 IOC）、动态代理。
  - 缺点：破坏封装性、性能开销。
- **泛型**
  - 本质：**类型擦除**。Java 泛型只在编译期有效，运行期会被擦除为 `Object` 或边界类型。
  - `List<String>` 和 `List<Integer>` 在运行时是同一个类型（`List`）。
  - **PECS 原则**：生产者使用 `extends`，消费者使用 `super`。
- **序列化**
  - `Serializable` 接口：只是一个标记接口。
  - `serialVersionUID`：版本号，反序列化时如果 UID 不一致会报错。

**ai**: 这是一个非常经典的 Java IO 面试题。面试官问这个问题，主要考察你对 **IO 模型**、**多路复用** 以及 **操作系统底层原理** 的理解。

建议按照 **“概念 -> 机制对比 -> 场景 -> 经典比喻”** 的逻辑来回答，这样条理清晰且显得有深度。

---

### 🗣️ 标准回答逻辑

#### 1. 开篇总述（一句话总结）
> BIO 是同步阻塞，NIO 是同步非阻塞，AIO 是异步非阻塞。Java 中的 IO 模型随着版本的迭代，主要是为了解决高并发场景下线程资源消耗过大的问题。

#### 2. 逐一解释

**BIO (Blocking I/O) - 同步阻塞 IO**
- **机制**：传统的 Java IO (`java.io`)。服务器实现模式为**“一连接一线程”**。
- **流程**：当客户端发起连接请求时，服务器端必须启动一个线程进行处理。如果这个连接不做任何事情（比如暂时没有数据传输），这个线程也会被阻塞，无法干别的事。
- **特点**：代码简单，但并发能力极其有限。线程是昂贵的资源，几千个连接就能拖垮服务器。

**NIO (Non-blocking I/O) - 同步非阻塞 IO**
- **机制**：Java 1.4 引入 (`java.nio`)。服务器实现模式为**“一请求一线程”**（实际上是基于 Reactor 模式）。
- **核心组件**：
  - **Channel (通道)**：双向，相当于流。
  - **Buffer (缓冲区)**：数据读入 buffer 再处理。
  - **Selector (选择器)**：**这是核心**。单线程使用 Selector 可以轮询多个 Channel，只有当 Channel 真正有读写事件发生时，才分配线程去处理。
- **流程**：线程发起读写请求时，如果没数据，直接返回，不会阻塞线程。线程可以通过 Selector 去查询哪些 IO 准备好了。
- **特点**：适用于高并发、连接多但读写短的场景。一个线程可以管理成千上万个连接。

**AIO (Asynchronous I/O) - 异步非阻塞 IO**
- **机制**：Java 1.7 引入 (NIO.2)。服务器实现模式为**“一个有效请求一个线程”**。
- **流程**：线程发起读写请求后，**立刻返回**，去做别的事。**操作系统**完成 IO 操作后，会回调通知应用程序（通过回调函数 `CompletionHandler`）。
- **特点**：真正的异步 IO，不需要应用程序去轮询。适用于连接数特别多且连接持续时间很长的场景（如相册服务器）。

---

### 🧪 核心区别总结表（建议背下来）

| 特性 | BIO | NIO | AIO |
| :--- | :--- | :--- | :--- |
| **IO模型** | 同步阻塞 | 同步非阻塞 | 异步非阻塞 |
| **线程对应关系** | 一连接一线程 | 一线程多连接（多路复用） | 有效请求一线程 |
| **谁来做IO** | 程序自己读写 | 程序轮询读写 | 操作系统回调通知 |
| **核心组件** | Stream (流) | Channel, Buffer, Selector | Channel, Future/CompletionHandler |
| **代码复杂度** | 简单 | 复杂 | 较复杂 |
| **适用场景** | 连接数少、架构固定 | 高并发、高性能 | 高并发、长连接 |

### 🍽️ 经典的“餐厅服务员”比喻（必杀技）

如果面试官让你**形象化地描述**，用这个比喻非常加分：

*   **BIO (阻塞)**：
  你去饭馆吃饭，招手叫来一个服务员。你点菜，服务员就在那里站着等你选好、点完。
  在你点菜期间，**这个服务员被你一个人占用了**，他不能去服务别的客人。如果很多客人同时来，饭馆就需要雇很多服务员（线程爆炸）。

*   **NIO (同步非阻塞/多路复用)**：
  你去饭馆吃饭，招手叫来服务员。你点菜，服务员给你一张菜单就走了（**不阻塞**）。
  你选好了菜，按一下桌上的呼叫器（**Selector 事件**）。前台经理看到后，派一个服务员来处理你的订单。
  这样，**少量的服务员**可以应付很多桌客人，只有在客人真正需要服务时才过去。

*   **AIO (异步非阻塞)**：
  你去饭馆吃饭，你点完菜，服务员给你一个取餐器（**回调机制**），然后就去忙别的了。
  你不用一直盯着厨房，当你把餐弄好了，取餐器响了/甚至服务员直接把饭端上来了（**操作系统完成工作后通知**）。
  在这个过程中，你完全被解放了。

### 💡 面试加分项（延伸回答）

如果面试官对你的回答比较满意，可以顺势补充以下几点，展示技术深度：

1.  **Netty 与 NIO 的关系**：
    虽然原生 NIO 很强大，但开发极其复杂（Selector 处理空轮询 Bug 等）。实际工作中，我们基本都用 **Netty** 框架。Netty 对 Java NIO 进行了封装，提供了更易用的 API 和更好的性能（它是基于 NIO 的 Reactor 线程模型实现的）。

2.  **底层区别**：
    - **BIO/NIO** 底层都基于操作系统内核的 `select`/`poll` 或 `epoll`（Linux），应用层线程依然需要去发起系统调用（读/写），所以叫**同步**。
    - **AIO** 底层依赖操作系统的异步 IO（如 Linux 的 `io_uring` 或 Windows 的 IOCP），应用层不需要调用读写函数，只需要等回调，所以叫**异步**。

3.  **应用现状**：
    目前 Java 后端开发中，**NIO (Netty)** 是主流。Tomcat 从 8.x/9.x 开始也支持了 NIO 模式。AIO 在 Linux 上的支持早期不够成熟，普及率不如 NIO 高。

**ai**: 这是一个考察你对 **对象引用** 和 **内存模型** 理解的经典问题。面试官想确认你是否理解 Java 中对象传递的本质。

以下是回答的逻辑结构，你可以参考这些内容来完善你的 `[[Java SE]]` 笔记：

---

### 🗣️ 回答策略

#### 1. 开篇直述定义
> **浅拷贝**和**深拷贝**主要针对对象中含有引用类型成员变量的情况。
> - **浅拷贝**：创建一个新对象，但新对象的属性如果是引用类型，那么拷贝的是**内存地址**（引用）。也就是说，原对象和新对象的引用属性指向同一个内存空间。
> - **深拷贝**：创建一个新对象，并且递归地将原对象中的所有引用类型属性都复制一份新的对象。原对象和新对象完全独立，互不影响。

#### 2. 举例说明（关键步骤，一定要举例）
假设我们有一个 `Student` 类，里面有一个基本类型 `int age` 和一个引用类型 `Address` 对象。

```java
class Address {
    String city;
    // ...
}
class Student implements Cloneable {
    int age;
    Address addr;
    // ...
}
```

- **如果是浅拷贝**：
  当你拷贝 `student1` 得到 `student2` 后：
  - 修改 `student2.age = 20`，不影响 `student1.age`。
  - 修改 `student2.addr.city = "上海"`，**`student1.addr.city` 也会变成 "上海"**。
  - **原因**：`student1` 和 `student2` 的 `addr` 指向的是同一个 `Address` 对象。

- **如果是深拷贝**：
  当你拷贝 `student1` 得到 `student3` 后：
  - 修改 `student3.addr.city = "北京"`，`student1.addr.city` **保持不变**。
  - **原因**：`student3` 的 `addr` 是一个新的 `Address` 对象，与原对象无关。

#### 3. 如何实现
- **浅拷贝**：
  类实现 `Cloneable` 接口，并调用 `super.clone()`。Java 中 `Object` 类的 `clone()` 方法默认就是浅拷贝。

- **深拷贝**：
  1. **重写 `clone()` 方法**：在 `clone()` 方法中，对引用类型的属性也手动调用其 `clone()` 方法（前提是该引用类型也实现了 `Cloneable`）。
  2. **序列化（推荐）**：将对象序列化为字节流，再反序列化回对象。这种方式不需要修改每个类的代码，可以完美处理复杂的引用嵌套图。

#### 4. 常见陷阱（加分项）
- **关于 String**：
  `String` 虽然是引用类型，但因为它是**不可变** 的。
  在浅拷贝中，原对象和新对象会引用同一个字符串常量（或对象）。当你修改其中一个 String 变量时（如 `s = "new"`），实际上是在常量池（或堆）中创建了新对象，并把引用指向它，**不会影响原对象的 String**。
  所以，浅拷贝对 String 类型表现得像深拷贝。

面试官问这个问题，通常是想知道你是否知道除了最常见的“强引用”之外，JVM 还提供了另外三种引用来辅助垃圾回收。

以下是高分的回答逻辑：

---

### 🗣️ 回答逻辑

#### 1. 开篇总述
> Java 中的引用不仅仅是我们平时写的 `Object obj = new Object()`（强引用）。在 JDK 1.2 之后，Java 将引用分为了 **四种强度**，依次是：**强引用、软引用、弱引用、虚引用**。不同的引用强度决定了对象在 JVM 垃圾回收时不同的“存活”行为。

#### 2. 逐一详解（必背）

**① 强引用 - 永不回收**
- **定义**：最常见的引用，如 `User user = new User()`。
- **特点**：只要强引用还存在，垃圾回收器**永远不会**回收被引用的对象。
- **风险**：即使发生了 OOM（内存溢出），程序宁愿终止也不会回收强引用对象。这是造成**内存泄漏**的主要原因。

**② 软引用 - 内存不足才回收**
- **定义**：用 `SoftReference` 类来描述。
- **特点**：**“有用但非必须”**。如果内存足够，GC 不会回收它；如果内存不足（即将 OOM），GC 会回收这些对象。
- **应用场景**：**高速缓存**。比如图片缓存、网页缓存。当内存吃紧时，自动释放缓存，避免 OOM。

**③ 弱引用 - GC 见即收**
- **定义**：用 `WeakReference` 类来描述。
- **特点**：**“更弱的生命周期”**。无论内存是否充足，**只要**垃圾回收器发现了它，就**一定会**回收它。
- **应用场景**：
    - `ThreadLocal` 中的 `ThreadLocalMap` 的 Key 使用了弱引用，防止内存泄漏。
    - `WeakHashMap`，当外部没有强引用指向 Key 时，Map 自动移除该键值对。

**④ 虚引用 - 影子引用**
- **定义**：用 `PhantomReference` 类来描述。
- **特点**：
    - **最弱**的引用，无法通过虚引用获取对象实例（`get()` 方法永远返回 `null`）。
    - **唯一目的**：是为了在对象被垃圾回收器回收时，能够收到一个**系统通知**。必须配合 `ReferenceQueue`（引用队列）使用。
- **应用场景**：**管理堆外内存**（如 Netty、NIO 的 `DirectByteBuffer`）。因为 JVM 无法直接回收堆外内存，所以通过虚引用来监控对象是否被回收，如果回收了，就去通知系统释放对应的堆外内存。

### 💡 面试加分项（深入追问）

如果面试官问得比较深，可以补充以下细节：

1.  **关于 `ReferenceQueue`**：
    软引用、弱引用、虚引用都可以在构造时传入一个队列。当对象即将被 GC 回收时，JVM 会将这个引用对象本身加入到队列中。我们可以通过监控这个队列，知道哪些对象被回收了。虚引用**必须**配合队列使用才有意义。

2.  **ThreadLocal 为什么用弱引用？**
    `ThreadLocalMap` 的 Key 是 `ThreadLocal` 对象。为了防止 ThreadLocal 被外部置为 null 后，由于 Entry 还持有强引用导致 Key 无法回收（内存泄漏），所以 Key 被设计为弱引用。这样当外部强引用断开，GC 时 Key 就会被回收（变为 null），虽然 Value 依然需要手动清理，但这极大地降低了泄漏风险。

3.  **缓存策略对比**：
    软引用适合做通用缓存（LRU 策略的替代品），而弱引用适合做短暂性的缓存或观察者模式中的弱监听器。
