**user**: # Redis 基础与线程模型
## 1. Redis 为什么这么快？
### 15 秒简答
Redis 快主要是因为：纯内存操作、IO 多路复用、单线程避免上下文切换和锁开销、高效数据结构（如哈希表、跳表）。
### 3 分钟详答
Redis 的高性能主要来自 4 点：  
1）**纯内存操作**：数据在内存中，读写延迟远低于磁盘数据库。  
2）**IO 多路复用**：使用 epoll/kqueue 等多路复用机制，一个线程就能监听大量连接，减少线程切换开销。  
3）**单线程模型**：核心命令处理是单线程，避免了多线程的锁竞争和上下文切换，在内存操作场景下简单高效。  
4）**高效数据结构**：如哈希表、跳表、压缩列表等，使很多操作时间复杂度接近 O(1) 或 O(logN)。
Redis 6.0 之后在网络读写上引入了多线程，但命令执行仍是单线程。
### 追问
**Q：Redis 既然是单线程，那还能利用多核 CPU 吗？**  
A：可以部署多个 Redis 实例或使用 Redis Cluster 分片，把不同键分布到不同节点，从而利用多核。
---
## 2. Redis 和 Memcached 有什么区别？
### 15 秒简答
Redis 支持更丰富的数据类型、支持持久化、原生集群和主从复制，Memcached 只支持简单字符串、无持久化、无原生集群。
### 3 分钟详答
共同点：都是内存 KV 数据库，常用作缓存，性能都很高。  
主要区别：  
1）**数据类型**：Redis 支持 String、List、Set、ZSet、Hash 等复杂结构；Memcached 只支持简单字符串。  
2）**持久化**：Redis 有 RDB、AOF；Memcached 重启数据丢失。  
3）**集群与高可用**：Redis 3.0 后原生支持 Cluster，还支持主从和哨兵；Memcached 无原生集群，需客户端分片。  
4）**线程模型**：Memcached 多线程；Redis 核心是单线程 + IO 多路复用。  
5）**特性**：Redis 支持事务、Lua、发布订阅等，Memcached 不支持。
### 追问
**Q：什么场景更适合用 Memcached？**  
A：如果只需要简单 KV 缓存、对持久化无要求、追求极致性能，可以考虑 Memcached；需要复杂数据结构或持久化时更适合 Redis。

## 3. Redis 有哪些常用数据类型？分别适合什么场景？
### 15 秒简答
String：缓存、计数、分布式锁。Hash：对象、购物车。List：消息队列、分页。Set：去重、共同关注。ZSet：排行榜、带权队列。
### 3 分钟详答
五大基础类型：  
1）**String**：可存字符串、整数、二进制，常用于缓存对象、计数器、分布式锁、Session 共享。  
2）**Hash**：键值对集合，适合存储对象（如用户信息、购物车），可以对单个字段做原子操作。  
3）**List**：有序可重复列表，底层为双向链表，适合做简易消息队列、最新列表、分页。  
4）**Set**：无序唯一集合，适合去重（UV）、标签、共同关注、抽奖。  
5）**ZSet**：带分数的有序集合，适合排行榜、带权重的队列。
高级类型：Bitmap、HyperLogLog、Geo、Stream 等，用于签到、基数统计、地理位置、消息队列等场景。
### 追问
**Q：存储用户信息，用 String 还是 Hash 更合适？**  
A：如果经常需要更新单个字段，用 Hash 更合适，可以避免每次都读写整个对象，节省带宽和序列化开销。

## Redis 持久化与内存管理
## 4. Redis 有哪些持久化方式？各有什么优缺点？
### 15 秒简答
RDB：定时快照，文件紧凑，恢复快，但可能丢最后一次持久化后的数据。AOF：追加写命令，数据更安全，但文件更大、恢复较慢。混合持久化结合两者优点。
### 3 分钟详答
1）**RDB（Redis Database）**  
- 原理：将内存数据生成快照写入磁盘，如 `bgsave` fork 子进程写入 RDB 文件。  
- 优点：文件紧凑，适合备份；恢复速度快；对写性能影响小。  
- 缺点：无法做到秒级持久化，两次快照之间的数据可能丢失。
2）**AOF（Append Only File）**  
- 原理：将写命令追加到 AOF 文件，可通过 `appendfsync` 控制刷盘策略。  
- 优点：数据安全性更高，最多丢一秒数据；AOF 可读性好。  
- 缺点：文件较大，恢复较慢，对写性能有一定影响。
3）**混合持久化（4.0+）**  
- RDB 做全量，AOF 做增量，兼顾恢复速度和数据安全。
### 追问
**Q：如果 Redis 宕机，怎么选择持久化方案？**  
A：允许分钟级数据丢失可只用 RDB；要求少丢数据可开启 AOF；生产环境常用混合持久化。

## 5. Redis 的过期删除策略是怎样的？
### 15 秒简答
Redis 采用“惰性删除 + 定期删除”：访问时检查是否过期；同时后台定期随机抽样部分键删除过期键。
### 3 分钟详答
1）**惰性删除**：读取键时检查 TTL，过期则删除；简单但可能残留大量过期键占用内存。  
2）**定期删除**：每隔一段时间随机抽查一部分键，删除其中过期的，避免全量扫描消耗 CPU。  
两者结合：既避免每次访问都额外开销，又避免过期键长期占用内存。  
若内存仍不足，则进入内存淘汰策略。
### 追问
**Q：如果大量 key 同时集中过期，会有什么问题？**  
A：可能导致 CPU 飙升，甚至短暂卡顿；建议给过期时间加随机偏移，避免同时失效。
---
## 6. Redis 有哪些内存淘汰策略？
### 15 秒简答
常用策略：noeviction（不淘汰，报错）、allkeys-lru、volatile-lru、volatile-ttl、allkeys-random、volatile-random 等。
### 3 分钟详答
当内存达到上限时，Redis 按配置的策略淘汰键：  
- **noeviction**：不淘汰，新写入报错（默认）。  
- **allkeys-lru**：从所有键中移除最近最少使用的。  
- **volatile-lru**：从设置了过期时间的键中移除最近最少使用的。  
- **volatile-ttl**：优先淘汰 TTL 较短的键。  
- **allkeys-random / volatile-random**：随机淘汰。  
选择建议：缓存场景常用 allkeys-lru 或 volatile-lru；热点数据多时可考虑 volatile-lru。
### 追问
**Q：缓存用 allkeys-lru 合适，还是 volatile-lru 合适？**  
A：如果所有数据都可能被淘汰，用 allkeys-lru；如果只希望淘汰可能过期的临时缓存，用 volatile-lru。

## Redis 缓存常见问题
## 7. 什么是缓存穿透？怎么解决？
### 15 秒简答
查询不存在的 key，请求直接打到数据库；解决：缓存空值、布隆过滤器、接口层校验和限流。
### 3 分钟详答
**现象**：大量请求查询一个既不存在缓存也不存在数据库的 key，导致数据库压力骤增。  
**解决方案**：  
1）**缓存空值**：将不存在的 key 缓存为空，设置较短 TTL，避免重复查库。  
2）**布隆过滤器**：在请求前判断 key 是否可能存在，不存在直接拦截，存在再查缓存/数据库。  
3）**接口层校验**：过滤明显非法参数（如 id <= 0）。  
4）**限流与熔断**：对异常 IP 或高频请求限流。
### 追问
**Q：布隆过滤器为什么会有“误判”？**  
A：它用多个哈希函数映射到位数组，可能存在不同 key 映射到相同位，导致“可能存在”的误判。
---
## 8. 什么是缓存击穿？怎么解决？
### 15 秒简答
热点 key 过期瞬间，大量并发同时查数据库；解决：设置热点数据永不过期、互斥锁重建缓存或逻辑过期。
### 3 分钟详答
**现象**：某个热点 key 过期，同时大量请求到达，都发现缓存失效，一起去查数据库，导致数据库压力瞬间很大。  
**解决方案**：  
1）**设置热点数据永不过期**：逻辑过期时间存到 value 中，后台异步更新。  
2）**互斥锁**：只让一个线程去查库并回写缓存，其他等待或返回旧数据。  
3）**逻辑过期**：缓存中存储逻辑过期时间，过期时返回旧数据，同时异步重建。
### 追问
**Q：互斥锁方案有什么缺点？**  
A：会牺牲一定性能，且实现不当可能导致死锁或等待时间过长，一般配合合理的超时时间使用。
---
## 9. 什么是缓存雪崩？怎么解决？
### 15 秒简答
大量 key 同时失效或 Redis 宕机，导致请求全部打到数据库；解决：随机过期时间、高可用集群、多级缓存、限流降级。
### 3 分钟详答
**现象**：大量缓存同一时间失效，或 Redis 宕机，所有请求直达数据库，可能导致数据库崩溃。  
**解决方案**：  
1）**随机过期时间**：在基础过期时间上加随机偏移，避免同时失效。  
2）**高可用架构**：主从 + 哨兵或 Redis Cluster，避免单点故障。  
3）**多级缓存**：本地缓存 + Redis，减轻数据库压力。  
4）**限流与降级**：在入口层限流，对非核心服务降级。
### 追问
**Q：如果 Redis 集群宕机，有什么应急方案？**  
A：快速重启或切换到备用集群；前端限流降级；关键数据走数据库 + 本地缓存。
---
## 10. 如何保证缓存与数据库的数据一致性？
### 15 秒简答
常用策略：先更新数据库再删除缓存，配合延迟双删、消息队列重试、或分布式锁保证强一致。
### 3 分钟详答
典型问题：更新数据库后如何同步缓存。  
常见策略：  
1）**Cache Aside（旁路缓存）**：读时先缓存，未命中再查库并回写；写时先更新数据库，再删除缓存。  
2）**延迟双删**：先删除缓存 -> 更新数据库 -> 延迟后再删除缓存，减少读写不一致窗口。  
3）**订阅 binlog + MQ**：通过 Canal 等订阅 MySQL binlog，异步删除/更新缓存。  
4）**分布式锁**：对读写加锁，保证强一致但影响性能。  
实际中多采用最终一致性，而不是强一致。
### 追问
**Q：为什么是“删除缓存”而不是“更新缓存”？**  
A：删除更简单且避免并发写导致脏数据；另外可以结合懒加载，按需重建缓存。

## Redis 集群与高可用
## 11. Redis 主从复制流程是怎样的？
### 15 秒简答
从节点连接主节点，先全量同步（RDB + 缓冲区），之后主节点持续发送写命令给从节点进行增量同步。
### 3 分钟详答
1）**建立连接**：从节点配置 `slaveof`，与主节点建立连接。  
2）**全量同步**：主节点执行 `bgsave` 生成 RDB 发送给从节点，从节点加载；同时主节点用缓冲区记录期间的写命令。  
3）**增量同步**：主节点持续将新写命令发送给从节点，从节点执行以保持数据一致。  
4）**断线重连**：若从节点断线，会尝试部分重同步，使用复制偏移量和积压缓冲区。
### 追问
**Q：主从复制会出现数据不一致吗？**  
A：会，由于异步复制，主节点宕机时可能存在未同步的数据；可通过半同步或等待从节点确认来降低风险。

## 12. Redis Sentinel 是什么？有什么用？
### 15 秒简答
哨兵是监控 Redis 主从节点的系统，自动完成故障发现、故障转移，并通知客户端新主节点地址。
### 3 分钟详答
**功能**：监控、通知、自动故障转移。  
1）**监控**： Sentinel 定期向主从节点发送 PING，判断是否主观下线。  
2）**客观下线**：多个 Sentinel 对同一主节点达成主观下线，则认为客观下线。  
3）**选举领头 Sentinel**：负责故障转移。  
4）**故障转移**：从节点中选一个升级为主，其他从节点改为复制新主，通过发布订阅通知客户端。
### 追问
**Q：哨兵集群至少需要几个节点？**  
A：通常至少 3 个，以保证多数派决策，避免脑裂。

## 13. Redis Cluster 是怎么分片的？
### 15 秒简答
Redis Cluster 将键空间划分为 16384 个槽，每个节点负责一部分槽；根据 key 的 CRC16 计算槽位，实现数据分片。
### 3 分钟详答
1）**哈希槽**：16384 个槽，每个节点负责一部分。  
2）**路由**：客户端连接任一节点，若 key 所在槽不在该节点，返回 MOVED 或 ASK 指向正确节点。  
3）**扩容/缩容**：通过迁移槽位实现数据迁移。  
4）**高可用**：每个主节点可以有多个从节点，主节点故障时从节点顶替。
### 追问
**Q：为什么是 16384 个槽？**  
A：这是一个折中值：槽数量足够保证分布均匀，同时节点间信息交换量可控。

## Redis 应用与分布式锁
## 14. Redis 可以用来做什么？
### 15 秒简答
典型场景：缓存、分布式锁、排行榜、计数器、简单消息队列、会话存储、限流、附近的人等。
### 3 分钟详答
1）**缓存**：减轻数据库压力，提高响应速度。  
2）**分布式锁**：基于 SETNX 或 Redisson 实现。  
3）**排行榜**：使用 ZSet 实现实时排行榜。  
4）**计数器**：文章阅读数、点赞数、粉丝数，利用 INCR/DECR。  
5）**消息队列**：List 或 Stream 实现简单队列。  
6）**会话存储**：分布式 Session 共享。  
7）**限流**：String + INCR 或滑动窗口实现。
### 追问
**Q：Redis 做消息队列有什么缺点？**  
A：不支持完备的 ACK、重试、消息堆积等机制，可靠性不如 Kafka、RabbitMQ，适合轻量场景。

## 15. 如何用 Redis 实现一个简单的分布式锁？
### 15 秒简答
使用 `SET key value EX seconds NX` 一条命令实现加锁，解锁时用 Lua 脚本比较 value 后删除，避免误删其他客户端的锁。
### 3 分钟详答
**基本实现**：  
- 加锁：`SET lock_key uuid EX 10 NX`，保证原子性和超时时间。  
- 解锁：使用 Lua 脚本，判断 value 是否是自己持有的，是则删除，否则不做操作。  
**需要注意**：  
1）**过期时间**：防止客户端宕机导致锁无法释放。  
2）**唯一标识**：用 UUID 或线程 ID 标识持有者，避免误删别人的锁。  
3）**锁续期**：业务执行时间可能超过过期时间，需要看门狗机制续期（如 Redisson）。
### 追问
**Q：SETNX + EXPIRE 有什么问题？**  
A：两条命令不是原子操作，如果在 SETNX 之后宕机，锁无法设置过期时间，可能导致死锁。
---
## 16. Redisson 分布式锁的原理是什么？
### 15 秒简答
Redisson 使用 SET key UUID PX NX 加锁，Lua 脚本解锁，支持可重入锁、看门狗自动续期、公平锁等特性。
### 3 分钟详答
1）**加锁**：使用 `SET key uuid PX ttl NX`，保证原子性。  
2）**可重入**：使用 Hash 结构存储锁持有者和重入次数，同一线程可多次加锁，次数减到 0 才真正释放。  
3）**看门狗机制**：后台线程定期续期锁，防止业务未执行完就过期。  
4）**公平锁**：使用 ZSet 排队，按请求顺序获取锁。  
Redisson 通过封装这些细节，让分布式锁使用更简单可靠。
### 追问
**Q：Redisson 分布式锁在主从架构下一定安全吗？**  
A：在主从切换时可能存在锁丢失问题，强一致场景可考虑 RedLock 或 ZooKeeper 方案。

## Redis 事务与 Pipeline
## 17. Redis 事务是怎样的？支持回滚吗？
### 15 秒简答
Redis 事务通过 MULTI/EXEC 将一组命令打包顺序执行，不支持回滚，某条命令失败后后续命令仍会执行。
### 3 分钟详答
**使用方式**：  
- `MULTI` 开启事务，命令入队。  
- `EXEC` 提交执行，`DISCARD` 取消事务。  
- `WATCH` 可监视键，若事务执行前键被修改，则事务不会执行（乐观锁）。  
**特点**：  
1）**不保证原子性**：单个命令原子，事务整体不保证，失败不会回滚。  
2）**没有回滚机制**：Redis 不像 MySQL 有 undo log，不支持回滚。  
3）**隔离性**：事务执行期间不会被其他命令插队。
### 追问
**Q：如何在 Redis 中实现“要么全成功，要么全失败”的效果？**  
A：可以使用 Lua 脚本，将多个操作封装成一个原子操作；或业务层自己实现补偿逻辑。
---
## 18. Redis Pipeline 有什么用？
### 15 秒简答
Pipeline 可以一次发送多条命令，减少网络往返时间，提高批量操作性能，但不保证原子性。
### 3 分钟详答
**原理**：客户端将多个命令打包一起发送，服务端一次性执行并返回结果，减少多次网络交互开销。  
**特点**：  
1）适合批量写入/查询场景。  
2）不保证原子性，命令可能被其他请求插队。  
3）与事务区别：事务保证顺序执行，Pipeline 只是一次发送多条命令。
### 追问
**Q：Pipeline 和 Lua 脚本有什么区别？**  
A：Pipeline 减少网络开销，但命令间可被插队；Lua 脚本保证多条命令原子执行，但更耗 CPU。

## Redis 性能与实战问题
## 19. 什么是 bigkey？有什么危害？
### 15 秒简答
bigkey 指键值内容很大（如很大的 String、很多元素的 Hash/Set/ZSet），会导致内存不均、阻塞主线程、删除卡顿等问题。
### 3 分钟详答
**常见类型**：  
- 大 String：几个 MB 甚至 GB 的缓存。  
- 大集合：元素很多的 Hash、Set、ZSet。  
**危害**：  
1）**内存不均**：单个节点内存占用过大。  
2）**阻塞**：删除、序列化、网络传输耗时，可能阻塞 Redis 主线程。  
3）**慢查询**：相关命令耗时变长。  
**处理方式**：  
- 拆分为多个小 key。  
- 使用 UNLINK 异步删除。  
- 控制单个 key 的元素数量和大小。
### 追问
**Q：如何发现 bigkey？**  
A：使用 `redis-cli --bigkeys` 扫描，或通过 RDB 分析工具离线分析。

## 20. 什么是 hotkey？怎么解决？
### 15 秒简答
hotkey 是访问频率极高的 key，可能导致该节点压力过大甚至被打满；解决：本地缓存、key 加随机前缀分散、限流等。
### 3 分钟详答
**问题**：某个 key 访问量远高于其他，造成单节点热点，可能影响整体性能。  
**解决方案**：  
1）**本地缓存**：在应用层缓存热点数据，减少对 Redis 的访问。  
2）**分片/多副本**：将热点 key 复制到多个节点，通过客户端随机读不同节点。  
3）**限流与降级**：对热点 key 访问做限流。  
4）**拆分**：将大 key 拆成多个小 key，分散压力。
### 追问
**Q：如何发现 hotkey？**  
A：通过 Redis 的 `--hotkeys` 选项（4.0+）、客户端统计、或监控系统访问量分析。

## 21. Redis 中如何实现延迟队列？
### 15 秒简答
使用 ZSet，以执行时间作为 score，定时任务用 ZRANGEBYSCORE 查询到期任务并处理。
### 3 分钟详答
**实现方式**：  
1）生产者：`ZADD delay_queue score member`，score 为执行时间戳。  
2）消费者：定时执行 `ZRANGEBYSCORE delay_queue 0 当前时间` 获取到期任务，用 `ZREM` 删除并处理。  
3）为防止重复消费，需要保证 `ZRANGEBYSCORE + ZREM` 的原子性，可用 Lua 脚本。

**Q：Redis 延迟队列和 Kafka 延时消息有什么区别？**

# Redis 核心原理与实战面试指南

## 一、基础数据结构与底层原理

## 1. Redis 为什么这么快？
**15秒简答：**
主要基于三点：纯内存操作、单线程模型避免了多线程上下文切换和锁竞争、使用了IO多路复用机制（如epoll）处理高并发连接。此外，高效的数据结构（如跳表、SDS）也是关键。
**3分钟详答：**
Redis的高性能是多方面因素共同作用的结果：
1.  **基于内存的存储**：Redis所有数据都存储在内存中，内存的读写速度远快于磁盘，这是其性能高的基石。
2.  **单线程模型**：Redis的主要工作线程（处理命令）是单线程的。这看似是个缺点，实则是优点。它避免了多线程环境下频繁的上下文切换开销，也不需要考虑各种锁带来的性能损耗（如死锁、加锁解锁耗时）。
3.  **IO多路复用**：Redis利用 epoll（Linux下）等IO多路复用技术，可以同时监控多个IO事件。当连接有数据准备好时，才会通知线程处理，实现了单个线程高效处理大量并发连接。
4.  **高效数据结构**：如 SDS 获取长度 O(1)、跳表 O(logN) 查询等，优化了操作效率。
**补充：Redis 6.0 多线程**
Redis 6.0 引入了多线程，但**仅用于网络数据的读写和协议解析**，核心命令执行依然是单线程。目的是提高网络 IO 瓶颈，保持逻辑处理的简单和原子性。

## 2. Redis 和 Memcached 有什么区别？
**15秒简答：**
Redis 支持更丰富的数据类型、支持持久化、原生集群和主从复制；Memcached 只支持简单字符串、无持久化、无原生集群，且是多线程模型。
**3分钟详答：**
共同点：都是内存 KV 数据库，高性能，常作缓存。
主要区别：
1.  **数据类型**：Redis 支持 String、List、Set、ZSet、Hash 等复杂结构；Memcached 仅支持 Key-Value 字符串。
2.  **持久化**：Redis 有 RDB、AOF，重启后数据可恢复；Memcached 重启数据丢失。
3.  **集群与高可用**：Redis 支持 Sentinel（哨兵）和 Cluster（原生分片）；Memcached 无原生集群，需客户端通过一致性哈希分片。
4.  **线程模型**：Memcached 是多线程；Redis 核心是单线程（6.0 后网络 IO 多线程）。
5.  **特性**：Redis 支持事务、Lua 脚本、发布订阅；Memcached 不支持。

## 3. Redis 的常用数据类型及其底层实现？
**15秒简答：**
五种基础类型：String（SDS）、List（QuickList/双向链表）、Hash（字典/压缩列表）、Set（字典/整数集合）、ZSet（跳表+字典/压缩列表）。
**3分钟详答：**
1.  **String（字符串）**：底层是 SDS（Simple Dynamic String），类似于 Java 的 ArrayList。二进制安全，获取长度 O(1)，通过空间预分配减少内存分配。
2.  **List（列表）**：早期版本用双向链表或压缩列表。现在的版本多用 **quicklist**（快速列表），是一个双向链表，每个节点是一个压缩列表，平衡了指针开销和连续内存优势。
3.  **Hash（哈希）**：底层使用字典（哈希表）。当数据量小且元素少时，使用压缩列表节省内存。
4.  **Set（集合）**：底层是字典（Value 为 Null）或者整数集合。
5.  **ZSet（有序集合）**：底层使用 **跳表（SkipList）** 和 **字典** 组合。字典保证 O(1) 查找分数，跳表保证范围查询和排序的高效。
**追问：存储用户信息用 String 还是 Hash？**
**简答：** 如果经常需要更新对象的单个字段（如积分），用 **Hash** 更合适，可以网络传输更少的数据，避免序列化整个对象；如果是整体存取且无部分更新需求，String 更简单。

## 4. 什么是 Redis 的跳跃表？
**15秒简答：**
跳跃表是一种有序数据结构，通过多层索引实现快速访问，平均 O(logN)。它是 ZSet 的底层实现之一，实现比红黑树简单，范围查询性能好。
**3分钟详答：**
你可以把它想象成“多层链表”。最底层包含所有节点，上层是下层索引。
*   **查找过程**：从最高层开始，如果下一个节点大于目标值，就降一层，像下楼梯一样快速定位。
*   **优势**：相比平衡树，跳表实现简单，插入删除不需要复杂的旋转操作，且范围查询（如 ZRANGE）可以直接遍历底层链表，非常高效。

## 5. Redis String 类型的 SDS 是什么？有什么优点？
**15秒简答：**
SDS（Simple Dynamic String）是 Redis 自定义字符串。相比 C 字符串，它记录了长度（O(1) 获取）、二进制安全、通过空间预分配减少内存重分配次数。
**3分钟详答：**
C 字符串以 `\0` 结尾，无法存图片等二进制数据，获取长度需遍历 O(N)。SDS 结构体包含 `len`、`free` 和 `buf`。
*   **常数获取长度**：直接读 `len`。
*   **杜绝缓冲区溢出**：修改前检查空间不足自动扩容。
*   **减少内存分配**：扩容时预分配额外空间，惰性释放（缩小时不立即物理回收内存）。

## 二、持久化机制

## 6. Redis 的 RDB 和 AOF 有什么区别？
**15秒简答：**
RDB 是快照，定时将内存数据生成快照存磁盘，文件小恢复快，但可能丢数据。AOF 是日志，记录每个写命令，数据安全性高，文件大恢复慢。
**3分钟详答：**
1.  **RDB**：`bgsave` 时 fork 子进程生成二进制文件。适合备份，恢复极快。缺点是每隔一段时间才保存，宕机会丢失最后一次快照后的数据。
2.  **AOF**：记录所有写命令。配置 `appendfsync always/everysec/no` 控制刷盘频率。数据丢失最少（everysec 最多丢 1 秒），但文件体积大，恢复需要重放所有命令，慢。
3.  **混合持久化（4.0+）**：AOF 重写时，将 RDB 格式的内容写入 AOF 文件开头，后续增量命令以 AOF 格式追加。**兼顾了 RDB 的恢复速度快和 AOF 的数据安全性**。

## 7. 什么是 AOF 重写？
**15秒简答：**
AOF 重写是压缩 AOF 文件体积的过程。Redis 后台 fork 子进程，根据当前内存数据生成新的写命令集合，去除冗余命令。
**3分钟详答：**
随着写操作增多，AOF 文件会变大（如多次 Set 同一个 Key）。重写机制：
*   **原理**：子进程遍历内存数据，生成新的最小化 AOF 文件。
*   **重写期间的新命令**：主进程同时将新写命令追加到 AOF 重写缓冲区，子进程结束后，主进程将缓冲区内容追加到新文件，原子性地替换旧文件。

## 三、架构与高可用

## 8. Redis 主从复制的原理是什么？
**15秒简答：**
分为全量复制和增量复制。初次连接全量同步（主节点发 RDB + 缓冲区命令），断线重连后根据复制偏移量进行增量同步。
**3分钟详答：**
1.  **建立连接**：从节点发送 `SYNC`（或 `PSYNC`）。
2.  **全量同步**：主节点 `bgsave` 生成 RDB 发送给从节点，同时记录此期间的写命令到复制积压缓冲区。
3.  **增量同步**：从节点加载 RDB 后，主节点将缓冲区期间的增量命令发给从节点。
4.  **命令传播**：之后主节点每执行一个写命令，就同步给从节点。
**追问：主从复制会有延迟吗？**
**简答：** 会有。命令传播是异步的。读写分离场景下，从节点可能读到旧数据。

## 9. 什么是 Redis 哨兵？
**15秒简答：**
哨兵是监控主从状态的分布式系统。功能：监控、自动故障转移（主节点挂了选新主）、通知客户端。
**3分钟详答：**
*   **监控**：定时 PING 主从节点，判断是否下线。
*   **主观下线 vs 客观下线**：单个哨兵认为挂了是主观下线；多数哨兵（Quorum）确认挂了是客观下线。
*   **故障转移**：客观下线后，哨兵集群选举领头哨兵，由它从从节点中选一个升级为主，并让其他从节点指向新主。
**追问：哨兵集群至少需要几台？**
**简答：** 一般 3 台，且必须是奇数，防止脑裂，保证大多数派决策。

## 10. Redis Cluster 集群是如何工作的？
**15秒简答：**
无中心架构，通过 **16384 个槽（Slot）** 实现分片。每个节点负责一部分槽。客户端通过 CRC16(key) % 16384 计算槽位来定位节点。
**3分钟详答：**
*   **数据分片**：数据分散在不同节点，突破单机内存限制。
*   **重定向**：客户端访问节点 A，如果 Key 不在 A 负责的槽，A 返回 `MOVED` 错误，指引客户端去正确节点。
*   **Smart 客户端**：客户端会缓存槽位映射，减少重定向。
*   **高可用**：每个主节点有从节点，主挂了从会顶上。
**追问：为什么是 16384 个槽？**
**简答：** 如果槽位太少（如 256），节点存储的心跳信息会太大；如果太多（如 65536），消息头压缩比低且没必要。16384 是折中值。

## 四、缓存三大经典问题

## 11. 什么是缓存穿透？怎么解决？
**15秒简答：**
查询不存在的数据，缓存没命中，请求直达数据库。恶意攻击会导致 DB 压力大。解决：缓存空对象、布隆过滤器、接口校验。
**3分钟详答：**
*   **缓存空对象**：查库为空时，将 Key 缓存为 Null，设置短 TTL。简单但占内存，若 Key 随机变化仍可能打穿。
*   **布隆过滤器**：在 Redis 前加个过滤器，利用位图判断 Key **一定不存在**，直接拦截。适用于数据相对固定的场景。
*   **互斥锁**：当缓存失效时，只让一个线程查库（适用于击穿，也可辅助防止穿透时的并发击穿）。
**追问：布隆过滤器为什么会有误判？**
**简答：** Hash 冲突。多个 Key 可能映射到同一个位。判断“不存在”是绝对准确的，判断“存在”可能是误判。

## 12. 什么是缓存击穿？怎么解决？
**15秒简答：**
热点 Key 突然过期，大量并发请求瞬间击穿缓存打到 DB。解决：热点 Key 永不过期、互斥锁、逻辑过期。
**3分钟详答：**
*   **互斥锁**：当缓存失效，使用 `SETNX` 加锁。只让一个线程查库回填，其他等待或返回旧值。防止 DB 瞬间崩溃。
*   **逻辑过期**：不设置 TTL，在 Value 中存过期时间。由后台异步线程负责更新缓存，前台请求总能读到数据（哪怕稍微旧一点）。
**追问：互斥锁有什么缺点？**
**简答：** 可能导致线程阻塞，增加响应时间；如果锁释放失败可能导致死锁（需设置超时）。

## 13. 什么是缓存雪崩？怎么解决？
**15秒简答：**
大量 Key 同时过期或 Redis 宕机，所有请求涌向 DB。解决：随机过期时间、高可用架构、限流降级、多级缓存。
**3分钟详答：**
*   **过期时间随机化**：在原 TTL 基础上加随机值（如 1-5 分钟），避免集体失效。
*   **Redis 高可用**：搭建 Sentinel 或 Cluster，杜绝单点故障。
*   **限流降级**：当缓存失效压力过大，启动限流保护 DB，或返回默认兜底数据。

## 14. 如何保证缓存与数据库的双写一致性？
**15秒简答：**
追求最终一致性。推荐：先更新数据库，再删除缓存。配合延时双删或订阅 Binlog 异步删除。
**3分钟详答：**
*   **Cache Aside 模式**：
    *   **读**：先读缓存，没有则读库并回写缓存。
    *   **写**：**先更新数据库，再删除缓存**。删缓存比更新缓存好，避免并发写覆盖。
*   **为什么不先删缓存？**：先删缓存，若还没更库，另一个线程读了旧数据写回缓存，导致永久脏数据。
*   **为什么是删除不是更新？**：删除是懒加载，效率高；并发写时，更新缓存可能导致脏数据。
*   **延时双删**：先删缓存 -> 更新 DB -> 休眠（如 500ms） -> 再删缓存。防止更新期间旧数据回填。
*   **Binlog 异步**：利用 Canal 订阅 MySQL Binlog，解析后发到 MQ 或直接删除 Redis，解耦业务代码。

## 五、内存管理与实战优化

## 15. Redis 的过期键删除策略是什么？
**15秒简答：**
惰性删除 + 定期删除。访问时检查过期，后台定期随机抽查删除。
**3分钟详答：**
*   **惰性删除**：访问 Key 时检查，过期则删。缺点是若 Key 没被访问，会一直占内存。
*   **定期删除**：每隔一段时间（默认 10 次/秒），随机抽取设置了过期时间的 Key，检查并删除。
*   若以上都没删掉且内存不足，触发内存淘汰策略。

## 16. Redis 有哪些内存淘汰策略？
**15秒简答：**
`noeviction`（不淘汰报错）、`allkeys-lru`（所有键选最近最少用）、`volatile-lru`（过期键中选）、`allkeys-random`、`volatile-ttl` 等。
**3分钟详答：**
当内存达到 `maxmemory` 限制时：
*   `noeviction`：默认策略，不淘汰，写请求返回错误。
*   `allkeys-lru`：优先淘汰最久未被访问的数据（推荐用于缓存）。
*   `volatile-lru`：只在设置了过期时间的键中淘汰。
*   `allkeys-random`：随机淘汰。
*   `volatile-ttl`：优先淘汰剩余时间短（TTL小）的键。

## 17. 什么是 bigkey？有什么危害？如何处理？
**15秒简答：**
bigkey 指值很大（如几 MB 的 String）或元素很多（如万级元素的 Hash）。危害：阻塞线程、内存不均。处理：拆分、异步删除。
**3分钟详答：**
*   **危害**：
    1.  **阻塞**：读取或删除 bigkey 会耗时很长，阻塞主线程（Redis 单线程），导致后续请求排队甚至超时。
    2.  **内存不均**：集群模式下，某节点内存占用过高，无法迁移。
*   **发现**：使用 `redis-cli --bigkeys` 扫描，或 `MEMORY USAGE key` 查单个 key 占用。
*   **处理**：
    1.  **拆分**：将大 Hash 拆分成多个小 key（如 Hash+Field 拆为 Key:Field）。
    2.  **删除**：不要用 `DEL`，用 `UNLINK`（非阻塞删除，在后台线程释放内存）。

## 18. 什么是 hotkey？怎么解决？
**15秒简答：**
访问频率极高的 Key。危害：导致单节点负载过高。解决：本地缓存、读写分离/备份、Key 分散。
**3分钟详答：**
*   **发现**：`redis-cli --hotkeys`（需 4.0+），或客户端统计。
*   **解决**：
    1.  **本地缓存**：在应用层 JVM 缓存热点数据，减少对 Redis 压力。
    2.  **副本/分片**：将热点 Key 加后缀（如 Key_1, Key_2）分散到不同节点，写入时同步，读取时随机读。

## 六、分布式特性与实战应用

## 19. 如何用 Redis 实现分布式锁？
**15秒简答：**
使用 `SET key value NX PX timeout` 原子命令加锁。解锁使用 Lua 脚本验证 value 并删除。
**3分钟详答：**
*   **加锁**：`SET lock_key uuid NX PX 30000`。NX 保证不存在才加锁，PX 设置自动过期时间防止死锁。
*   **解锁**：Lua 脚本：
    ```lua
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else
        return 0
    end
    ```
    确保只有锁的持有者才能解锁。
*   **注意**：**不要**使用 `SETNX` + `EXPIRE` 分两步操作，非原子性可能导致 SETNX 成功后崩溃，锁未设置过期时间造成死锁。

## 20. Redisson 分布式锁的原理是什么？
**15秒简答：**
Redisson 封装了锁的实现，支持可重入、Watchdog 自动续期、公平锁。
**3分钟详答：**
*   **可重入**：利用 Hash 结构记录 `Key -> {ThreadId: Count}`，同一线程加锁 Count+1，释放 Count-1。
*   **Watchdog（看门狗）**：如果业务未执行完，锁快过期时，后台线程自动给锁续期（默认 30秒）。解决了业务超时导致锁提前释放的问题。
*   **RedLock 算法**：在主从架构下，主节点宕机可能导致锁丢失。Redisson 实现了 RedLock（红锁），向多个独立的 Redis 节点同时加锁，只要大多数节点成功即算加锁成功，提高安全性。

## 21. Redis 事务是怎样的？支持回滚吗？
**15秒简答：**
Redis 事务通过 `MULTI`、`EXEC`、`DISCARD` 实现。**不支持回滚**，命令失败后其他命令仍会执行。
**3分钟详答：**
*   **流程**：`MULTI` 开启，命令入队，`EXEC` 批量执行。
*   **隔离性**：事务执行期间不会被其他命令插队。
*   **不支持回滚**：即使某条命令语法错误，Redis 也不会回滚之前已执行的命令。这是为了保持简单和高性能。
*   **乐观锁（WATCH）**：`WATCH key` 可以在事务前监控 Key，如果事务执行前 Key 被修改，事务执行失败。可以实现类似 CAS 的效果。

## 22. 如何用 Redis 实现限流？
**15秒简答：**
固定窗口（String+INCR）、滑动窗口（ZSet）、令牌桶/漏桶。
**3分钟详答：**
*   **固定窗口**：`INCR key`，第一次设置过期时间。如果超过阈值则拒绝。缺点：临界点可能双倍流量。
*   **滑动窗口**：使用 ZSet，Value 为唯一标识，Score 为时间戳。每次请求移除时间窗口外的旧数据，统计 ZSet 长度。更平滑。
*   **令牌桶**：按速率向 ZSet/List 放令牌，请求获取令牌。适合突发流量。

## 23. 如何用 Redis 实现延迟队列？
**15秒简答：**
利用 ZSet，Score 为执行时间戳。消费者轮询 `ZRANGEBYSCORE` 获取到期任务处理。
**3分钟详答：**
*   **生产者**：`ZADD tasks <timestamp> <task_id>`。
*   **消费者**：轮询执行 `ZRANGEBYSCORE tasks -inf <current_timestamp>` 获取任务，再使用 `ZREM` 删除。
*   **原子性**：使用 Lua 脚本将“查询”和“删除”原子化，防止多个消费者同时抢到同一个任务。适合订单超时取消场景。

## 24. Redis Pipeline 有什么用？
**15秒简答：**
Pipeline 一次发送多条命令，减少网络往返时间（RTT），提高批量操作性能。
**3分钟详答：**
*   **原理**：客户端打包多条命令一次性发送给服务端，服务端处理完一次性返回。将 N 次网络交互变成 1 次。
*   **注意**：不保证原子性，Pipeline 中的命令可能被其他客户端的命令插队。需要原子性请用事务或 Lua。
*   **适用**：批量导入数据、批量查询。
