在当前的 [[Spring]] 笔记中，详细记录了 Spring 是如何利用**三级缓存**机制来解决这种“A依赖B，B又依赖A”的问题的。

简单来说，Spring 的处理流程如下：

- **创建 A**：Spring 先实例化 A，并将 A 的对象工厂放入**三级缓存**。
- **A 注入 B**：在给 A 填充属性时发现需要 B，于是去创建 B。
- **创建 B**：Spring 实例化 B，将 B 的对象工厂放入**三级缓存**。
- **B 注入 A**：B 需要注入 A，此时 A 还没完全初始化好。Spring 会去缓存中找 A。
    - 先看一级缓存（没有），再看二级缓存（没有），最后在三级缓存找到了 A 的工厂。
    - 调用工厂的 `getObject()` 方法获取 A 的早期引用（此时 A 已实例化但未初始化），并将 A 移入**二级缓存**。
- **B 完成**：B 初始化完成，放入**一级缓存**。
- **A 完成**：A 此时可以拿到完全初始化好的 B，继续完成自己的初始化，最后也放入**一级缓存**。

笔记中还提到，之所以需要**三级缓存**而不是**二级缓存**，主要是为了解决 AOP（面向切面编程）代理的问题，确保在需要生成代理对象时，整个 Bean 的生命周期中不会存在两个不同的 Bean 实例。

**ai**: 在 Spring 中，事务的传播机制（Transaction Propagation）主要解决的是**多个事务方法相互调用时，事务应该如何传播或共享**的问题。

Spring 定义了 7 种事务传播行为，通过 `@Transactional` 注解的 `propagation` 属性来指定（默认是 `REQUIRED`）：

1. **REQUIRED (默认值)**
   - **行为**：如果当前存在事务，就加入该事务；如果当前不存在事务，就新建一个事务。
   - **场景**：这是最常用的配置，适用于绝大多数业务逻辑。

2. **REQUIRES_NEW**
   - **行为**：新建事务。如果当前存在事务，把当前事务挂起（暂停）。
   - **场景**：当需要保证某个方法独立运行，不受外层事务回滚影响时使用（例如日志记录）。
   - **注意**：外层事务失败回滚，不会导致该方法的提交回滚；反之，该方法抛出异常回滚，也不会导致外层事务回滚（前提是外层捕获了异常）。

3. **SUPPORTS**
   - **行为**：如果当前存在事务，就加入该事务；如果当前不存在事务，就以非事务方式执行。
   - **场景**：适用于非核心逻辑，比如读取某些配置信息，有没有事务都行。

4. **NOT_SUPPORTED**
   - **行为**：以非事务方式执行操作。如果当前存在事务，把当前事务挂起。
   - **场景**：对于耗时较长且不需要事务的操作，可以使用这个方式来避免占用数据库连接。

5. **MANDATORY**
   - **行为**：如果当前存在事务，就加入该事务；如果当前不存在事务，就抛出异常。
   - **场景**：强制要求方法必须在事务中运行，通常用于确保数据一致性。

6. **NEVER**
   - **行为**：以非事务方式执行。如果当前存在事务，就抛出异常。
   - **场景**：强制要求方法不能在事务中运行。

7. **NESTED**
   - **行为**：如果当前存在事务，就在嵌套事务内执行。如果当前不存在事务，就新建一个事务（类似于 REQUIRED）。
   - **场景**：嵌套事务是外层事务的一部分。外层事务回滚，嵌套事务一定回滚；但嵌套事务回滚（仅回滚到保存点 Savepoint），不会导致外层事务回滚。
   - **注意**：它依赖于底层数据库对保存点的支持。

**ai**: Spring Boot 的自动装配原理是面试中的高频问题，其核心思想是**“约定优于配置”**，通过 `@EnableAutoConfiguration` 注解来实现。

整个过程主要可以分为三个步骤：**启动注解导入** -> **加载候选配置** -> **按需筛选装配**。

以下是详细的解析：

### 1. 核心入口：@SpringBootApplication
项目启动类上的 `@SpringBootApplication` 是一个复合注解，它包含了三个核心注解：
- `@SpringBootConfiguration`：标识这是一个配置类。
- `@ComponentScan`：自动扫描组件。
- **`@EnableAutoConfiguration`**：**开启自动装配**，这是自动装配的关键。

### 2. 加载候选配置
`@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 导入了 `AutoConfigurationImportSelector` 类。

- 这个类内部会去加载 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (Spring Boot 2.7/3.x+) 或 `META-INF/spring.factories` (旧版本) 文件。
- 这些文件里配置了所有**候选**的自动配置类全限定名（比如 `DataSourceAutoConfiguration`, `WebMvcAutoConfiguration` 等），可能有 100 多个。

### 3. 按需筛选装配
虽然加载了所有的候选配置，但 Spring Boot 并不会把它们全部生效。它会根据条件注解（`@Conditional` 系列注解）来决定是否需要加载某个配置类。

常见的条件判断有：
- `@ConditionalOnClass`：类路径中存在指定的类（比如引入了 `spring-boot-starter-web` 才会加载 Web 相关配置）。
- `@ConditionalOnMissingBean`：容器中不存在该 Bean 时才自动配置（允许用户自定义配置覆盖默认配置）。
- `@ConditionalOnProperty`：配置文件中是否开启了某个配置项。

### 总结
简单来说，Spring Boot 自动装配就是：
1.  **从配置文件中读取**所有可能的配置类。
2.  **根据条件注解过滤**掉不符合当前环境的配置（比如没引入相关依赖，或者用户已经自己配置了 Bean）。
3.  **把过滤后剩下的配置**注册到 Spring 容器中。

**ai**: 你的理解中，**关于“先写日志”这个策略是对的，但关于 Redo Log 的作用（回滚）是完全错误的**。

这是两个很容易混淆的概念，请务必区分开：

### 1. 错误点：Redo Log 不是用来“回滚”的
- **Redo Log (重做日志)**：它的作用是**重做**。如果系统崩溃了，它会将那些**已经提交了但还没来得及写入数据文件的数据**重新恢复一遍。它保证的是事务的**持久性**（Durability）。
- **Undo Log (回滚日志)**：这才是用来**回滚**的。如果事务执行失败或者调用了 rollback，Undo Log 会将数据**恢复到修改前的样子**。它保证的是事务的**原子性**（Atomicity）。

### 2. 修正后的理解
WAL (Write-Ahead Logging) 的核心意思是：**修改数据前，必须先把日志写到磁盘上**。

**正确的描述应该是：**
> WAL：先写日志后刷盘。即在修改操作（将内存中的脏页刷入磁盘）之前，先将修改记录写入 Redo Log。这样即使系统崩溃，也能根据 Redo Log **重做**（恢复）已提交的事务，保证数据不丢失。

你可以检查一下你的 [[MySQL]] 笔记，里面其实已经正确区分了这两者：
- `redo log(物理日志): 保证事务持久性`
- `undo log: 回滚和MVCC`

### 1. 你的理解（正确部分）

### 2. 需要补充的关键点（不严谨部分）
“什么时候刷盘”是性能和数据安全性的博弈点。MySQL 提供了 `innodb_flush_log_at_trx_commit` 参数来控制这个行为，它有三种取值：

- **设置为 1（默认值，最安全）**：
  - **行为**：每次事务提交时，都**立即**调用 `fsync` 将 Redo Log 从 Buffer 刷到磁盘。
  - **评价**：你的描述完全符合这种情况。即使系统崩溃，丢数据风险最小。

- **设置为 0（最快，最不安全）**：
  - **行为**：事务提交时不会立即刷盘。Redo Log 只留在内存 Buffer 中。由 **后台 Master Thread** 每隔 1 秒钟把 Buffer 写入 OS 缓存，并刷入磁盘。
  - **评价**：如果你的 MySQL 崩溃了（没断电），Buffer 里的数据就丢了；即使提交成功了，数据也可能没落盘。

- **设置为 2（折中）**：
  - **行为**：每次事务提交时，只把 Redo Log 写入 **OS 缓存**（文件系统缓冲区），不直接 `fsync` 到物理磁盘。由 Master Thread 每秒负责 `fsync`。
  - **评价**：如果数据库挂了，不丢数据（因为 OS 缓存里还在）；但如果服务器**断电**了，OS 缓存里的数据就丢了，最多丢 1 秒的数据。

> - 设为 0 时，交由后台线程每秒刷盘。
> - 设为 1 时（默认），每次事务提交时都刷盘（最安全）。
