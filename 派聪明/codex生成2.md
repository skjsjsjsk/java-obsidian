# PaiSmart 面试提问与追问参考

---

## 一、先抓简历真实性的高频问题

### 1. 你先整体介绍一下这个项目的架构和核心链路

#### 追问

- 这是微服务还是单体？
- MySQL、Redis、MinIO、Elasticsearch 分别存什么？
- 这个项目里 RAG 的完整链路是什么？

#### 参考答案

这个项目本质上是一个基于 Spring Boot 的单体 RAG 平台，不是微服务拆分。后端负责认证、上传、异步解析、向量化、混合检索、WebSocket 对话、配额和管理后台，前端是独立的 Vue 3 应用。

核心存储分工是：

- MySQL 存业务元数据，比如用户、组织标签、文件上传记录、文本切块、订单、配额相关数据
- Redis 存上传进度 bitmap、限流计数、聊天会话历史、配额统计、组织标签缓存
- MinIO 存原始分片和合并后的源文件
- Elasticsearch 存检索索引，包括文本内容、页码、锚点、向量、权限字段

RAG 主链路是：

1. 前端分片上传文件
2. 后端写 MinIO，并记录文件元数据和上传进度
3. 合并后发 Kafka 异步任务
4. 消费者解析文件、切 chunk、生成 embedding、写入 ES
5. 用户提问时先做带权限的混合检索
6. 把检索结果拼成上下文 prompt
7. 调大模型流式生成答案，并通过 WebSocket 推回前端

---

### 2. 你简历里写了“模块化分层架构”，具体怎么分层？

#### 追问

- 你的 controller、service、repository 分别负责什么？
- 为什么不把解析、检索、聊天全写在一个 service 里？
- 你说的“模块化”体现在哪些包上？

#### 参考答案

后端采用的是比较典型的分层设计：

- `controller` 负责对外接口
- `service` 负责业务编排
- `repository` 负责数据库访问
- `config` 放安全、Kafka、ES、WebSocket、配额等基础设施配置
- `client` 放外部模型调用
- `consumer` 放 Kafka 异步消费
- `handler` 放 WebSocket handler

模块化主要体现在业务域拆分上，而不是 Maven 多模块。比如上传链路集中在 `UploadController`、`UploadService`、`FileProcessingConsumer`；检索集中在 `HybridSearchService`；聊天集中在 `ChatWebSocketHandler`、`ChatHandler`、`LlmProviderRouter`。这样做的好处是职责清晰，链路解耦，后续替换存储或模型供应商时影响面更可控。

## 二、上传、Kafka、异步处理

### 3. 你为什么要把文件上传、解析、向量化拆成异步链路？

#### 追问

- 如果同步处理会有什么问题？
- Kafka 在这里解决了什么问题？
- 为什么不是直接开线程池异步？

#### 参考答案

主要原因是解析和向量化都比较重，尤其是大文件、PDF、多页文档，解析耗时长，调用 embedding 接口也可能有网络波动。如果在上传接口里同步做，用户请求会很慢，接口容易超时，而且 Tomcat 工作线程会被长时间占用。

Kafka 的作用主要有三点：

- 解耦上传和后处理，上传成功后先快速返回
- 提升系统抗峰值能力，流量高时消费者按能力消费
- 便于失败重试和死信处理，避免解析失败直接影响上传主流程

相比直接用线程池，Kafka 更适合这种“可堆积、可重试、和请求生命周期分离”的任务。线程池适合同 JVM 内短时异步，Kafka 更适合明确的异步业务链路。

### 4. 你们分片上传是怎么做的？为什么要用 Redis bitmap？

#### 追问

- 为什么不用数据库记录每个 chunk 状态？
- bitmap 相比 set/list 有什么优势？
- 断点续传是怎么判断缺哪些分片的？

#### 参考答案

前端会把文件按固定大小切片，每个 chunk 携带 `fileMd5`、`chunkIndex`、`fileName` 等信息上传。后端收到后先把 chunk 存到 MinIO 的 `chunks/{fileMd5}/{chunkIndex}`，再用 Redis bitmap 记录这个 chunk 是否上传完成。

用 bitmap 的原因是它非常适合这种“一个文件有很多离散编号分片，每片只需要一位状态”的场景：

- 内存占用小
- 读取某个 chunk 是否上传是 O(1)
- 可以快速遍历拿到已上传分片列表

断点续传时，前端会先调状态接口，后端根据 bitmap 返回已上传 chunk 列表，前端只补传缺失分片。

如果全部落数据库也能做，但单文件 chunk 多时写放大更明显，查询状态也更重。数据库更适合存最终需要追踪的 chunk 元数据，不适合高频状态打点。

### 5. 合并分片后为什么还要保留 MinIO，而不是直接解析文件流？

#### 追问

- 为什么不把文件直接存在本地磁盘？
- MinIO 在这个项目里除了存储还有什么好处？
- 后续预览和下载是不是也依赖 MinIO？

#### 参考答案

MinIO 是对象存储，更适合管理上传文件。它和本地磁盘相比，有几个实际优势：

- 更适合服务化部署，不依赖单机磁盘路径
- 支持预签名 URL，方便下载和预览
- 文件对象和应用实例解耦，横向扩容更方便

这个项目里 MinIO 不只是上传暂存，还承担源文件长期存储。后续文档下载、PDF 预览、单页预览都依赖 MinIO 的对象读取或预签名 URL。

---

### 6. Kafka 消费失败了怎么办？你们是如何保证任务可靠性的？

#### 追问

- 如果解析失败会不会丢任务？
- 有没有重试和死信队列？
- 怎么避免重复消费导致重复索引？

#### 参考答案

项目里 Kafka 配置了生产幂等、重试和 DLT 主题。消费者异常时会把异常继续抛出，让 Kafka 的错误处理机制接管，而不是自己吞掉。这样可以触发重试，重试耗尽后再进入死信队列。

重复消费的问题主要通过业务幂等来兜底。比如重建索引或重新处理文档时，会先按 `fileMd5` 清理旧的 ES 文档和旧的向量记录，再重新解析和写入。这样即使重复消费，也不会无限叠加脏数据。

更稳的答法可以补一句：严格意义上的“端到端 exactly once”很难，这里做的是“Kafka 至少一次 + 业务侧幂等”。

## 三、解析、切块、向量化

### 7. 你们为什么要把文档先切 chunk 再做 embedding？

#### 追问

- 为什么不整篇文档直接向量化？
- chunk 多大合适？
- chunk 切太大或太小分别有什么问题？

#### 参考答案

整篇文档直接向量化的问题是粒度太粗，检索时召回结果不够精确，而且长文档可能超出 embedding 输入限制。切 chunk 后，每个 chunk 对应一个更小的语义单元，既便于召回，也便于把引用定位到具体页面和片段。

chunk 大小本质上是召回精度和上下文完整性的平衡：

- 太大：一个 chunk 混入太多主题，召回精度下降
- 太小：语义上下文不完整，容易召回碎片化内容

项目里不是简单按固定字数硬切，而是优先按段落、句子做语义切分，特别长的句子再用 HanLP 分词兜底切开，这样语义更完整。

---

### 8. 你们解析 PDF 时做了哪些特殊处理？

#### 追问

- PDF 最难的地方是什么？
- 你怎么处理页眉页脚干扰？
- 你们为什么要保留页码和 anchorText？

#### 参考答案

PDF 难点主要在于它不是天然结构化文本，常见问题是页眉页脚重复、换行混乱、跨页断裂。项目里对 PDF 做了按页解析，并额外做了页眉页脚清洗：会统计每页边界几行的重复模式，把高频重复的边界文本过滤掉，减少噪声进入索引。

保留页码和 `anchorText` 有两个目的：

- 检索命中后能把引用定位到具体页面
- 前端点击引用时，可以跳到对应 PDF 页或者展示命中的文本锚点

这对 RAG 的可解释性很关键，因为用户不只是要答案，还想知道“答案来自哪一页哪一段”。

---

### 9. 你简历里写了 2048 维向量，这个维度是怎么来的？维度越高越好吗？

#### 追问

- 维度高会带来什么成本？
- 维度低会有什么问题？
- 这个维度是你们自己训练的吗？

#### 参考答案

2048 维不是我们自己训练出来的，而是所接 embedding 模型的输出维度配置。这个项目里 embedding 供应商是外部模型服务，系统只是调用它的 API。

维度不是越高越好。维度高通常表示表达能力更强，但也会带来：

- 存储成本增加
- ES 向量检索内存和计算开销变大
- 索引构建速度变慢

维度低则可能损失语义表达能力，导致召回质量下降。所以一般是结合模型默认维度、检索效果和系统成本做权衡。

---

### 10. 解析后的文本为什么既存 MySQL，又存 Elasticsearch？

#### 追问

- 会不会数据冗余？
- 哪边算主数据？
- 如果两边不一致怎么办？

#### 参考答案

这是典型的“业务存储”和“检索存储”分离。

- MySQL 里的 `DocumentVector` 更偏业务主数据，保存 chunk 文本、页码、锚点、权限字段等，方便做重建索引、审计、管理
- Elasticsearch 里的索引是为了检索性能，把文本和向量组织成更适合查询的数据结构

确实有冗余，但这是有意设计。检索系统通常不直接当主库。两边不一致时，主思路是以 MySQL 和 MinIO 源文件为准，必要时触发 reindex 重建 ES。

## 四、混合检索与 Elasticsearch

### 11. 你说你们做了混合检索，具体怎么混的？

#### 追问

- 是先 BM25 再向量，还是先向量再 BM25？
- KNN 和 BM25 各自解决什么问题？
- 为什么不只用向量检索？

#### 参考答案

项目里的思路是“向量召回 + 关键词约束 + BM25 重排”。具体来说：

- 先对 query 做 embedding
- 在 ES 上做 KNN 检索，拿到语义相近候选
- 同时要求 `textContent` 有文本匹配，不完全放弃关键词约束
- 再用 BM25 做 rescore，让关键词命中的 chunk 排名更合理

KNN 更擅长找语义相似，BM25 更擅长处理实体词、关键词、精确术语。只用向量检索时，语义泛化太强，容易召回“像但不准”的内容；只用关键词时，又容易漏掉同义表达。所以企业知识问答里混合检索通常更稳。

---

### 12. 你们在 Elasticsearch 里怎么做权限过滤？

#### 追问

- 为什么权限过滤不只放在业务层？
- 如果只是先搜后过滤，会有什么风险？
- 你们过滤条件有哪些？

#### 参考答案

权限过滤不只放在业务层，而是直接进入 ES 查询 filter。这样做的原因是防止“先召回后过滤”导致无权限数据参与打分，甚至被间接泄露。

项目里的过滤条件主要有三类：

- 用户自己的文档
- 公开文档
- 用户所属组织标签下的文档

这些字段在 ES 文档里是显式存储的，比如 `userId`、`orgTag`、`public`。这样检索阶段就能完成权限裁剪。

---

### 13. 召回 topK 你们怎么定？是不是越大越好？

#### 追问

- recallK 太大有什么问题？
- topK 太小会怎样？
- 你会怎么做调优？

#### 参考答案

不是越大越好。`topK` 和召回窗口本质是效果和性能的平衡。

- 太小：有价值的 chunk 可能进不了候选集，答案容易缺料
- 太大：查询更慢、噪声更多，后续 prompt 也会膨胀

调优时一般看两类指标：

- 线上体验：平均响应时间、超时率
- 检索质量：命中率、人工评估、常见问答场景下的回答准确性

项目里比较稳妥的说法是：线上先用较小的 `topK`，再结合 recall window 和 BM25 rescore 控制质量，不会无限放大。

## 五、WebSocket、流式输出、多轮对话

### 14. 你为什么用 WebSocket 做聊天，而不是 SSE 或轮询？

#### 追问

- WebSocket 在这个项目里最核心的价值是什么？
- SSE 能不能做？
- 你们有没有做心跳和重连？

#### 参考答案

最核心的价值是双向长连接，适合流式输出和交互控制。这个项目里除了服务端持续推送模型输出，还支持前端发送停止生成指令，所以 WebSocket 比 SSE 更合适。

SSE 也能做单向流式输出，但它天然偏服务端单向推送。如果只是展示打字机效果，SSE 可以；但如果还要支持心跳、停答、连接态管理，WebSocket 更统一。

项目里前端已经做了心跳、自动重连、鉴权失败处理，后端也会对连接做 token 校验。

---

### 15. 你们 WebSocket 的鉴权是怎么做的？

#### 追问

- 为什么把 token 放在路径里？
- 这么做有没有风险？
- 连接建立后如何识别用户？

#### 参考答案

当前实现是通过 `/chat/{token}` 建连接，后端在 `ChatWebSocketHandler` 里解析路径上的 token，再做 JWT 校验，校验通过后提取 userId，把 session 和用户绑定。

严格来说，把 token 放路径里不是最理想方案，因为可能出现在日志里。更稳妥的做法通常是：

- 放在 header
- 或者连接建立前先换短期 ws token

这个项目已经有一个给前端下发内部命令 token 的机制，说明后续其实可以继续往“短期 token”方向演进。答题时不要硬说这是最优方案，而是说“这是当前实现，安全上还能继续优化”。

---

### 16. 流式输出你们是怎么做的？如何知道一次回答结束了？

#### 追问

- 大模型返回 chunk 时你们怎么处理？
- 为什么要做 completion 通知？
- 如果模型中途报错，前端怎么感知？

#### 参考答案

模型侧是流式 API，后端会订阅每个 chunk，然后每收到一个文本片段就通过 WebSocket 立刻发给前端。前端拿到 chunk 后做增量拼接，形成打字机效果。

回答结束除了依赖模型流结束，还额外有一层完成通知。因为前端不能只靠“暂时没新 chunk”判断结束，所以后端会发送一个 `completion` 类型消息，明确告诉前端本轮已经结束。

如果模型报错，后端会通过统一错误消息回传前端，前端就能停止 loading 并提示用户。

---

### 17. 你们多轮对话的上下文是怎么管理的？

#### 追问

- 会话历史放哪？
- 为什么放 Redis，不放 MySQL？
- 历史太长怎么办？

#### 参考答案

当前聊天会话历史主要缓存到 Redis。因为聊天链路对读取延迟比较敏感，而 Redis 更适合这种短期、高频、按会话读写的数据。每次用户发消息，会先拿历史，再和当前检索结果一起构造 prompt。

历史不会无限增长，项目里会限制保留最近若干条消息，避免 prompt 膨胀和 token 成本失控。长期审计类会话如果有需要，可以再异步落 MySQL，但在线对话上下文更适合先放 Redis。

---

### 18. 用户点击引用后，你们是怎么定位到原文的？

#### 追问

- 为什么要做引用编号映射？
- 映射里存了哪些字段？
- 仅存文件名够不够？

#### 参考答案

检索结果在组装上下文时，会按顺序分配引用编号，比如 `[1]`、`[2]`。同时后端把编号和实际文档信息的映射保存下来，包括：

- `fileMd5`
- `fileName`
- `pageNumber`
- `anchorText`
- `chunkId`
- 检索方式和得分

这样前端点击引用时，不是靠文本匹配去猜，而是直接根据编号查后端映射，再打开对应文档或页码。仅存文件名不够，因为同名文件可能重复，而且无法精确定位到页和 chunk。

## 六、权限、多租户、组织标签

### 19. 你简历里写了“组织标签权限模型”，这个模型具体是什么？

#### 追问

- 这是物理隔离还是逻辑隔离？
- 用户能访问哪些文档？
- `orgTag` 和 `isPublic` 怎么配合？

#### 参考答案

这个项目不是租户级物理隔离，而是基于组织标签的逻辑隔离。每个文档会带 `orgTag` 和 `isPublic` 字段，用户自身也持有多个组织标签，并且有 `primaryOrg`。

访问规则大致是：

- 自己上传的文档可以访问
- 公开文档可以访问
- 自己所属组织标签下的文档可以访问
- 管理员有更高权限

所以这是“单库多租户 + 组织标签逻辑隔离”的模型。

---

### 20. 组织标签为什么要支持树状层级？

#### 追问

- 层级标签在业务上有什么价值？
- 父子标签怎么做继承？
- 如果出现循环依赖怎么办？

#### 参考答案

树状层级的价值在于更贴近真实组织结构，比如集团、事业部、部门。这样权限配置不需要把所有叶子节点都手工分配给用户，而是可以通过父标签继承子标签的访问范围。

项目里会缓存用户的有效组织标签集合，也就是把层级关系展开后的结果缓存起来，用于检索和授权判断。更新标签层级时需要清理有效标签缓存。

循环依赖是必须防的，否则权限展开会出问题。代码里更新组织标签父子关系时会检查是否形成环。

---

### 21. 你们是怎么保证文档接口和检索接口都走权限控制的？

#### 追问

- 如果只在控制器里判断权限够不够？
- 为什么还要有 `OrgTagAuthorizationFilter`？
- 业务层和检索层分别做了什么？

#### 参考答案

权限控制分了两层：

- 请求层：`OrgTagAuthorizationFilter` 和 Spring Security 负责用户身份和接口访问边界
- 数据层：业务查询和 ES 检索时继续叠加 `userId/orgTag/public` 的条件

这样做的原因是避免只做一层判断。仅在 controller 判断权限，容易出现后续调用复用时遗漏；仅在业务层过滤，又可能让无权限数据先参与查询。双层控制更稳。

## 七、限流、配额、稳定性

### 22. 你们为什么既做“请求限流”，又做“token 配额”？

#### 追问

- 这两个概念有什么区别？
- 聊天和 embedding 分别怎么控？
- 为什么不能只限 QPS？

#### 参考答案

请求限流和 token 配额解决的是两个不同问题：

- 请求限流控制请求频率，防止瞬时打爆服务
- token 配额控制资源消耗，防止少量超长请求把成本打穿

比如用户一分钟只发 5 次请求，不代表消耗就低；如果每次都让模型生成超长内容，token 成本还是很高。所以大模型系统通常要同时控制“次数”和“消耗量”。

项目里聊天和 embedding 都有这两层控制。聊天侧会按用户消息频率限流，同时做 LLM token 预算；embedding 侧既有限调用次数，也有限上传和查询的 token 预算。

### 23. 你们 token 配额是怎么实现的？

#### 追问

- 如何在模型真正返回前做预算预留？
- 如果模型中途失败，预算怎么回滚？
- 你们为什么有 `QuotaConfiguration`？

#### 参考答案

思路是“先预留，后结算”。

在发起 LLM 或 embedding 请求前，会根据输入文本估算 token，先在 Redis 里做一次 reservation。如果后面调用成功，就根据实际 usage 做结算；如果失败，就把预留额度回滚。

`QuotaConfiguration` 的作用是根据配置切换不同的配额实现。当前项目支持两种模式：

- 每日额度模式
- 用户 token 余额模式

所以 `UsageQuotaService` 不是直接标注 `@Service`，而是通过配置决定实际注入哪个实现。

### 24. 如果 Redis 挂了，这个系统会受到什么影响？

#### 追问

- 哪些功能直接不可用？
- 哪些功能只是降级？
- 你会怎么做兜底？

#### 参考答案

Redis 在这个项目里是横切基础设施，影响面比较大。主要影响包括：

- 上传进度 bitmap 失效，断点续传能力受影响
- 聊天会话历史拿不到，多轮对话会降级
- 限流和 token 配额无法正常工作
- 组织标签缓存失效，可能退化到查数据库
- PDF 单页预览缓存失效

严格来说，有些功能不是完全不可用，而是退化。比如组织标签可以回源数据库，但限流和配额如果没有兜底策略，可能只能临时放开或直接拒绝高风险请求。答题时可以补一句：生产上通常会给 Redis 做高可用，核心限流和预算逻辑不能假设 Redis 永远稳定。

## 八、模型接入与 Prompt

### 25. 你们调用 DeepSeek 或其他模型时，Prompt 是怎么构造的？

#### 追问

- system prompt 放了什么？
- 检索结果怎么插进 prompt？
- 为什么要强调引用格式？

#### 参考答案

Prompt 的构造大致是：

- system 部分放回答规则，比如语言、结论优先、引用格式要求、无结果时的兜底策略
- 然后把检索结果组织成结构化参考上下文
- 最后拼接历史对话和当前用户问题

之所以强调引用格式，是因为这个系统不是单纯聊天，而是知识问答。用户需要能从回答跳回原文，所以 prompt 会要求模型按统一引用格式输出，后端也保留引用编号映射，方便前端做引用预览。

### 26. 你们为什么又有 `DeepSeekClient`，又有 `LlmProviderRouter`？

#### 追问

- 是不是历史遗留？
- 为什么要抽一层 router？
- 如果切模型供应商，需要改哪些地方？

#### 参考答案

可以理解为项目从“单模型直连”演进到“模型路由抽象”。

早期可能是直接用 `DeepSeekClient` 调固定供应商，后面为了支持模型配置化和供应商切换，引入了 `LlmProviderRouter` 和 `ModelProviderConfigService`。这样聊天链路不需要感知底层具体是哪家供应商。

如果切模型供应商，主要改的是供应商配置和调用层，不需要把聊天业务流程整体重写。这种抽象更利于后期扩展。

这比强行编 MyBatis XML、动态 SQL 更稳。资深面试官一听你乱讲，很快就能识别。

### 29. 这个项目里你觉得最有技术含量的点是什么？

- 是上传链路、检索链路还是聊天链路？

#### 参考答案

我会重点讲三块：

如果从个人贡献看，我会优先讲自己最熟的一条主链，比如“上传到 Kafka 异步处理”或者“HybridSearch 的权限过滤和检索编排”。

### 30. 如果让你再做一版，你最想优化什么？

#### 追问

- 你觉得当前架构的明显短板是什么？
- 哪些点是校招候选人容易忽略但你注意到了的？

#### 参考答案

我会优先优化四点：

- WebSocket 鉴权方式，避免 token 直接放 URL
- 检索评估体系，补线下数据集和可量化指标，不只凭主观体验调参

- ES 和 MySQL 之间的数据一致性治理，比如更标准的重建索引任务和状态机

这类回答的关键不是“吹架构”，而是说明你知道当前方案是工程折中，不是完美方案。
