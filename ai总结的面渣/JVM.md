# 一、JVM 内存区域与对象创建
## 1. JVM 运行时数据区有哪些？哪些线程共享，哪些线程私有？
**15 秒简答**  
线程私有：程序计数器、Java 虚拟机栈、本地方法栈。  
线程共享：Java 堆、方法区（JDK 8 以后主要是 Metaspace/元空间）。
**3 分钟详答**  
JVM 在执行 Java 程序时会把管理的内存划分为若干“运行时数据区”：  
- 程序计数器：记录当前线程执行的字节码指令地址，是**线程私有**，唯一不会 OOM 的区域。  
- Java 虚拟机栈：每个方法调用创建一个栈帧，存储局部变量表、操作数栈等，**线程私有**，StackOverflowError / OOM 与此相关。  
- 本地方法栈：为 Native 方法服务，**线程私有**。  
- Java 堆：存放对象实例，几乎所有对象都在这里分配，**线程共享**，GC 主要区域。  
- 方法区：存储类信息、常量、静态变量等，JDK 8 前是“永久代”，JDK 8 后是“元空间（Metaspace）”，**线程共享**。
**追问**  
- Q：为什么栈是线程私有的，堆是线程共享的？  
  **A**：栈保存方法调用链和局部变量，随线程创建销毁，不需要跨线程共享；堆是对象共享数据区，多线程共同访问同一个对象，所以设计为共享。
---
## 2. 简述 Java 对象的创建过程？
**15 秒简答**  
类加载检查 → 分配内存 → 初始化零值 → 设置对象头 → 执行 `<init>` 方法（构造函数）。
**3 分钟详答**  
当 JVM 遇到 `new` 指令时：  
1. 类加载检查：检查常量池中是否有该类的符号引用，未加载则先加载、验证、准备、解析。  
2. 分配内存：在堆中为对象分配空间，方式包括“指针碰撞”（内存规整时）或“空闲列表”（内存不规整时）；并发分配可用 CAS + TLAB。  
3. 初始化零值：将分配的内存（除对象头外）初始化为零值，保证字段无“野值”。  
4. 设置对象头：存储对象是哪个类的实例、哈希码、GC 分代年龄、锁状态等信息。  
5. 执行 `<init>` 方法：按程序意愿初始化字段，执行构造代码块和构造函数。
**追问**  
- Q：对象分配内存时，如何保证线程安全？  
  **A**：常用 CAS + 失败重试保证更新指针的原子性，另外每个线程可以使用 TLAB（Thread Local Allocation Buffer），优先在自己的小堆里分配，避免竞争。

## 3. String s = new String("abc") 创建了几个对象？
**15 秒简答**  
如果常量池中没有 "abc"：堆中 1 个对象，常量池 1 个，共 2 个；常量池已有则只在堆中创建 1 个。
**3 分钟详答**  
- 字面量 `"abc"` 在编译期会放入 Class 文件常量池，类加载后进入运行时常量池；在真正执行到这行时：  
  1. 先在堆中创建一个 String 对象，内容是 "abc"；  
  2. `intern()` 会把该对象引用放入字符串常量池（JDK7 以后是引用，而非复制）。  
- 如果常量池已有 "abc"，则 `new String` 只在堆中创建一个对象，不会再在常量池创建。  
- 所以是“1 或 2 个对象”，取决于常量池是否已存在该字面量。
**追问**  
- Q：intern() 方法的作用是什么？  
  **A**：如果字符串常量池中已包含等于此 String 的字符串，则返回池中对象引用；否则将此 String 对象的引用添加到池中并返回该引用。

# 二、垃圾回收基础
## 4. 如何判断一个对象是否可以回收？GC Roots 有哪些？
**15 秒简答**  
主流 JVM 使用可达性分析：从 GC Roots 出发，通过引用链不可达的对象即为可回收对象。可作为 GC Roots 的有：虚拟机栈中引用的对象、本地方法栈中引用的对象、方法区中静态属性和常量引用的对象等。
**3 分钟详答**  
- 引用计数法：对象被引用计数 +1，引用失效计数 -1，计数为 0 可回收；缺陷是无法解决循环引用问题。  
- 可达性分析：从 GC Roots 开始向下搜索，搜索走过的路径称为引用链；如果一个对象到 GC Roots 没有任何引用链相连，则证明对象不再使用。  
- 常见 GC Roots：  
  - 虚拟机栈（栈帧中的局部变量表）中引用的对象；  
  - 本地方法栈中 JNI（Native 方法）引用的对象；  
  - 方法区中类静态属性引用的对象；  
  - 方法区中常量引用的对象；  
  - 被同步锁（synchronized）持有的对象等。
**追问**  
- Q：强引用、软引用、弱引用、虚引用的区别？  
  **A**：强引用：只要存在就不回收；软引用：内存不足时回收；弱引用：下次 GC 时回收；虚引用：无法通过虚引用获取对象，用于跟踪对象被回收时的通知。
---
## 5. JVM 中常见的垃圾回收算法有哪些？各有什么特点？
**15 秒简答**  
标记-清除、标记-复制、标记-整理，以及基于分代思想的“分代收集”。
**3 分钟详答**  
- 标记-清除（Mark-Sweep）：标记出所有需要回收的对象，统一回收。缺点：效率不高，产生大量内存碎片。  
- 标记-复制（Copying）：将内存分为两块，每次只用一块，用完后将存活对象复制到另一块。适合新生代，浪费一半空间，但无需碎片。  
- 标记-整理（Mark-Compact）：标记后不直接清除，而是让存活对象向一端移动，然后清理掉边界以外的内存。适合老年代。  
- 分代收集：根据对象存活周期不同将内存划分为新生代、老年代，不同代采用不同算法：新生代用复制，老年代用标记-清除或标记-整理。
**追问**  
- Q：为什么新生代用复制算法，老年代用标记-整理或标记-清除？  
  **A**：新生代对象“朝生夕死”，存活率低，复制算法只需复制少量存活对象；老年代对象存活率高，没有额外空间做分配担保，且复制成本高，更适合整理或清除。
---
## 6. 新生代和老年代是如何划分的？Eden 与 Survivor 区默认比例是多少？
**15 秒简答**  
Java 堆一般分为新生代（Young）和老年代（Old）。新生代默认占堆的 1/3，老年代占 2/3。新生代内部 Eden:Survivor = 8:1:1（-XX:SurvivorRatio=8）。
**3 分钟详答**  
- 新生代：存放短生命周期对象，分为 Eden 区和两个 Survivor 区（From、To）。新对象首先在 Eden 分配，GC 时存活对象复制到 Survivor 区，并增加“年龄”。  
- 老年代：存放长期存活对象，对象在 Survivor 区中年龄达到阈值（默认 15）后晋升到老年代。  
- 默认比例：  
  - -XX:NewRatio=2 表示新生代 : 老年代 = 1 : 2；  
  - -XX:SurvivorRatio=8 表示 Eden : S0 : S1 = 8 : 1 : 1。
**追问**  
- Q：大对象会直接进入老年代吗？  
  **A**：可以配置 -XX:PretenureSizeThreshold，超过该值的大对象直接在老年代分配，避免在新生代 Survivor 区之间发生大量内存复制。
---
## 7. Minor GC、Major GC、Full GC 有什么区别？
**15 秒简答**  
Minor GC：只回收新生代；Major GC：回收老年代；Full GC：回收整个堆（新生代 + 老年代）和方法区。
**3 分钟详答**  
- Minor GC / Young GC：Eden 区满时触发，新生代 GC，速度快，Stop-The-World 时间短。  
- Major GC：回收老年代，常伴随至少一次 Minor GC（不是绝对）。  
- Full GC：回收新生代 + 老年代 + 方法区（永久代/元空间），GC 速度较慢，STW 时间长，应尽量避免或调优。
**追问**  
- Q：什么时候会触发 Full GC？  
  **A**：老年代空间不足、方法区/元空间空间不足、显式调用 System.gc()、CMS 出现 "Concurrent mode failure" 等情况都可能触发 Full GC。
---
## 8. 常见的垃圾收集器有哪些？Serial、Parallel、CMS、G1 各自特点？
**15 秒简答**  
新生代收集器：Serial、ParNew、Parallel Scavenge；老年代收集器：Serial Old、Parallel Old、CMS；以及 G1 收集器（分代 + 分区）。
**3 分钟详答**  
- Serial：单线程收集器，进行 GC 时必须暂停其他所有工作线程，简单高效，适合客户端模式。  
- ParNew：Serial 的多线程版本，可与 CMS 配合使用。  
- Parallel Scavenge：关注吞吐量（CPU 运行用户代码时间占比），适合后台计算场景。  
- Serial Old：Serial 的老年代版本。  
- Parallel Old：Parallel Scavenge 的老年代版本，关注吞吐量。  
- CMS（Concurrent Mark Sweep）：以“最短回收停顿时间”为目标，基于标记-清除，并发收集，低停顿，但有内存碎片问题。  
- G1：将堆划分为多个大小相等的 Region，跟踪各 Region 里垃圾堆积的价值，优先回收价值最大的 Region，可预测停顿时间，兼顾吞吐量和延迟。
**追问**  
- Q：CMS 的主要优缺点是什么？  
  **A**：优点：并发收集、低停顿；缺点：基于标记-清除有内存碎片，对 CPU 资源敏感，无法处理“浮动垃圾”，可能出现 "Concurrent mode failure"。
---
# 三、类加载机制
## 9. JVM 类加载过程分为哪几个阶段？
**15 秒简答**  
加载、验证、准备、解析、初始化五个阶段。
**3 分钟详答**  
1. 加载：通过类的全限定名获取 .class 文件字节流，加载到内存中，生成 Class 对象。  
2. 验证：确保 Class 文件字节流符合当前虚拟机要求，不会危害虚拟机安全。  
3. 准备：为类变量（static）分配内存并设置初始零值，而非程序中指定的初始值。  
4. 解析：将常量池内的符号引用替换为直接引用（指针、偏移量等）。  
5. 初始化：执行类构造器 `<clinit>()` 方法，按程序代码初始化静态变量和静态代码块。
**追问**  
- Q：准备阶段和初始化阶段的区别？  
  **A**：准备阶段只给类变量分配内存并设“零值”；初始化阶段才真正执行类定义的初始化代码，给变量赋正确初值。
---
## 10. 什么是双亲委派模型？为什么要有双亲委派？
**15 秒简答**  
当一个类加载器收到类加载请求时，先委派父类加载器加载，只有父加载器无法加载时，自己才尝试加载。
**3 分钟详答**  
- 类加载器层次：启动类加载器（Bootstrap）→ 扩展类加载器（Extension）→ 应用类加载器（Application），还可自定义类加载器。  
- 双亲委派模型：  
  - 自下而上委派：AppClassLoader → ExtClassLoader → BootstrapClassLoader；  
  - 如果父加载器可以加载，就成功返回；如果父加载器无法加载，子加载器才尝试自己加载。  
- 作用：  
  - 避免类的重复加载：父加载器已加载的类，子加载器没必要再加载；  
  - 保护核心类库安全：防止用户自定义类“冒充”核心类（如 java.lang.String）。
**追问**  
- Q：如何打破双亲委派模型？  
  **A**：自定义类加载器并重写 loadClass() 方法，改变默认委派逻辑，常见于 Tomcat、OSGi 等容器。
---
## 11. 能不能自己写一个 java.lang.String 类？为什么？
**15 秒简答**  
不能。因为 java.lang.String 由启动类加载器加载，用户自定义的 String 类由于双亲委派机制，不会被加载，即使写了也会被忽略，保证核心类安全。
**3 分钟详答**  
- 包名以 java. 开头的类属于核心类库，由启动类加载器加载。  
- 当应用类加载器尝试加载自定义的 java.lang.String 时，会先委派给扩展类加载器，再委派给启动类加载器。  
- 启动类加载器已经在核心库中找到了 java.lang.String，直接返回，因此用户自定义的 String 类不会被加载。  
- 这保证了核心 API 不被篡改，是双亲委派机制的安全意义之一。
**追问**  
- Q：如果我想定义一个与 JDK 同名类，可以怎么做？  
  **A**：可以放在不同包名下（如 my.lang.String），或者自定义类加载器并破坏双亲委派模型，但一般不推荐这么做。
---
# 四、JVM 调优与工具
## 12. 常见的 JVM 参数有哪些？
**15 秒简答**  
-Xms：初始堆大小；-Xmx：最大堆大小；-Xmn：新生代大小；-XX:NewRatio、-XX:SurvivorRatio 控制新生代比例；-XX:+PrintGCDetails 打印 GC 日志等。
**3 分钟详答**  
- 堆大小：  
  - -Xms：堆初始大小；  
  - -Xmx：堆最大大小；生产环境建议将 -Xms 和 -Xmx 设置成相同值，避免堆动态扩缩容带来的性能抖动。  
- 新生代控制：  
  - -Xmn：设置新生代大小；  
  - -XX:NewRatio：老年代 : 新生代比例；  
  - -XX:SurvivorRatio：Eden : Survivor 比例。  
- GC 日志与诊断：  
  - -XX:+PrintGCDetails：打印详细 GC 日志；  
  - -XX:+HeapDumpOnOutOfMemoryError：OOM 时导出堆快照；  
  - -XX:+PrintGCDateStamps / -XX:+PrintGCTimeStamps：输出时间戳便于分析。
**追问**  
- Q：设置 -Xmx 为 1G，实际 Java 进程占用了 1.3G，多出来的 0.3G 可能来自哪里？  
  **A**：堆外的直接内存、元空间、线程栈、NIO 缓冲区、JVM 自身数据结构、本地库等，这些都不在 -Xmx 管辖的 Java 堆内。
---
## 13. 你用过哪些 JVM 调优或诊断工具？
**15 秒简答**  
jps、jstat、jmap、jstack、jconsole、VisualVM 等。
**3 分钟详答**  
- jps：列出当前机器上运行的 JVM 进程。  
- jstat：监控类加载、内存、GC 等信息，如 `jstat -gcutil pid 1000` 每秒打印 GC 统计。  
- jmap：查看堆内存快照、对象统计，如 `jmap -histo:live pid` 或 `jmap -dump:format=b,file=heap.hprof pid`。  
- jstack：打印线程堆栈跟踪，用于排查死锁、CPU 飙高问题。  
- jconsole、VisualVM：图形化监控内存、线程、类加载等，支持 JMX 连接远程 JVM。
**追问**  
- Q：线上 CPU 飙高，你会如何排查？  
  **A**：先用 `top` 找到占用 CPU 高的 Java 进程，再用 `top -Hp pid` 找到高 CPU 线程，将线程 ID 转成 16 进制，用 `jstack pid | grep 0x<tid>` 定位具体堆栈，分析是哪个方法频繁执行。
---
## 14. 什么是内存溢出（OOM）和内存泄漏？如何排查堆溢出？
**15 秒简答**  
内存溢出：程序申请内存时没有足够空间；内存泄漏：不再使用的对象无法被回收，最终导致 OOM。
**3 分钟详答**  
- 内存溢出（OOM）：当堆、元空间、栈等内存区域无法满足分配需求时抛出，例如 java.lang.OutOfMemoryError: Java heap space。  
- 内存泄漏：对象不再被使用，但仍然被 GC Roots 直接或间接引用，导致无法回收，常见于静态集合、监听器未注销、连接未关闭等。  
- 排查步骤：  
  1. 通过 `jmap -dump:format=b,file=heap.hprof pid` 导出堆快照；  
  2. 使用 MAT、VisualVM 等工具分析，找出占用内存最大的对象；  
  3. 查看这些对象到 GC Roots 的引用链，定位代码位置。
**追问**  
- Q：如何区分是内存泄漏还是内存不足？  
  **A**：如果 Full GC 后老年代仍然持续增长，对象无法回收，多为内存泄漏；如果 GC 能回收大部分对象，只是堆本身不够用，则是内存配置不足。

# 五、综合与对比题
## 15. JVM、JRE、JDK 有什么区别？
**15 秒简答**  
JVM：Java 虚拟机，运行字节码；JRE = JVM + 核心类库；JDK = JRE + 工具（javac、javadoc 等）。
**3 分钟详答**  
- JVM：Java Virtual Machine，负责加载、验证、解释/编译字节码，屏蔽底层操作系统差异，是 Java 跨平台的基础。  
- JRE：Java Runtime Environment，包含 JVM + Java 核心类库，用来运行 Java 程序。  
- JDK：Java Development Kit，包含 JRE + 编译器（javac）、调试器、文档工具等，是开发环境。  
- 关系：JDK > JRE > JVM，一般开发安装 JDK，生产环境可只安装 JRE。
**追问**  
- Q：为什么说 Java 是“一次编写，到处运行”？  
  **A**：Java 源代码编译成字节码，运行在不同平台的 JVM 上，由 JVM 屏蔽底层差异，实现跨平台。
---
## 16. Java 是解释执行还是编译执行？
**15 秒简答**  
Java 既是编译型（javac 编译成 .class 字节码），又是解释型（JVM 解释执行字节码），同时支持 JIT 即时编译成机器码执行。
**3 分钟详答**  
- 前端编译：javac 将 .java 编译为 .class 字节码，这是静态编译。  
- JVM 执行方式：  
  - 解释器：逐条解释字节码执行，启动快，但执行慢；  
  - JIT 编译器：将热点代码编译成本地机器码，执行效率高，但编译消耗资源。  
- HotSpot 采用解释器 + JIT 混合模式：初始解释执行，对热点代码进行 JIT 编译，兼顾启动速度和运行性能。
**追问**  
- Q：什么是热点代码？  
  **A**：被频繁调用的方法或循环体，JVM 通过计数器探测这些代码，触发 JIT 编译，提高执行效率。
---
## 17. Java 中强引用、软引用、弱引用、虚引用分别是什么？
**15 秒简答**  
强引用：普通引用，只要存在就不会回收；软引用：内存不足时回收；弱引用：下次 GC 时回收；虚引用：无法通过引用访问对象，用于跟踪对象回收通知。
**3 分钟详答**  
- 强引用：`Object obj = new Object()`，只要 obj 还指向该对象，GC 就不会回收。  
- 软引用（SoftReference）：用于缓存，内存足够时保留，内存不足时会被回收。  
- 弱引用（WeakReference）：比软引用更弱，GC 工作时无论内存是否足够都会回收。  
- 虚引用（PhantomReference）：无法通过虚引用获取对象，必须配合引用队列使用，用于在对象被回收时收到一个系统通知。  
- 引用级别：强引用 > 软引用 > 弱引用 > 虚引用。
**追问**  
- Q：软引用和弱引用在实际开发中有哪些使用场景？  
  **A**：软引用常用于实现内存敏感的缓存；弱引用常用于 ThreadLocal、WeakHashMap 等结构，避免内存泄漏。

## 18. 堆和栈有什么区别？
**15 秒简答**  
栈：线程私有，存储局部变量和方法调用链，由系统自动分配和释放，速度快；堆：线程共享，存储对象实例，由 GC 管理，分配稍慢。
**3 分钟详答**  
- 栈：  
  - 线程私有，生命周期与线程相同；  
  - 每个方法调用创建一个栈帧，存储局部变量表、操作数栈、动态链接、方法出口等；  
  - 无需 GC，方法结束时自动弹栈释放。  
- 堆：  
  - 线程共享，JVM 启动时创建；  
  - 存放对象实例和数组，是 GC 的主要区域；  
  - 由垃圾收集器回收不再使用的对象。
**追问**  
- Q：局部变量表中的引用在栈还是堆？  
  **A**：引用变量本身在栈中，它指向的对象实例在堆中；如果是基本类型，则直接在栈中存储值。
---
## 19. 方法区/永久代/元空间之间是什么关系？
**15 秒简答**  
方法区是 JVM 规范逻辑区域；永久代是 HotSpot 对方法区的实现（JDK7 前后逐步废弃）；元空间是 JDK8 以后使用本地内存实现的方法区。
**3 分钟详答**  
- 方法区：JVM 规范定义，存储类信息、常量、静态变量等。  
- 永久代：HotSpot 在 JDK7 之前用永久代实现方法区，容易 OOM，受 -XX:MaxPermSize 限制。  
- JDK7：将字符串常量池、静态变量等移出永久代。  
- JDK8：完全废弃永久代，改用元空间（Metaspace），使用本地内存，只受本地内存限制，通过 -XX:MaxMetaspaceSize 控制。
**追问**  
- Q：元空间 OOM 通常是什么原因？  
  **A**：加载了过多类（如使用大量动态代理、JSP、框架生成类），或者元空间上限设置太小，可通过增大 -XX:MaxMetaspaceSize 或排查类加载泄漏解决。
---
## 20. 说说你对 JVM “Stop-The-World”（STW）的理解。
**15 秒简答**  
GC 时需要暂停所有应用线程，这段时间称为 Stop-The-World，用户线程完全停止，GC 线程工作。
**3 分钟详答**  
- STW 是大多数 GC 算法中不可避免的阶段，为了保证可达性分析的准确性，必须冻结所有应用线程。  
- 不同 GC 和不同阶段 STW 时间不同：Serial 和 Parallel GC 的 STW 时间相对较长，CMS 和 G1 尽量缩短 STW，但仍有短暂停顿。  
- 调优目标：在满足业务延迟要求的前提下，尽量减少 STW 时间和频率。  
- 过长或频繁的 STW 会导致服务响应变慢，甚至超时，需要通过 GC 日志和监控工具分析优化。
**追问**  
- Q：为什么可达性分析要在 STW 下进行？  
  **A**：如果应用线程和 GC 线程同时运行，引用关系会不断变化，分析结果不准确，可能漏标或错标对象，导致回收错误或内存泄漏。

## 21. 什么是 JIT 编译？逃逸分析？
**15 秒简答**  
JIT：即时编译，将热点字节码编译成本地机器码；逃逸分析：分析对象是否“逃逸”出方法或线程，以进行优化（如栈上分配、锁消除）。
**3 分钟详答**  
- JIT（Just-In-Time）编译器：  
  - HotSpot 对热点代码进行编译，生成高效机器码，后续直接执行，提升性能。  
  - 分为 C1（Client 编译器）和 C2（Server 编译器），分层编译时先用 C1 快速编译，再用 C2 深度优化。  
- 逃逸分析：  
  - 如果对象在方法内创建后不会被外部引用，则认为“不逃逸”，可进行栈上分配、同步消除、标量替换等优化，减少堆分配和 GC 压力。
**追问**  
- Q：逃逸分析失效会导致什么问题？  
  **A**：对象本可栈上分配，却仍分配在堆上，增加 GC 压力；锁消除失效会导致不必要的同步开销。
---
## 22. JVM 中对象分配内存的方式有哪些？
**15 秒简答**  
指针碰撞（Bump the Pointer）和空闲列表（Free List），取决于堆内存是否规整；并发分配时使用 CAS + TLAB。
**3 分钟详答**  
- 指针碰撞：内存规整时，JVM 用一个指针记录已用内存边界，分配对象时指针向空闲方向移动对象大小的距离。  
- 空闲列表：内存不规整时，JVM 维护一个记录可用内存块的列表，分配时从列表中找到足够大的空间。  
- 并发分配：  
  - 采用 CAS + 重试保证指针更新的原子性；  
  - 每个线程可以分配一块 TLAB（Thread Local Allocation Buffer），在 TLAB 中分配对象避免竞争。
**追问**  
- Q：TLAB 的优缺点是什么？  
  **A**：优点是分配对象时线程安全，无需加锁；缺点是可能造成内存“碎片化”，只在 TLAB 区域有效。

## 23. 你在项目中做过哪些 JVM 调优或问题排查？
**15 秒简答**  
结合场景回答：如调整 -Xms/-Xmx、选择合适的 GC（如 G1）、开启 GC 日志、使用 jstat/jstack 排查 Full GC 频繁或 CPU 飙高等问题。
**3 分钟详答**（示例回答）  
在实习项目中，我遇到过一个服务在高峰期频繁出现 Full GC，导致接口超时：  
1. 使用 `jstat -gcutil pid 1000` 监控 GC 情况，发现老年代持续增长，且 Full GC 频繁但回收效果差。  
2. 通过 `jmap -histo:live pid` 查看存活对象，发现某缓存集合占用大量内存。  
3. 分析代码发现该缓存未设置失效策略，导致内存泄漏；修改为使用 Caffeine 缓存并设置过期时间。  
4. 同时调整 -Xms 和 -Xmx 为相同值，避免堆动态扩缩容，并切换到 G1 收集器控制停顿时间。  
最终 Full GC 频率明显下降，系统稳定性提升。
**追问**  
- Q：如果让你现在给一个系统做 JVM 调优，你会从哪几步开始？  
  **A**：先确定性能指标（延迟/吞吐量），监控 GC 日志和系统指标，定位瓶颈（内存、GC、线程），再调整堆大小、GC 策略和代码逻辑。

## 24. 常见的 OOM 类型有哪些？
**15 秒简答**  
常见的有 java.lang.OutOfMemoryError: Java heap space、Metaspace、GC overhead limit exceeded、StackOverflowError 等。
**3 分钟详答**  
- Java heap space：堆内存不足，对象过多或过大。  
- Metaspace / PermGen space：类元信息过多，加载类过多或动态生成类过多。  
- GC overhead limit exceeded：应用花费过多时间做 GC 但效果很小。  
- StackOverflowError：线程请求栈深度超过允许值，常见于无限递归。  
- Direct buffer memory：NIO 使用直接内存过多，超出限制。
**追问**  
- Q：如何区分 StackOverflowError 和 OOM？  
  **A**：StackOverflowError 是线程栈深度不足，通常因递归调用过深；OOM 是堆、元空间等内存区域不足，两者抛出场景和排查方向不同。
---
## 25. 为什么需要 GC？能不能完全不进行垃圾回收？
**15 秒简答**  
需要 GC 自动回收不再使用的对象，避免手动释放内存带来的泄漏和错误；Java 程序运行过程中会不断创建对象，没有 GC 会很快耗尽内存，无法稳定运行。
**3 分钟详答**  
- Java 程序在运行过程中不断创建对象，如果没有自动内存管理：  
  - 开发者需要手动释放内存，容易忘记或释放错误，导致内存泄漏或崩溃；  
  - 程序复杂后，内存管理极其复杂。  
- GC 提供自动内存管理：  
  - 自动识别不再使用的对象，释放内存；  
  - 减少 OOM 和内存泄漏风险，提升开发效率和程序稳定性。  
- 虽然可以调用 `System.gc()` 建议 JVM 进行 GC，但并不保证一定执行，一般不建议依赖该方法。
**追问**  
- Q：哪些情况下 GC 仍然无法避免 OOM？  
  **A**：内存泄漏导致对象无法回收、堆/元空间配置过小、对象创建速度远快于 GC 速度等情况下，仍会 OOM。

## 26. 你是如何学习 JVM 的？
**15 秒简答**  
先看《深入理解 Java 虚拟机》系统学习基础，再结合牛客、知乎面经整理高频问题，辅以实战调优和工具（jstat、jmap、VisualVM）加深理解。
**3 分钟详答**（示例回答）  
我的学习路径大致是：  
1. 先通过《深入理解 Java 虚拟机》了解 JVM 整体结构、内存模型、GC、类加载等基础知识。  
2. 在牛客、知乎上看大量后端实习/校招面经，把 JVM 相关问题整理成问题清单，逐个查资料和写笔记。  
3. 自己写一些小程序，故意制造 OOM、栈溢出、频繁 Full GC 等场景，用 jstat、jmap、jstack、VisualVM 排查分析。  
4. 在实习项目中，结合真实业务问题（如缓存导致的内存泄漏、Full GC 频繁）进行实际调优，加深理解。  
这样既能应付面试，也能在实战中落地 JVM 知识。
**追问**  
- Q：你觉得 JVM 中哪一块最难理解？  
  **A**：通常 GC 算法和 JVM 内存模型（JMM）部分最抽象，需要结合实际日志和工具多实践；类加载和字节码部分也比较抽象，需要看源码或调试。
