### 1. 线程池的参数设置与进阶
你记录了线程池的工作流程，但面试官常问：**“实际项目中线程池参数怎么配置？”**

- **CPU 密集型任务**：`N + 1`（N 为 CPU 核心数）。尽量减少线程上下文切换。
- **IO 密集型任务**：`2N` 或者 `N / (1 - 阻塞系数)`。因为线程大部分时间在等待 IO，可以多开线程利用 CPU。
- **`execute()` vs `submit()` 的区别**：
  - `execute()` 用于提交不需要返回值的任务，无法捕获异常。
  - `submit()` 用于提交需要返回值的任务，返回 `Future` 对象，可以通过 `Future.get()` 获取结果或捕获异常。
- **线程池的关闭**：
  - `shutdown()`：不再接受新任务，但会执行完队列中已存在的任务。
  - `shutdownNow()`：不再接受新任务，并尝试中断正在执行的任务，清空队列。

### 2. JUC 并发工具类
你提到了 `ConcurrentHashMap`，但 JUC 中还有三个非常常用的辅助工具类，常用于多线程协作：

- **CountDownLatch（倒计时门闩）**：
  - 场景：一个线程需要等待其他 N 个线程执行完毕才能运行（例如：主线程等待所有服务初始化完成）。
  - 核心方法：`await()`（阻塞等待）、`countDown()`（计数减 1）。
- **CyclicBarrier（循环栅栏）**：
  - 场景：一组线程互相等待，直到所有线程都到达某一个屏障点，再同时继续执行（例如：多线程计算数据，最后合并结果）。它可以循环使用。
- **Semaphore（信号量）**：
  - 场景：控制同时访问特定资源的线程数量（例如：限流，只允许 3 个线程同时进行数据库写入）。
  - 核心方法：`acquire()`（获取许可）、`release()`（释放许可）。

### 3. 原子类的增强：LongAdder
你提到了 CAS 和原子类，但在高并发场景下，`AtomicInteger` 有个短板：高并发时，大量线程通过 CAS 自旋重试，会浪费大量 CPU 资源。

- **LongAdder（JDK 8 引入）**：
  - 原理：分散热点数据。`LongAdder` 内部维护了一个 `Cell` 数组（类似 ConcurrentHashMap 的分段锁思想），线程会去竞争不同的 Cell 进行累加，最后将所有 Cell 的值求和。
  - 适用场景：高并发统计场景（如计数器），优先使用 `LongAdder` 替代 `AtomicLong`。

### 4. 锁优化细节（JIT 层面）
除了锁升级，JIT 编译器还有其他锁优化手段：

- **锁消除**：JVM 通过逃逸分析，发现对象不可能被其他线程访问，则会直接消除锁（例如方法内的 `StringBuffer` 操作）。
- **锁粗化**：如果 JVM 检测到一系列连续操作都对同一个对象反复加锁解锁，它会将加锁范围扩大（粗化）到整个操作序列的外部（例如在循环体内加锁，粗化到循环体外）。

### 5. 异步编程：CompletableFuture
虽然不是纯“并发”机制，但现代 Java 面试非常看重异步编排能力，它是 JUC 的重要组成部分。

- 传统的 `Future` 获取结果必须阻塞（`get()` 方法），不支持回调。
- **CompletableFuture**：
  - 支持异步回调：`whenComplete`、`exceptionally`。
  - 支持任务编排：`thenApply`（A做完做B）、`thenCombine`（A和B都做完做C）、`applyToEither`（A和B谁快用谁）。
  - 这是现在解决“回调地狱”和实现高并发业务编排的标准工具。

### 6. AQS 的核心细节补充
虽然你提到了 ReentrantLock 基于 AQS，但可以补充一点关于 `Condition` 的知识：

- **Condition**：ReentrantLock 的 `newCondition()` 返回 `Condition` 对象。
- 作用：类似于 `synchronized` 的 `wait/notify` 机制，但更强大。它支持**多路通知**（例如：创建两个 Condition，一个通知生产者，一个通知消费者，避免“ signalAll” 唤醒所有线程导致的惊群效应）。
