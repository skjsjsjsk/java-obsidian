# 一、存储引擎与表设计
## 1. MySQL 常用存储引擎有哪些？MyISAM 和 InnoDB 有什么区别？
**15 秒简答**  
常见：InnoDB、MyISAM、Memory 等。  
核心区别：  
- InnoDB：支持事务、行级锁、外键、MVCC，默认引擎，适合写多、需要事务的场景。  
- MyISAM：不支持事务和行锁，只有表锁，读取性能好，适合只读或读多写少场景。  
**3 分钟详答**  
存储引擎是 MySQL 中真正负责“怎么存、怎么读”的底层模块，同一台 MySQL 上不同表可以使用不同引擎。常见的有：  
- InnoDB：MySQL 5.5 之后的默认存储引擎，支持 ACID 事务、行级锁、外键约束，通过 redo log 实现崩溃恢复，通过 MVCC 提高并发性能，是互联网公司最常用的选择。  
- MyISAM：早期默认引擎，不支持事务，只支持表级锁，适合读多写少的报表、日志类场景；在 `SELECT COUNT(*)` 这类操作上更快，因为它内部维护了总行数变量。  
- Memory：数据全在内存，用哈希索引，适合临时表、缓存场景。  
主要对比点：  
- 事务与崩溃恢复：InnoDB 支持，MyISAM 不支持。  
- 锁粒度：InnoDB 支持行级锁，MyISAM 只有表锁。  
- 外键：InnoDB 支持，MyISAM 不支持。  
- 全文索引：早期 MyISAM 支持更好，InnoDB 5.6+ 也支持。  
追问：  
1）你们项目为什么选 InnoDB？  
> 一般都是：需要事务、外键、高并发写，所以用默认的 InnoDB。

## 2. 什么是数据库三范式？你在建表时怎么应用？
**15 秒简答**  
1NF：字段不可再分。  
2NF：在 1NF 基础上，非主键字段完全依赖主键，不能只依赖主键的一部分（消除部分依赖）。  
3NF：在 2NF 基础上，非主键字段不能传递依赖于其他非主键字段（消除传递依赖）。  
**3 分钟详答**  
范式是关系数据库设计的规范，目的是减少冗余、避免更新异常：  
- 第一范式（1NF）：每个字段都是原子的，不可再分。比如“地址”字段不要拆成省市区存成一串。  
- 第二范式（2NF）：先满足 1NF，并且每个非主键字段完全依赖于整个主键，而不是只依赖主键的一部分。常见于联合主键场景，比如订单明细表（订单ID + 商品ID）中，商品名称应该只依赖商品ID，而不是订单ID。  
- 第三范式（3NF）：先满足 2NF，消除非主键字段对其他非主键字段的传递依赖。比如学生表中，学院名称可以通过学院ID 决定，就不应该再存学院名称。  
实际设计中，通常会：  
- 先按 3NF 设计，避免大量冗余。  
- 再根据查询性能做适度反范式，比如冗余一些统计字段、名称字段，减少联表查询。  
追问：  
1）你有没有做过反范式设计？  
> 有，比如在订单表中冗余“用户昵称”，避免频繁连表查询用户信息。

## 3. CHAR 和 VARCHAR 有什么区别？使用场景？
**15 秒简答**  
CHAR：定长，不足部分补空格，最大 255 字符。适合长度固定或接近固定的字段，如手机号、MD5 值。  
VARCHAR：变长，按实际长度存储，最大 65535 字节（受行总长度限制）。适合长度变化大的字段，如用户昵称、地址。  
**3 分钟详答**  
CHAR 和 VARCHAR 是最常用的字符串类型：  
- CHAR：  
  - 定义长度 `CHAR(n)` 表示 n 个字符。  
  - 存储时如果不足 n，右侧补空格，检索时再去除。  
  - 适合固定长度，如手机号、身份证号、状态码等。  
- VARCHAR：  
  - `VARCHAR(n)` 中 n 也是最大字符数，实际存储长度 = 实际字符数 + 1~2 字节长度前缀。  
  - 更新时可能发生行溢出（列过长会存到溢出页）。  
  - 适合长度变化大的字符串，如用户名、简介、URL。  
选择建议：  
- 长度基本固定，用 CHAR，减少碎片。  
- 长度变化较大或平均长度比最大长度小很多，用 VARCHAR，节省空间。  
追问：  
1）VARCHAR(50) 和 VARCHAR(255) 有什么区别？  
> 最大长度不同，存储时前者最多 50 字符，后者最多 255 字符；实际存储都是变长，但字段长度会影响索引键长度和排序缓冲区等。

## 4. 为什么要尽量避免 NULL 字段？
**15 秒简答**  
NULL 会占用额外存储空间（位图或额外字节），增加索引复杂度，影响查询优化器判断，还可能导致统计结果不准确。实际设计中尽量将字段设为 NOT NULL，并给默认值。  
**3 分钟详答**  
在 MySQL 中，NULL 的处理比非 NULL 复杂：  
- 存储层面：每行有一个 NULL 位图，标识哪些列是 NULL，占用额外空间。  
- 索引层面：索引中需要额外标记 NULL，索引条目更大，且优化器在选择索引时需要考虑 NULL 值的分布，判断更复杂。  
- 查询层面：`COUNT(col)` 会忽略 NULL 行，但 `COUNT(*)` 不会；`WHERE col = NULL` 永远为 NULL，要用 `IS NULL`。  
- 统计层面：含有 NULL 的列，计算 `COUNT(DISTINCT col)` 等时需要额外处理。  
实践中：  
- 尽量将字段定义为 `NOT NULL DEFAULT xxx`，比如默认空字符串、0 等。  
- 只有业务上确实“未知”或“不存在”的语义才用 NULL。  
追问：  
1）NULL 在索引里是怎么存的？  
> InnoDB 中，索引会存储 NULL 值的标记，每个索引条目额外多一个字节标识是否为 NULL。

# 二、索引与 SQL 优化
## 5. 什么是索引？为什么 MySQL 索引采用 B+ 树？
**15 秒简答**  
索引是帮助数据库高效查找数据的数据结构，类似书的目录。MySQL 默认使用 B+ 树索引，因为 B+ 树：  
- 树高度低（通常 2–4 层），磁盘 I/O 少。  
- 非叶子节点只存索引键，能存更多索引项。  
- 叶子节点按顺序链表连接，非常适合范围查询和排序。  
**3 分钟详答**  
索引本质上是通过“空间换时间”来加速查询的数据结构。MySQL 中索引是在存储引擎层实现的，不同引擎实现不同，InnoDB 默认使用 B+ 树。  
为什么不用二叉树 / 红黑树 / B 树，而用 B+ 树：  
- 二叉树 / 红黑树：  
  - 树高度大，几千万数据可能树高二十几层，意味着二十几次磁盘 I/O。  
  - 每个节点只存一个 key，对磁盘不友好。  
- B 树：  
  - 非叶子节点也存储数据，单页能存的索引项少，树高比 B+ 树高。  
  - 范围查询需要回溯到父节点，效率不如 B+ 树。  
- B+ 树：  
  - 非叶子节点只存索引键和指针，能存更多索引项，树更矮。  
  - 所有数据都放在叶子节点，叶子节点通过双向链表连接，范围查询、排序、全表扫描都非常高效。  
追问：  
1）B+ 树和 Hash 索引有什么区别？  
> Hash 索引只支持等值查询，不支持范围、排序和最左前缀；B+ 树支持范围、排序，适合大多数业务场景。

## 6. 什么是聚簇索引？什么是非聚簇索引 / 二级索引？
**15 秒简答**  
聚簇索引：数据行按主键顺序存储，主键索引的叶子节点就是整行数据，一张表只能有一个聚簇索引。  
非聚簇索引（二级索引）：叶子节点存储的是主键值，查询时需要先找到主键，再回聚簇索引查数据（回表）。  
**3 分钟详答**  
在 InnoDB 中：  
- 聚簇索引：  
  - 主键索引就是聚簇索引，数据行直接存在主键索引的叶子节点上。  
  - 如果没有主键，会选择一个唯一非空索引做聚簇索引；都没有则生成 6 字节的 ROWID 作为隐式主键。  
- 二级索引（非聚簇索引）：  
  - 叶子节点存储的是“索引列 + 主键值”。  
  - 查询时，先在二级索引树找到主键值，再去聚簇索引树查找整行数据，这个过程叫“回表”。  
MyISAM 中：  
- 主键索引和二级索引都是非聚簇索引，叶子节点存储的是数据行的物理地址（指针）。  
聚簇索引优点：  
- 主键查询非常快，一次 B+ 树查询就能拿到整行数据。  
- 数据物理顺序与主键逻辑顺序一致，范围查询、排序友好。  
追问：  
1）非聚簇索引一定回表吗？  
> 不一定，如果查询列都在二级索引中，就是“覆盖索引”，不需要回表。  
---
## 7. 什么是回表？什么是覆盖索引？
**15 秒简答**  
回表：通过二级索引查到主键值，再根据主键去聚簇索引查整行数据。  
覆盖索引：查询所需要的列全部包含在某个索引中，不需要回表，Extra 字段显示 “Using index”。  
**3 分钟详答**  
在 InnoDB 中：  
- 二级索引叶子节点存的是“索引列 + 主键值”。  
- 当 SQL 需要 `SELECT *` 或其他未在索引中的列时，需要再根据主键去聚簇索引查找，这就是回表。  
- 如果查询列都在某个索引中，比如在 `name` 上有索引，查询 `SELECT id, name FROM user WHERE name = '张三'`，name 索引已经包含了 id 和 name，就不需要回表，这就是覆盖索引。  
覆盖索引的好处：  
- 少一次 B+ 树查询，减少 I/O，提升性能。  
- 常见于统计查询：`SELECT COUNT(*) FROM user WHERE name = '张三'`，如果 name 有索引，就能走覆盖索引。  
追问：  
1）如何判断 SQL 是否用了覆盖索引？  
> 用 `EXPLAIN` 查看，如果 Extra 字段有 “Using index”，说明使用了覆盖索引。

## 8. 什么是最左前缀原则？索引什么时候会失效？
**15 秒简答**  
最左前缀原则：联合索引 `(a, b, c)`，查询条件要从 a 开始匹配，可以是 `a`、`a,b`、`a,b,c`，否则索引失效或部分失效。  
常见失效场景：  
- 条件中有 `OR`，且 OR 前后存在非索引列。  
- 对索引列做函数运算、表达式计算。  
- 隐式类型转换（字符串字段用数字查询）。  
- `LIKE '%abc'`。  
- `!=`、`<>`、`NOT IN` 等负向条件。  
**3 分钟详答**  
联合索引在 B+ 树中是按字段顺序排序的：先按 a 排序，a 相同再按 b 排序，b 相同再按 c 排序。因此：  
- `WHERE a = 1` 能走索引。  
- `WHERE a = 1 AND b = 2` 能走索引。  
- `WHERE b = 2` 或 `WHERE b = 2 AND c = 3` 无法使用该联合索引，因为缺少最左列 a。  
索引失效的常见原因：  
- 使用函数或计算：`WHERE UPPER(name) = 'ABC'`。  
- 隐式转换：字段是 `VARCHAR`，却用 `WHERE name = 123`。  
- 左模糊：`WHERE name LIKE '%abc'`。  
- `OR` 两侧只要有一侧没有索引，整个索引可能失效。  
- MySQL 优化器判断“走索引成本 > 全表扫描”，也会放弃索引。  
追问：  
1）联合索引 (a,b,c)，`WHERE a = 1 AND c = 3` 能用到索引吗？  
> 能用到 a 的部分，索引只用到 a；c 无法利用索引排序，需要回表过滤。

## 9. 索引设计有哪些原则？什么时候适合 / 不适合建索引？
**15 秒简答**  
适合建索引：  
- 频繁作为 WHERE / JOIN / ORDER BY / GROUP BY 的列。  
- 区分度高（选择性接近 1）。  
不适合建索引：  
- 数据量很小的表。  
- 频繁更新的字段。  
- 区分度极低的字段（如性别）。  
**3 分钟详答**  
索引不是越多越好，需要权衡查询性能与写入成本：  
设计原则：  
1. 在经常出现在 WHERE、JOIN ON、ORDER BY、GROUP BY 的列上建索引。  
2. 选择区分度高的列，比如用户名、订单号，而不是性别、状态。  
3. 尽量使用联合索引，减少索引数量，利用最左前缀覆盖多查询场景。  
4. 字符串较长可考虑前缀索引，比如 `VARCHAR(255)` 只对前 10 个字符建索引。  
5. 避免在频繁更新的列上建过多索引，写入时要维护索引。  
不适合建索引的场景：  
- 表数据量很小，全表扫描也很快。  
- 字段区分度极低，如性别、是否删除等。  
- 字段更新非常频繁，索引维护成本高。  
追问：  
1）你们项目上有没有建索引踩过坑？  
> 有，比如在状态字段上建索引，结果区分度太低，优化器仍然选择全表扫描，后来删掉了该索引。

## 10. 如何定位慢 SQL？怎么优化？
**15 秒简答**  
开启慢查询日志，设置阈值（如 2 秒），记录慢 SQL。  
用 `EXPLAIN` 查看执行计划，关注：  
- `type`：是否为 ALL（全表扫描）、index（全索引扫描）。  
- `key`：是否真正用到了索引。  
- `rows`：预估扫描行数。  
- `Extra`：是否有 `Using filesort`、`Using temporary` 等。  
**3 分钟详答**  
定位慢 SQL 的步骤：  
1. 配置慢查询日志：  
   - `long_query_time` 设置阈值。  
   - `slow_query_log` 开启日志。  
2. 通过日志或监控系统（如 SkyWalking）找到慢 SQL。  
3. 使用 `EXPLAIN` 分析执行计划：  
   - `type`：`system > const > eq_ref > ref > range > index > ALL`，尽量达到 ref 以上。  
   - `key`：实际使用的索引。  
   - `key_len`：使用的索引长度，联合索引可判断用了多少列。  
   - `rows`：预估扫描行数。  
   - `Extra`：`Using index` 表示覆盖索引，`Using filesort` 表示需要额外排序，`Using temporary` 表示用到临时表。  
优化思路：  
- 加合适的索引，避免全表扫描。  
- 避免在索引列上做函数、计算。  
- 只返回需要的列，少用 `SELECT *`。  
- 大分页优化：延迟关联，先查主键再回表。  
追问：  
1）EXPLAIN 中的 `rows` 是准确值吗？  
> 不是，是优化器的估算值，基于统计信息，可能不准确。

## 11. 一条 SELECT 查询 SQL 的执行流程是怎样的？
**15 秒简答**  
客户端 → 连接器（认证）→ 查询缓存（8.0 之前）→ 分析器（词法/语法分析）→ 优化器（选执行计划、索引）→ 执行器（调用存储引擎接口）→ 返回结果。  
**3 分钟详答**  
以一条 `SELECT * FROM t WHERE id = 10` 为例：  
1. 连接器：验证用户名、密码、权限，获取连接。  
2. 查询缓存：MySQL 8.0 之前会查看缓存，命中则直接返回；8.0 以后缓存被移除，因为命中率低且维护成本高。  
3. 分析器：  
   - 词法分析：识别出 `SELECT`、`FROM t`、`WHERE id = 10`。  
   - 语法分析：检查语法是否正确。  
4. 优化器：  
   - 决定使用哪个索引。  
   - 多表连接时决定连接顺序。  
   - 选择成本最低的执行计划。  
5. 执行器：  
   - 检查是否有权限。  
   - 调用存储引擎接口，根据索引或全表扫描查找满足条件的行。  
6. 存储引擎：返回结果给执行器，再返回给客户端。  
追问：  
1）为什么 MySQL 8.0 移除查询缓存？  
> 表更新会清空相关缓存，命中率低；且并发环境下锁竞争严重，得不偿失。

## 12. 什么是索引下推（ICP）？
**15 秒简答**  
索引下推是指 MySQL 将 WHERE 条件中“索引能过滤的部分”下推到存储引擎层处理，而不是全返回给 Server 层再过滤。  
好处：减少回表次数和上层交互次数，提升查询性能。  
**3 分钟详答**  
以联合索引 `(name, age)` 为例：  
- SQL：`SELECT * FROM user WHERE name LIKE '张%' AND age = 20`。  
- 没有 ICP 时：  
  - 存储引擎通过 name 索引找到所有 `name LIKE '张%'` 的主键，回表查出整行数据，再给 Server 层过滤 `age = 20`。  
- 有 ICP 时：  
  - 存储引擎在索引内部就利用 `age` 过滤，只返回满足 `age = 20` 的记录，减少回表次数。  
ICP 主要适用于：  
- 联合索引。  
- 条件中有索引列，但不符合最左前缀，或需要回表查询的场景。  
追问：  
1）如何判断是否使用了索引下推？  
> `EXPLAIN` 的 Extra 字段中有 `Using index condition`。

# 三、事务与锁
## 13. 什么是事务？ACID 特性分别是什么？
**15 秒简答**  
事务是一组数据库操作，要么全部成功，要么全部失败，是保证数据一致性的基本单元。  
ACID：  
- 原子性（A）：事务不可分割，要么全做，要么全不做。  
- 一致性（C）：事务前后，数据库保持一致性状态。  
- 隔离性（I）：多个事务并发执行时，互不干扰。  
- 持久性（D）：事务提交后，对数据的修改是永久的。  
**3 分钟详答**  
事务是数据库中区别于普通“单条 SQL”的重要概念：  
- 原子性：通过 undo log 实现，回滚时将数据恢复到修改前状态。  
- 一致性：由原子性、隔离性、持久性以及业务逻辑共同保证。  
- 隔离性：通过锁和 MVCC 实现，防止并发事务互相干扰。  
- 持久性：通过 redo log 实现，提交前先写日志，宕机后可恢复。  
事务常用语句：  
- `START TRANSACTION` / `BEGIN`。  
- `COMMIT`。  
- `ROLLBACK`。  
追问：  
1）你们项目中用事务的场景有哪些？  
> 比如下单：扣库存、生成订单、扣减账户余额，必须放在一个事务中，保证要么全部成功，要么全部回滚。

## 14. MySQL 有哪些事务隔离级别？默认是什么？
**15 秒简答**  
四个隔离级别：  
- READ UNCOMMITTED：读未提交，可能脏读。  
- READ COMMITTED：读已提交，避免脏读，可能不可重复读。  
- REPEATABLE READ：可重复读，MySQL 默认级别，避免脏读和不可重复读，InnoDB 在很大程度上避免幻读。  
- SERIALIZABLE：串行化，避免所有问题，并发性能最差。  
**3 分钟详答**  
隔离级别越高，并发性能越低，问题越少：  
- READ UNCOMMITTED：可能读到其他事务未提交的数据（脏读）。  
- READ COMMITTED：只能读到已提交的数据，避免脏读，但同一事务中两次读取可能结果不同（不可重复读）。  
- REPEATABLE READ：同一事务中多次读取同一行数据，结果一致，InnoDB 通过 MVCC 和 Next-Key Lock 在很大程度上避免幻读。  
- SERIALIZABLE：强制事务串行执行，完全避免脏读、不可重复读、幻读，但并发度最低。  
MySQL 默认隔离级别是 REPEATABLE READ（可重复读）。  
追问：  
1）什么是脏读、不可重复读、幻读？  
> 脏读：读到未提交的数据。不可重复读：同一事务两次读同一行，结果不同（被其他事务修改并提交）。幻读：同一事务两次范围查询，结果集数量不同（被其他事务插入或删除）。

## 15. MySQL 是如何实现事务的 ACID 的？
**15 秒简答**  
- 原子性：通过 undo log，回滚时恢复旧值。  
- 持久性：通过 redo log，WAL 机制，保证提交后不丢失。  
- 隔离性：通过锁 + MVCC 实现并发控制。  
- 一致性：由原子性、隔离性、持久性以及约束共同保证。  
**3 分钟详答**  
MySQL InnoDB 通过多种机制实现 ACID：  
- 原子性：  
  - 事务修改数据时，将旧值写入 undo log。  
  - 回滚时根据 undo log 恢复数据。  
- 持久性：  
  - 采用 WAL（Write-Ahead Logging），修改先写 redo log，再写数据页。  
  - 宕机重启后通过 redo log 重放已提交事务，保证不丢数据。  
- 隔离性：  
  - 读操作通过 MVCC 在某些版本下实现非阻塞读。  
  - 写操作通过行锁、间隙锁等控制并发。  
- 一致性：  
  - 由 A、I、D 以及数据库约束（主键、外键、非空等）共同保证。  
追问：  
1）redo log 和 binlog 有什么区别？  
> redo log 是 InnoDB 引擎层物理日志，用于崩溃恢复；binlog 是 Server 层逻辑日志，用于主从复制和备份恢复。

## 16. 什么是 MVCC？它的作用是什么？
**15 秒简答**  
MVCC（多版本并发控制）通过保存数据的历史版本，实现读写互不阻塞，提高并发性能。在 InnoDB 中，READ COMMITTED 和 REPEATABLE READ 两个隔离级别下使用 MVCC。  
**3 分钟详答**  
MVCC 的核心思想：  
- 每行数据保存两个隐藏列：创建时间版本号（DB_TRX_ID）和删除时间版本号（DB_ROLL_PTR）。  
- 事务根据版本号判断哪一行对自己可见，实现非阻塞读。  
在 REPEATABLE READ 下：  
- 事务启动时拍一个“快照”，后续读都基于这个快照。  
- 更新操作会写 undo log，通过回滚指针构建版本链。  
MVCC 好处：  
- 读不加锁，写加行锁，读写互不阻塞。  
- 避免脏读、不可重复读，在一定程度上避免幻读。  
追问：  
1）MVCC 能完全避免幻读吗？  
> 在 REPEATABLE READ 下，快照读通过 MVCC 避免幻读；当前读（`SELECT ... FOR UPDATE`）通过 Next-Key Lock 避免幻读。

## 17. MySQL 有哪些锁？InnoDB 的行锁是怎么实现的？
**15 秒简答**  
按粒度：表锁、行锁、页锁。  
按类型：共享锁（S）、排他锁（X）。  
InnoDB 默认使用行级锁，通过给索引项加锁实现；如果没有索引，行锁会退化为表锁。  
**3 分钟详答**  
MySQL 中常见的锁：  
- 表锁：开销小，加锁快，不会死锁，但并发度低。  
- 行锁：开销大，加锁慢，会死锁，并发度高。  
- 页锁：介于表锁和行锁之间。  
InnoDB 的行锁：  
- 是通过给索引上的记录加锁实现的，如果 SQL 没有走索引，行锁会升级为表锁。  
- 行锁类型：Record Lock（记录锁）、Gap Lock（间隙锁）、Next-Key Lock（记录锁 + 间隙锁）。  
共享锁（S）：  
- `SELECT * FROM t WHERE id = 1 LOCK IN SHARE MODE`。  
- 允许其他事务也加 S 锁，但不能加 X 锁。  
排他锁（X）：  
- `SELECT * FROM t WHERE id = 1 FOR UPDATE`。  
- 其他事务不能再加任何锁。  
追问：  
1）什么情况下 InnoDB 会升级为表锁？  
- 事务中更新字段没有索引，或索引失效，导致全表扫描，行锁退化为表锁。

## 18. 什么是死锁？怎么避免？
**15 秒简答**  
死锁是指两个或多个事务互相持有对方需要的锁，导致互相等待。  
避免方法：  
- 尽量按固定顺序访问表和行。  
- 避免长事务，及时提交。  
- 尽量使用索引，避免全表扫描导致的表锁。  
**3 分钟详答**  
死锁经典场景：  
- 事务 A 先锁 id=1，再锁 id=2；事务 B 先锁 id=2，再锁 id=1，就可能死锁。  
排查与解决：  
- 通过 `SHOW ENGINE INNODB STATUS` 查看死锁日志。  
- 业务侧按固定顺序加锁，比如都按 id 升序处理。  
- 尽量减少事务持有锁的时间。  
追问：  
1）你遇到过死锁吗？怎么定位和解决？  
> 有，在订单处理中，先查再更新，并发时出现死锁。后来统一按主键顺序处理，并给相关字段加索引，减少锁范围。

## 19. 什么是乐观锁和悲观锁？怎么在 MySQL 中实现？
**15 秒简答**  
乐观锁：假设冲突很少，在更新时检查版本号或时间戳，适合读多写少。  
悲观锁：假设冲突很多，每次操作都加锁，适合写多竞争激烈场景。  
**3 分钟详答**  
- 悲观锁：  
  - 数据库层面：`SELECT ... FOR UPDATE`，加行锁。  
- 乐观锁：  
  - 在表中加版本号字段 `version`，每次更新时：  
    - 读取数据时读出版本号。  
    - 更新时带上 `WHERE version = old_version`，同时 `version = version + 1`。  
  - 如果更新行数为 0，说明被其他事务修改过，需要重试或报错。  
适用场景：  
- 乐观锁：读多写少，冲突概率小，如商品详情页。  
- 悲观锁：写多且竞争激烈，如秒杀场景，需要强一致性。  
追问：  
1）你项目中用过乐观锁吗？  
> 用过，比如在更新库存时，用 `UPDATE stock SET num = num - 1, version = version + 1 WHERE id = 1 AND version = oldVersion`。

# 四、日志与主从复制
## 20. redo log、undo log、binlog 有什么区别和作用？
**15 秒简答**  
- redo log：InnoDB 引擎层物理日志，用于崩溃恢复，保证持久性。  
- undo log：回滚日志，记录修改前的数据，用于回滚和 MVCC。  
- binlog：Server 层逻辑日志，用于主从复制和数据备份。  
**3 分钟详答**  
三者是 MySQL 面试常考的“三大日志”：  
- redo log：  
  - 固定大小循环写入，空间满时会触发刷盘。  
  - 修改数据先写 redo log，再写数据页，宕机后可恢复。  
- undo log：  
  - 记录数据修改前的值，用于回滚。  
  - 同时用于 MVCC，构建历史版本。  
- binlog：  
  - 以事件形式记录所有修改数据的 SQL，有 STATEMENT、ROW、MIXED 三种格式。  
  - 用于主从复制、时间点恢复。  
区别：  
- 实现层次：redo log/undo log 属于 InnoDB 引擎；binlog 属于 Server 层。  
- 写入方式：redo log 是循环写；binlog 是追加写。  
- 用途：redo log 用于崩溃恢复；binlog 用于主从复制和备份。  
追问：  
1）为什么 redo log 和 binlog 要两阶段提交？  
> 保证两个日志的一致性，避免主从不一致或恢复数据缺失。  
---
## 21. MySQL 主从同步的原理是什么？
**15 秒简答**  
主库将数据变更写入 binlog，从库的 I/O 线程将 binlog 拉取到本地的 relay log，SQL 线程重放 relay log 中的事件，从而保持数据一致。  
**3 分钟详答**  
主从复制流程：  
1. 主库事务提交时，将修改记录写入 binlog。  
2. 从库 I/O 线程连接主库，读取 binlog，写入 relay log。  
3. 从库 SQL 线程读取 relay log，执行其中的 SQL，保持与主库一致。  
关键点：  
- 异步复制：主库提交后立即返回，不等待从库同步，可能丢数据。  
- 半同步复制：至少一个从库确认收到 binlog 后，主库才返回。  
常见问题：  
- 主从延迟：从库 SQL 线程单线程回放，容易成为瓶颈。  
- 解决：开启并行复制、分库分表、使用缓存等。  
追问：  
1）主从延迟对业务有什么影响？  
> 写主库后立刻读从库，可能读不到最新数据，需要考虑读主库或延迟读。

# 五、实战场景与综合题
## 22. 大表怎么优化？
**15 秒简答**  
从设计、索引、架构多方面入手：  
- 合理设计字段类型，避免不必要的 NULL。  
- 建立合适的索引，优化查询。  
- 分库分表：水平分表、垂直分表，或使用分布式数据库。  
- 引入缓存：减少数据库访问。  
**3 分钟详答**  
大表优化思路：  
1. 表设计优化：  
   - 选择合适的数据类型，比如用 `INT` 而不是 `VARCHAR` 存 ID。  
   - 避免过多字段，可垂直拆分为主表和扩展表。  
2. 索引优化：  
   - 避免冗余索引，定期清理无用索引。  
   - 使用覆盖索引减少回表。  
3. 分库分表：  
   - 垂直分库：按业务拆库，用户、订单、商品分不同库。  
   - 水平分表：按时间、ID 哈希等拆分多张表。  
4. 架构优化：  
   - 使用读写分离，减轻主库压力。  
   - 引入缓存（Redis），保护数据库。  
追问：  
1）你有没有做过分库分表？怎么分？  
> 有，比如按用户 ID 哈希水平分表，分 16 张表，用户 ID % 16 路由到对应表。

## 23. 慢 SQL 优化的完整流程？
**15 秒简答**  
1. 开启慢查询日志，定位慢 SQL。  
2. 使用 `EXPLAIN` 查看执行计划。  
3. 检查索引是否失效、扫描行数是否过多。  
4. 优化 SQL：加索引、改写 SQL、避免计算列、减少 `SELECT *`。  
5. 必要时调整表结构或拆分。  
**3 分钟详答**  
慢 SQL 优化是一个系统工程：  
- 发现慢 SQL：  
  - 通过慢查询日志、监控系统。  
- 分析原因：  
  - 是否没有索引。  
  - 是否索引失效（函数、隐式转换、左模糊等）。  
  - 是否扫描行数过多。  
- 优化方案：  
  - 加合适的索引。  
  - 优化 SQL 写法，比如避免 `SELECT *`，避免在索引列上做计算。  
  - 大分页使用“延迟关联”：先查主键，再回表。  
  - 事务尽量小，避免长事务。  
追问：  
1）遇到过哪些典型的慢 SQL？  
> 比如 `UPDATE` 全表扫、`LIKE '%abc'` 导致索引失效，通过加索引和改写 SQL 解决。

## 24. 自增主键用完了怎么办？
**15 秒简答**  
自增主键达到上限后，插入会报错。解决：  
- 提前将主键类型从 `INT` 改为 `BIGINT`。  
- 或设计新的主键生成策略，比如雪花算法，或使用业务唯一键。  
**3 分钟详答**  
`INT` 自增主键最大值为 2147483647，业务增长后可能不够用：  
- 提前规划：  
  - 使用 `BIGINT`，最大值 9223372036854775807，几乎不会用完。  
- 已经用完：  
  - 需要修改表结构，将主键改为 `BIGINT`，这在大表上是重操作，建议在低峰期使用在线改表工具。  
- 替代方案：  
  - 使用业务主键，比如订单号、用户 ID。  
  - 使用分布式 ID 生成方案（雪花算法、号段模式等）。  
追问：  
1）为什么推荐使用自增主键？  
- 插入时追加，减少页分裂。  
- 主键索引是聚簇索引，数据物理有序，查询和范围性能好。

## 25. MySQL 如何实现高可用？
**15 秒简答**  
常见方案：主从复制 + 读写分离、MHA / Orchestrator 等高可用组件、MGR（MySQL Group Replication）、分库分表 + 多活架构。  
**3 分钟详答**  
高可用目标是：尽量减少服务不可用时间，数据尽量不丢。常见做法：  
- 主从复制 + 故障切换：  
  - 一主多从，主库宕机时将从库提升为主库。

- MySQL Group Replication（MGR）：  
  - 基于 Paxos 的多主复制集群，支持自动故障转移。  
- 分库分表 + 多活：  
  - 不同机房部署多套数据库，通过中间件做读写路由。  
追问：  
1）你们项目的数据库架构是怎样的？  
> 主从架构：一主两从，应用写主库，读从库，通过中间件或配置读写分离。

## 26. SQL 语句中 WHERE 和 HAVING 有什么区别？
**15 秒简答**  
WHERE 在分组前过滤行，不能使用聚合函数。  
HAVING 在分组后过滤分组，可以使用聚合函数。  
**3 分钟详答**  
- WHERE：  
  - 作用于单行，过滤不满足条件的记录。  
  - 不能写 `COUNT()`、`SUM()` 等聚合函数。  
- HAVING：  
  - 作用于分组，过滤不满足条件的分组。  
  - 常见用法：`HAVING COUNT(*) > 10`。  
执行顺序：  
`WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT`。  
追问：  
1）能用 WHERE 的地方尽量用 WHERE 还是 HAVING？  
> 尽量用 WHERE，先过滤再分组，减少数据量，性能更好。

## 27. UNION 和 UNION ALL 有什么区别？
**15 秒简答**  
UNION 会合并并去重，UNION ALL 不去重。  
UNION ALL 性能更高，因为省去去重排序步骤。  
**3 分钟详答**  
- UNION：  
  - 对多个 SELECT 结果集进行合并，并去除重复行。  
  - 需要额外排序、去重，性能开销较大。  
- UNION ALL：  
  - 直接合并多个结果集，不去重。  
  - 性能更好，适合确定没有重复或允许重复的场景。  
使用建议：  
- 如果确定没有重复或允许重复，优先使用 UNION ALL。  
追问：  
1）UNION 会使用索引吗？  
- UNION 内部会对结果集做排序和去重，可能会使用临时表或文件排序，索引利用率取决于子查询本身。

## 28. COUNT(*)、COUNT(1)、COUNT(字段) 有什么区别？
**15 秒简答**  
- `COUNT(*)`：统计总行数，包含 NULL。  
- `COUNT(1)`：与 `COUNT(*)` 基本一致，性能差异可忽略。  
- `COUNT(字段)`：统计该字段非 NULL 的行数。  
**3 分钟详答**  
在 InnoDB 中：  
- `COUNT(*)` 和 `COUNT(1)`：  
  - 都是统计全表行数，不会忽略 NULL。  
  - MySQL 优化器会自动选择最小的非聚簇索引来扫描，不一定走主键索引。  
- `COUNT(字段)`：  
  - 忽略该字段为 NULL 的行。  
  - 如果字段上没有索引，可能会全表扫描并过滤 NULL。  
性能上：  
- 通常 `COUNT(*)` 和 `COUNT(1)` 差距极小。  
- 如果只需统计总数，优先使用 `COUNT(*)`。  
追问：  
1）`COUNT(DISTINCT col)` 呢？  
> 统计该列不同值的数量，忽略 NULL，需要排序或哈希去重，性能开销较大。

## 29. TRUNCATE、DELETE、DROP 有什么区别？
**15 秒简答**  
- DELETE：DML，按条件删除，可回滚，逐行删除，记录日志。  
- TRUNCATE：DDL，快速删除全表数据，不记录单行日志，不可回滚，自增主键重置。  
- DROP：DDL，删除表结构和数据，释放空间，不可回滚。  
**3 分钟详答**  
- DELETE：  
  - 可以带 WHERE 条件删除部分数据。  
  - 删除操作记录 undo log，可以回滚。  
  - 逐行删除，删除大量数据时较慢。  
- TRUNCATE：  
  - 一次性删除全表数据，释放数据页。  
  - 不记录单行日志，回滚依赖 redo log/undo log 的机制较特殊，一般视为不可回滚。  
  - 自增主键重新计数。  
- DROP：  
  - 删除表结构及其数据、索引、约束等。  
  - 空间立即释放。  
追问：  
1）删除全表数据用哪种方式更快？  
> TRUNCATE 比 DELETE 快很多，因为它是 DDL，不记录单行日志。

## 30. 如何防止 SQL 注入？
**15 秒简答**  
核心：不要信任用户输入，永远不要直接拼接 SQL。  
常用做法：  
- 使用预编译语句（PreparedStatement）和参数化查询。  
- 对用户输入进行校验和过滤。  
- 使用 ORM 框架，避免手写拼接 SQL。  
**3 分钟详答**  
SQL 注入是指用户输入中嵌入 SQL 片段，改变原有 SQL 语义。  
防御方法：  
1. 预编译：  
   - 使用 `PreparedStatement`，先发送带占位符的 SQL，再传参数。  
   - MySQL 会将参数视为数据，而不是可执行 SQL。  
2. 输入校验：  
   - 比如年龄只能是数字，用户名只能是字母数字等。  
3. 最小权限原则：  
   - 应用账号只给必要的权限，避免 `DROP`、`GRANT` 等高危权限。  
4. 使用 ORM：  
   - 框架内部一般会使用预编译，减少手写 SQL。  
追问：  
1）你项目中如何防范 SQL 注入？  
> 使用 MyBatis 或 Hibernate 的预编译机制，所有参数通过 `#{}` 占位符传递，禁止拼接 SQL。
