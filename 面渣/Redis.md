- redis有哪些数据结构: redis有字符串, 哈希, 列表, 集合和有序集合, 还有用于位操作的Bitmap, 用于基数统计的HyperLogLog以及用来存储和查找地理位置坐标的GEO
	- 字符串是最基本的数据类型, 它可以用来存储二进制串, 数字等等, 最大容量不超过512KB, 常可以用来存储token,验证码等等
	- 列表是有序的数据类型, 它可以在头/尾节点插入或者删除数据, 常可用做消息队列
	- 哈希是用来存储键值对的, 格式为key filed value, 通常用来存储用户信息, 商品信息等等
	- 集合是无序且不可重复的, 可以进行求交集, 并集等操作, 可用来计算共同关注等等
	- 有序集合, 相比于集合多了一列score, 每一个元素根据自己的分数排列, 支持范围查询, 可以用来当排行榜, 热门榜单等等
	- Bitmap将一串二进制数组存储在一块连续的内存里面, 每一位的变化代表一个元素的变化, 可以用来做签到, 用户是否活跃等等
	- HyperLogLog是一种用来统计一个集合中不重复元素个数的概率性数据结构, 可以用于大型网站的UV统计
	- GEO用来存储和查找地理位置坐标, 可以计算两点之间的距离, 用于外卖员与上架之间的距离或者打车等等
- redis为什么这么快呢:
	- redis数据存储在内存中, 内存的读写速度本来就比磁盘高几个量级
	- redis是单线程, 使用了IO多路复用技术的事件驱动模型, 是得一条线程可以连接成千上万个客服端, 避免了传统一条线程只能连接一个客服端的性能开销
	- redis对底层的数据结构做了极致的优化, 比如string的底层数据结构动态字符串它支持动态扩容以及预分配冗余空间
- 常用的持久化方式: RDB与AOF, RDB是存储内存中数据某一时间的快照, AOF是将redis的写命令记录下来用来恢复数据
	- RDB: 每隔一段时间就会将内存中的数据保存在磁盘中的RDB文件中,当redis重启时, 根据该RDB快照快速恢复数据, 但是缺点是, 可能会导致一段时间的数据消息, 如果redis宕机前没有保存快照
		- 持久化有两种方式, 一种通过save与bgsave命令, 另一种通过配置文件中的save命令自动执行
			- save命令会阻塞redis进程,直到RDB文件创建完成, bgsave会fork一个子进程去执行RDB文件的创建, 不阻塞主进程
			- 啥情况触发了RDB: 
				- 配置文件中设置了RDB持久化参数`save <seconds> <changes>`, 多少秒内有多少个key变化就会触发
				- 主从复制时, 当子节点第一次连接到主节点时, 主节点会自动执行bgsave命令创建出一个RDB文件传送给子节点
				- 如果没有启用AOF, 会在redis执行shutdown命令自动保存RDB快照
	- AOF: 将redis的写命令追加到磁盘中的AOF文件中, 当redis重启时, 重做该命令恢复文件, 缺点恢复慢
		- AOF会先将写命令保存在AOF缓存区, 然后再根据同步策略将它写入到AOF文件中
			- 同步策略: always: 每一次写命令都立即写入, everysec: 每秒调用一次fsync, 将多条命令一次性写入, no: 不刷盘, 根据操作系统决定
	- AOF与RDB的混合使用: 兼顾了两者的优点, 在写入AOF文件时, 先将此刻的RDB快照保存的AOF文件的头部,然后再将接下来的写命令追加到AOF文件的尾部, 这样redis重启恢复数据时, 可以先加载RDB快照恢复大量数据, 然后再重放写命令恢复剩下的数据, 既保证了完整性, 也加快了恢复速度
- 缓存穿透, 缓存击穿, 雪崩: 
	- 缓存穿透: 就是缓存中的一个热点key过期, 导致一段时间内大量请求直接涌向数据库, 给数据库造成很大的压力
		- 互斥锁: 当请求到达数据库时, 给该请求的线程加锁, 使他重建缓存, 其他请求则阻塞等待
		- 逻辑过期: 缓存不设置过期时间, 只设置一个逻辑过期时间, 当该缓存逻辑上过期时, 查询返回旧值的同时异步启动一个线程去更新缓存
	- 缓存击穿: 有大量请求访问数据库不存在的数据, 导致请求直接落在数据库, 数据库压力很大
		- 布隆过滤器: 将所有的数据hash在布隆过滤器里面, 请求到来时先查询布隆过滤器看是否存在, 如果不存在, 则返回不存在, 如果存在, 再查询数据库
		- 返回空值null: 当请求到达数据库时, 如果数据不存在, 直接返回一个null存储再redis中, 并设置一个合理的过期时间, 这样下回查询就会直接从数据库返回
	- 雪崩: 有大量的缓存同时过期, 导致请求在一段时间内会直接访问数据库, 导致数据库压力巨大, 甚至导致系统崩溃
		- 给缓存的过期时间增加一个随机值, 防止同时过期
		- 建立高可用的集群: 用redis cluster构建多节点集群, 确保数据在每一个节点上都有备份, 且支持故障数据自动转移
		- 缓存正常但是请求超过了缓存的承载能力, 方法是限流和降级措施
- 说说布隆过滤器: 布隆过滤器是一种空间效率利用极高的数据结构, 它用来快速查找某个元素是否在集合中, 特点是内存消耗极小
	- 底层是一个位数组和多个哈希函数, 位数组初始化全为0, 添加元素时, 通过哈希函数映射位置, 相对应的位置变为1, 查找时, 也通过哈希函数映射查找, 如果有一个为0, 则该元素不存在
	- 用来解决缓存穿透的
	- 有误判的, 因为哈希冲突, 多个不同的元素可能也会映射到相同的位置, 随着元素的增加, 位数组里面1越来越多, 误判概率也增大
- 缓存和数据库一致性: 查找时, 先查redis, 如果不存在再差MySQL; 更新时, 先更新MySQL, 再删除redis
	- 为什么先更新MySQL, 再删除redis呢: 因为如果先删除redis,再更新MySQL, 会导致数据不一致, 比如, 线程A要更新MySQL了, 那么它删除了缓存, 此时线程B要查询该数据, 发现缓存没有, 然后就从MySQL中查到然后再存储到redis中, 然后线程B结束了, 然后线程A也更新完成了, 那么此时MySQL中是新数据, redis中是旧数据
	- 如果要保证高度一致性呢: 
		- 使用消息队列: 当更新事务开始时, 先向MQ发送一条本地消息记录, 然后开始更新, 当事务提交以后, 再异步给MQ发送一条删除缓存的消息
			- 即使删除失败, MQ的重试机制也能保证最终的数据一致性
		- 使用Canal订阅binlog的变化, 当binlog发生变化时, 将变化的数据提交给消息队列, 消费者消息监听到变化后去删除缓存 
			- 这使得业务逻辑与缓存维护完全解耦
		- 一致性要求没有那么高可以使用延迟双删, 就是第一次删除缓存后, 在更新以后, 隔一段时间再删除一次缓存, 要异步启动一个线程去执行第二次删除缓存的代码, 因为它要执行睡眠操作, 这样不会阻塞主进程
- 淘汰策略: 八种:
	- noeviction: 内存不足时直接报错, 不删除任何key
	- allkeys-lru: 删除最近最少使用的key, 保证热点信息
	- allkeys-lfu: 删除使用次数最少的key, 维持系统长时间运行
	- allkeys-random: 随机删除一些key
	- volatile-lru: 同上
	- volatile-lfu: 同上, 保证重要数据不被淘汰
	- volatile-ttl: 删除离过期时间最近的key
	- volatile-random: 同上
- key的过期策略: 
- redis能实现分布式锁吗: 使用set key value nx px timeout来实现, redis执行时会把该语句视为原子指令, 同时由于nx是只有没有这个key时才会创建锁, 而redis也是单线程的, 保证了只有一个线程运行
	- 问题: 可能会误删锁, 线程A用该语句获取了锁, 然后该线程的执行周期比较长, 还没执行完锁过期了, 此时线程B进来获取锁成功, 然后线程A执行完了, 会尝试删除锁, 此时删除的就是线程B的锁
	- 解决: 通过锁的自动续期机制解决锁过期的问题, 比如Redisson的看门狗机制, 可以在后台启动一个定时任务, 用来检查该锁是否还是被该线程持有, 如果是则延迟过期时间
	- Redisson:
		- 提供了很多数据结构与服务，最常用的就是Redisson分布式锁，它可以让我们获取锁的时候不用手动输入过期时间，这就是看门狗机制
		- 看门狗机制: 当调用lock()方法获取锁时，如果没有显示传入过期时间，那么就会默认添加30s的过期时间，同时启动一个看门狗机制，就是启动一个定时任务，每隔10s（1/3的过期时间）检查锁是否还被当前线程持有，如果持有，则延长过期时间30s，   然后调用unlock()时，看门狗任务会被取消，如果忘记了unlock()也不影响，看门狗也会帮我们自动检查锁，如果锁已经不属于当前线程了，也会自动停止续期。
