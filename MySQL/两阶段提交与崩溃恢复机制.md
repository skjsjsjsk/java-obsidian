# 一 redo log + binlog

## redo log(物理日志)
- 是**InnoDB**引擎特有的
- 记录在哪个数据页做了什么修改
## binlog(逻辑日志)
- 是 **MySQL Server 层**实现的(所有引擎共用)
- 记录的是这个语句的原始逻辑(有两种模式，statement 格式的话是记sql语句， row格式会记录行的内容，记两条，更新前和更新后都有。)

## 如果不使用两阶段
- update 语句来做例子。假设当前 ID=2 的行，字段 c 的值是 0, +1操作

- **先写 redo log(commit) 后写 binlog**。
	- 假设在 redo log 写完，binlog 还没有写完的时候，MySQL 进程异常重启。redo log 写完之后，系统即使崩溃，仍然能够把数据恢复回来,恢复数据后c=1.但是binlog 没写,里面没有这条语句,如果使用binlog来恢复临时库的话,就会导致c=0,不一致

- **先写 binlog 后写 redo log**。
	- 因为redo log 还没写,崩溃恢复以后这个事务无效，所以这一行 c 的值是 0。(更新没成功),但是binlog里面已经记录了,之后用 binlog 来恢复的时候就多了一个事务出来，恢复出来的这一行 c 的值就是 1，不一致
	- 大概就是:binlog 已写入，但 redo log 尚未记录该事务，崩溃后主库无此修改，备库却会执行，导致主从不一致。

# 二 两阶段提交
- 由上述可知,需要两阶段提交,顺序为:**Redo Log (Prepare) → Binlog → Redo Log (Commit)**
保证了:
- 如果 binlog 写失败，事务不会提交（因为 redo log 仍是 prepare）
- 如果 redo log 写失败，binlog 也不会写（因为 binlog 是在 prepare 之后才写）

## 详细三步顺序
#### Prepare 阶段（InnoDB）
1. 执行 SQL，修改数据页（在内存 buffer pool 中） 
2. 生成 redo log 记录（包含 XID），写入 redo log buffer 
3. 根据 innodb_flush_log_at_trx_commit 决定是否刷盘： 
	- =1 → 立即 fsync 到磁盘（持久化） 
	- =0/2 → 可能还在 OS 缓存 
4. redo log 状态标记为 "prepare"

#### 写 binlog（Server 层）
1. 将本次事务的 SQL 操作写入 binlog cache 
2. 根据 sync_binlog 决定是否刷盘： 
	- =1 → 立即 fsync 到 binlog 文件（持久化） 
	- >1 → 多个事务后才刷 
3. binlog 中也包含相同的 XID，用于关联 redo log

**==此时，redo log（prepare）和 binlog 都已持久化到磁盘（当配置为 =1 时）==**

#### Commit 阶段（InnoDB）
1. Server 通知 InnoDB：“binlog 已写成功，可以提交” 
2. InnoDB 将 redo log 中该事务的状态从 "prepare" 改为 "commit" 
3. 再次确保 redo log 被刷盘（如果之前没刷） 
4. 事务正式完成

 ==**commit 阶段不是“再写一次数据”，而是“打一个已完成的标记”**==

# 三 崩溃恢复（Crash Recovery）机制
![[Pasted image 20251209195012.png]]
- 1.直接重做redo log里面的操作 (**快速恢复**这就叫做)
- 2.
	- (1)**首先**从 redo log 文件中读取所有事务记录。直到读取到一条
	  [XID=100, op=UPDATE ..., state=prepare]

	- (2)**然后**查找 binlog 中是否存在 XID=100 的事务
	- ==**(XID 是事务的唯一标识，确保崩溃恢复时能准确匹配 redo 和 binlog 中的同一事务)**==

- 如果找到 → 说明 binlog 已成功写入 → **可以安全提交**(这里是第二种,看第三步视为已提交后,重新执行一遍redo log)
- **最后**将事务“视为已提交”，并执行前滚（Redo)
	- InnoDB 将该事务 **当作已提交处理**
	- 开始 **前滚（Redo）**：把该事务的所有修改重做一遍
	- 修改应用到 buffer pool 中的数据页
	- 标记为脏页(内存中的数据与磁盘上的数据不一样)，后续由后台线程刷盘

- (3) 如果没找到 → 说明 binlog 写失败 → 必须回滚(这就是第三种,**撤销整个事务的所有修改**)
- 然后使用 undo log 回滚该事务:
	- 找到该事务对应的 undo log（记录了修改前的状态）
	- 从当前数据页开始，**逆向执行所有修改操作**
	- 把数据恢复到事务开始前的状态
