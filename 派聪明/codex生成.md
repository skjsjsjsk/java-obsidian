# 基于 RAG 的问答助手项目面试题与参考回答

### 2. 你要先背熟一版 1 分钟项目介绍

面试官大概率先问：

**Q：先简单介绍一下这个项目，你在里面主要负责什么？**

**A：** 这是一个面向企业内部文档检索和知识问答场景的 RAG 问答助手。整体上是一个 Spring Boot 单体后端加 Vue 前端的前后端分离项目，后端依赖 MySQL、Redis、Kafka、MinIO 和 Elasticsearch。  
我主要参与了三块：第一是文件上传、分片合并、异步解析和向量化链路；第二是基于 Elasticsearch 和 Embedding 的混合检索；第三是 WebSocket + 大模型流式输出的对话能力。另外我还参与了组织标签权限模型设计，用来控制不同用户对知识库的访问范围。

如果面试官继续问：

**Q：你认为这个项目最核心的主链路是什么？**

**A：** 核心链路是五步：

1. 用户上传文档，后端做分片上传和断点续传
2. 合并文件后投递 Kafka 异步任务
3. 消费者完成文档解析、切块、向量化和 ES 建索引
4. 用户提问时先做混合检索，召回相关文档片段
5. 把检索结果和历史上下文一起构造成增强 Prompt，再走大模型流式回答

## 一、模块化分层架构 + 多级存储体系 + 权限模型

你的简历原话是：

> 参与搭建模块化分层架构，整合 MySQL + Redis + MinIO + Elasticsearch 多级存储体系，分离元数据、对象与向量数据，并设计组织标签权限模型，实现精细化的访问控制。

### 面试官会怎么问

**Q1：你为什么要设计 MySQL、Redis、MinIO、Elasticsearch 这套多级存储，而不是只用一种存储？**

**A1：** 因为这几类数据的读写特征完全不一样：

- MySQL 主要存业务元数据，比如用户、组织标签、文件记录、文档状态、会话相关信息，适合强一致和关系查询
- Redis 主要存高频临时状态，比如上传进度、限流计数、聊天会话缓存，追求低延迟
- MinIO 主要存原始文件和合并后的对象，适合大文件对象存储
- Elasticsearch 主要存切块后的文本和向量，用于全文检索和向量检索

如果都放在 MySQL 里，一方面大文件和向量不适合，另一方面检索性能也会很差。所以我把“元数据、文件内容、检索数据”分开存，职责更清晰，也更容易扩展。

**Q2：你说“分离元数据、对象和向量数据”，具体是怎么分的？**

**A2：** 我的划分是这样的：

- 元数据放 MySQL，比如文件名、fileMd5、上传用户、组织标签、是否公开、状态、创建时间
- 原始文件和合并文件放 MinIO，比如 `merged/{fileMd5}`
- 切块后的文本以及向量索引放 Elasticsearch
- 切块文本在落 ES 之前，还会在 MySQL 保留一份结构化记录，方便做后续向量化和排查

这样做的好处是：

- 业务数据和检索数据解耦
- ES 重建索引时不影响文件源数据
- 文件权限和检索权限都可以基于 MySQL 元数据来约束

**Q3：你们为什么不把向量直接存 MySQL？**

**A3：** 主要是两个原因：

第一，向量检索本身就不是 MySQL 的强项，尤其是高维向量的近似最近邻检索，ES 原生支持 dense vector 和 KNN，更适合做召回。  
第二，我们还需要把关键词匹配和语义相似度结合起来做混合检索，ES 能把全文检索、过滤和向量检索统一起来，工程实现更顺。

**Q4：你设计的组织标签权限模型是怎么做的？**

**A4：** 核心是三类字段：

- `userId`
- `orgTag`
- `isPublic`

每个文档都绑定上传用户和组织标签，同时可以标记是否公开。查询时我会按下面的逻辑做权限过滤：

1. 用户自己的文档可以查
2. 公开文档可以查
3. 用户所属组织标签下的文档可以查

这样可以同时支持个人私有文档、组织共享文档和公开文档三种访问模式。

**Q5：如果一个用户有多个组织标签，查询怎么处理？**

**A5：** 查询时会先拿到这个用户的有效组织标签集合，然后把这个集合拼成过滤条件一起带到检索层。  
在 ES 查询里，本质上就是一个 `should` 条件集合，命中任意一个允许访问的标签即可。  
这样能支持一个用户同时属于多个部门或者多个项目组。

### 追问

**追问1：为什么权限过滤要下推到检索层，而不是先查出来再在 Java 里过滤？**

**回答：** 因为如果先召回再过滤，会有两个问题：

- 性能差，召回很多无效结果，浪费 ES 和网络资源
- 准确度差，过滤完之后 topK 可能不够，最终给模型的上下文质量下降

所以权限过滤最好在 ES 查询阶段就做掉，保证召回的结果一开始就是“用户可见的候选集”。

**追问2：如果组织层级是树状结构，比如父部门和子部门，你怎么处理？**

**回答：** 我的思路是先把用户的“有效标签集合”算出来，不只包含直接标签，也包含继承得到的可访问标签，然后缓存到 Redis。  
检索时直接用这个有效标签集合去过滤，避免每次查询都递归查组织树。

**追问3：如果 MinIO 挂了，但 MySQL 里还有文件元数据，会怎样？**

**回答：** 这时候文件元数据还在，但源文件不可读，异步解析或下载就会失败。  
工程上我会做两层保障：

- 上传成功后才更新文件状态，避免出现“只写元数据没写对象”的脏数据
- 消费端读取 MinIO 失败时走重试和死信队列，同时保留失败状态，方便人工补偿

## 二、Kafka 解耦上传、处理、向量化与检索 + 分片上传 + 断点续传

你的简历原话是：

> 使用 Kafka 解耦文件上传、处理、向量化与检索环节；实现分片上传与断点续传，结合 Redis 缓存文件状态、MinIO 存储内容，保障大文件可靠上传。

### 面试官会怎么问

**Q6：你为什么要引入 Kafka？直接上传完成后同步解析不行吗？**

**A6：** 同步解析的问题在于链路太长，用户体验和系统稳定性都会受影响。

上传完成后的后置步骤包括：

- 文件合并
- 文档解析
- 文本切块
- 调用 Embedding 模型
- 写入 Elasticsearch

这些步骤都比较重，如果同步做：

- HTTP 请求会很长，容易超时
- 大文件和高并发时会拖垮应用线程
- 外部 Embedding 服务一旦抖动，上传接口就直接受影响

所以我把上传成功和后续处理解耦：上传接口只负责把文件可靠落盘，后续解析和向量化交给 Kafka 异步消费。

**Q7：你们的上传链路具体是怎样的？**

**A7：** 完整流程是：

1. 前端把大文件切成多个分片
2. 后端接收每个分片，并校验文件类型和分片参数
3. 分片内容写入 MinIO 的 `chunks/{fileMd5}/{chunkIndex}`
4. Redis 用 bitmap 记录每个分片是否已上传，支持秒级查询进度
5. 全部分片上传完成后，调用合并接口
6. 后端通过 MinIO 的 compose 能力把分片合并成 `merged/{fileMd5}`
7. 然后发送 `FileProcessingTask` 到 Kafka
8. 消费者异步解析、向量化、建索引

**Q8：你说断点续传是怎么实现的？**

**A8：** 断点续传的关键是“服务端记住哪些分片已经成功上传”。  
我这里用了 Redis bitmap：

- key 类似 `upload:{userId}:{fileMd5}`
- 第 n 个 bit 代表第 n 个分片是否上传成功

当前端重新发起上传时，先调状态接口，服务端把已上传分片列表返回给前端，前端只补传缺失的分片。

**Q9：为什么 Redis bitmap 比直接存一个 List 更适合？**

**A9：** bitmap 更省空间，也更适合这种“某个索引位是否完成”的场景。

- 一个分片只占 1 bit
- 查询某个分片是否上传成功是 O(1)
- 统计和扫描也方便

对大文件来说，如果分片很多，bitmap 的空间效率明显优于直接存字符串列表或 JSON。

**Q10：Kafka 解耦后，如何保证消息可靠处理？**

**A10：** 我会从三层考虑：

1. 生产端尽量保证消息可靠投递，比如开启幂等、合理设置 ack
2. 消费端失败要能重试
3. 重试仍失败的消息进入死信队列，避免一直阻塞主消费组

这个项目里文件处理任务失败后会走重试，超过阈值进入 DLT，方便后续排查和补偿。

### 追问

**追问1：如果 Kafka 消费者重复消费了同一个文件任务怎么办？**

**回答：** 我会做幂等控制，幂等键可以用 `fileMd5`。  
因为同一个文件的解析、切块和索引结果理论上应该是可覆盖或者可检测重复的，所以消费者侧要先检查：

- 这个文件是否已经完成解析
- 对应 ES 索引是否已经存在
- 当前状态是否已经是完成态

如果已经完成，就直接跳过，避免重复向量化和重复索引。

**追问2：为什么用 Kafka，不用 RabbitMQ？**

**回答：** 这个场景更偏“吞吐型异步任务流水线”，不是复杂路由场景。  
Kafka 对批量异步处理、顺序日志、重试和死信都比较友好，后续如果还要扩展更多消费方，比如审计、埋点、分析任务，也更自然。  
如果只是简单异步通知，RabbitMQ 也能做，但在这个项目里 Kafka 更适合文件处理这种链路。

**追问3：如果分片已经在 Redis 里标记成功了，但 MinIO 里其实没写进去，怎么办？**

**回答：** 这个属于状态不一致问题，所以服务端不能只信 Redis。  
更稳妥的做法是：

- 先确保 MinIO 写成功
- 再标记 Redis bitmap
- 合并前最好还要再校验一次分片记录和对象是否存在

这样即使 Redis 状态脏了，最终也不会错误合并。

**追问4：为什么合并放在 MinIO 做，不落到本地磁盘？**

**回答：** 因为对象最终本来就要落在 MinIO，直接在对象存储层合并更省本地磁盘和应用服务器 I/O，也更适合多实例部署。  
如果走本地磁盘，应用实例还要承担大文件中转和清理成本，部署复杂度会更高。

## 三、WebSocket + DeepSeek Stream API + 打字机式对话体验

你的简历原话是：

> 基于 WebSocket 建立长连接，集成 DeepSeek 大模型 Stream API 实现流式响应，为用户提供“打字机式”逐字生成的对话体验。

### 面试官会怎么问

**Q11：为什么你这里选 WebSocket，不选 SSE？**

**A11：** 因为这个场景不只是服务端单向推送，还包含一些双向交互需求：

- 用户发消息
- 服务端持续回流式结果
- 心跳保活
- 停止生成
- 可能还要做会话级控制命令

SSE 更适合单向流式推送，但 WebSocket 对这种“长期连接 + 双向消息 + 控制指令”的场景更灵活，所以我选了 WebSocket。

**Q12：流式响应具体是怎么打通的？**

**A12：** 流程是这样的：

1. 前端通过 WebSocket 建立连接
2. 后端收到用户消息后，先检索知识库并构造 Prompt
3. 再调用大模型的 Stream API
4. 模型每返回一个 chunk，后端就立即通过 WebSocket 转发给前端
5. 前端按 chunk 逐步拼接展示，所以用户看到的就是“打字机式”输出

这样能明显降低用户感知延迟，哪怕完整回答要几秒，用户也会觉得系统已经在工作。

**Q13：WebSocket 的鉴权你怎么做的？**

**A13：** 建立连接时会把用户 token 带上，后端在握手或建连后第一时间校验 JWT。  
校验通过后，才把这个 session 跟 userId 绑定起来；如果 token 无效，就直接关闭连接。  
这样可以保证 WebSocket 通道本身也是受认证保护的。

**Q14：如果用户点击“停止生成”，你怎么实现？**

**A14：** 我会给 WebSocket 定义一个内部控制消息，比如 `stop`。  
服务端收到后，为当前 session 设置一个停止标记。  
后续在模型流式 chunk 回调里，发现这个标记就不再继续向前端推送，同时清理会话状态。

**Q15：为什么流式返回对用户体验更好？**

**A15：** 因为它优化的是“首字节时间”。

- 非流式：用户要等完整答案生成完才看到内容
- 流式：用户在几百毫秒到 1 秒内就能看到第一段输出

在问答系统里，用户对“是否马上有反馈”非常敏感，所以流式输出的体验提升很明显。

### 追问

**追问1：WebSocket 怎么保活？**

**回答：** 通常做法是心跳机制，比如前端定期发 `ping`，服务端返回 `pong`。  
如果一段时间收不到心跳或发送失败，就主动断开并清理 session，避免僵尸连接。

**追问2：如果前端断网重连，会话上下文还在吗？**

**回答：** 会话上下文最好不要只放在内存里，应该放 Redis 或数据库。  
这样即使 WebSocket 断开，重连后也能根据用户当前会话 ID 把历史记录恢复出来，保证多轮对话连续。

**追问3：如果同时有很多用户都在流式对话，会不会把线程池打满？**

**回答：** 有这个风险，所以要控制两件事：

- 模型请求和会话监控不能无限制开线程
- 要有并发上限和降级策略

这个项目里聊天监控用的是独立线程池，工程上还要结合限流，避免高并发下 WebSocket 和模型调用同时把资源吃满。

## 四、RAG 流程设计 + 增强 Prompt + 上下文理解

你的简历原话是：

> 设计检索增强生成（RAG）流程，通过用户提问与检索结果生成增强 Prompt，并结合上下文语义理解提升问答准确度。

### 面试官会怎么问

**Q16：你先说一下你理解的 RAG 是什么。**

**A16：** RAG 就是“先检索，再生成”。  
先从知识库里找和当前问题最相关的内容片段，再把这些片段作为上下文给大模型，让模型基于检索结果回答。  
它的核心价值是减少幻觉，让回答更贴近企业内部文档，而不是只依赖模型参数里的通识知识。

**Q17：你们项目里的 RAG 流程具体怎么做的？**

**A17：** 大致分成离线和在线两部分。

离线部分：

1. 文档上传
2. 文档解析
3. 文本切块
4. 生成向量
5. 写入 ES 索引

在线问答部分：

1. 用户提问
2. 用 query 做混合检索
3. 拿到 topK 文本片段
4. 把问题、检索结果、系统规则和历史对话拼成增强 Prompt
5. 调大模型流式生成答案

**Q18：你说增强 Prompt，具体增强了什么？**

**A18：** 主要增强了三部分：

- 系统规则：要求模型优先依据检索结果回答，不足时明确说明
- 检索上下文：把召回的文档片段按编号、文件名、页码等信息组织起来
- 多轮历史：把最近几轮用户问题和助手回答一起带上，增强上下文连续性

这样模型拿到的不是一个裸问题，而是“问题 + 可用证据 + 对话上下文”。

**Q19：RAG 怎么提升准确度？**

**A19：** 核心是让模型回答时“有依据”。

- 没有 RAG 时，模型只能靠参数记忆，容易幻觉
- 有 RAG 时，模型能看到业务文档里的实际内容，回答更贴合知识库

另外我会控制检索片段长度和数量，尽量保证进入 Prompt 的上下文是高相关、可读、不过长的，这对准确率也有帮助。

**Q20：如果检索不到结果怎么办？**

**A20：** 这种情况不能硬答。  
我会在 Prompt 里明确告诉模型：如果没有足够证据，就回答“暂无相关信息”或者说明当前知识库里没有命中结果。  
这样比模型瞎编一个答案更可信。

### 追问

**追问1：topK 你怎么定？**

**回答：** topK 不是越大越好。  
太小容易漏召回，太大又会引入噪声、增加上下文长度和模型成本。  
我一般会结合文档切块粒度、问题类型和模型上下文窗口来调。实践里可以先从 3 到 10 做 A/B，对准确率和延迟一起看。

**追问2：多轮对话里，历史消息全部拼进去吗？**

**回答：** 不会。  
因为历史太长会占上下文窗口，还可能把旧问题噪声带进来。  
通常只保留最近几轮对话，或者做摘要压缩，再加上当前问题的检索结果。

**追问3：RAG 一定能降低幻觉吗？**

**回答：** 不一定，RAG 只是降低风险，不是彻底消除。  
如果检索阶段召回错了、切块质量差、Prompt 拼得乱，模型仍然可能胡说。  
所以 RAG 的关键不是“加个检索”就完了，而是检索质量、上下文组织和回答约束都要做好。

**追问4：你怎么让回答能引用到来源？**

**回答：** 做法是把检索结果编号化，比如 `[1] [2] [3]`，并把文件名、页码、片段内容一起传给模型。  
回答时模型可以按编号引用。  
服务端同时保存“编号 -> 文档信息”的映射，这样前端点引用时还能反查到具体文件和页码。

## 五、Elasticsearch + IK 分词 + Embedding + KNN + BM25 混合检索

你的简历原话是：

> 集成 Elasticsearch + IK 分词器构建多格式文档索引，结合通义千问 Embedding 模型实现 2048 维向量转换，结合 KNN 向量召回与 BM25 重排序，实现关键词匹配与语义相似度的混合检索。

### 面试官会怎么问

**Q21：为什么你们要做混合检索，而不是只做向量检索？**

**A21：** 因为关键词检索和向量检索各有优缺点。

- BM25 更擅长精确关键词，比如类名、产品名、版本号、专有名词
- 向量检索更擅长语义相似，比如用户换一种说法提问时仍然能召回相关内容

只做 BM25，容易漏掉语义相近但字面不一致的内容。  
只做向量检索，专有名词和精确匹配又可能不稳定。  
所以我把两者结合起来，先做向量召回，再结合文本匹配重排，效果更稳。

**Q22：你们的混合检索是怎么落地的？**

**A22：** 大致思路是：

1. 用户问题先生成 query 向量
2. 用 KNN 在 ES 里做向量召回，拿到一批语义相近候选
3. 同时要求文本字段也要匹配 query，保证结果不是纯语义漂移
4. 最后再用 BM25 做二阶段 rescore，对文本相关度高的结果提权

这样既保留语义召回能力，又兼顾关键词精确性。

**Q23：为什么要用 IK 分词器？**

**A23：** 因为中文分词质量会直接影响全文检索效果。  
默认分词对中文不够友好，IK 分词器在中文场景下更适合做词项切分，尤其是企业内部文档里很多中文标题、名词和短语，分词质量会明显影响 BM25 召回效果。

**Q24：2048 维向量是怎么来的？**

**A24：** 这个维度是由所选 Embedding 模型决定的。  
项目里接的是通义千问兼容接口，对应模型输出 2048 维向量，所以 ES mapping 和向量字段维度也要跟着保持一致。  
这里不能写错，否则向量入库或者检索阶段都会报维度不匹配。

**Q25：为什么是“KNN + BM25 重排”这个组合？**

**A25：** 因为 KNN 适合做高召回，BM25 适合做高精排。

- KNN 先从大范围文档中快速找出语义接近的候选
- BM25 再根据关键词匹配程度做重排序

这样比单纯用其中一种效果更平衡，也更符合 RAG 场景对“既要召回全，又要结果准”的要求。

### 追问

**追问1：如果检索词里是版本号、接口名、报错码，向量检索还靠谱吗？**

**回答：** 这种场景往往是 BM25 更靠谱，所以混合检索就有价值。  
因为版本号、报错码、表名、类名这些内容属于强字面信息，关键词匹配比纯语义相似更稳定。

**追问2：为什么不在向量召回后自己在 Java 里重排？**

**回答：** 能做，但不划算。  
ES 本身就支持 rescore，把二阶段排序下推到检索引擎层，能减少应用层中转和排序成本，整体更高效。

**追问3：多格式文档索引怎么做的？PDF、Word、TXT 都一样吗？**

**回答：** 不一样，但会统一落成“纯文本块”。  
前面会先用解析组件把 PDF、Word、TXT 等格式抽取为文本，再按统一的切块规则入库。  
这样检索层不需要关心源文件是什么格式，只关心文本内容、页码、向量和权限字段。

**追问4：切块怎么做才更合理？**

**回答：** 切块太大，会导致一个 chunk 里主题太多，召回不精准；切块太小，又会丢上下文。  
所以我会按段落、句子语义边界切分，而不是简单固定字数硬切。  
在中文场景里，还可以借助分词工具做更自然的切分。

**追问5：如果 ES 里的向量维度配错了，会出现什么问题？**

**回答：** 会直接导致写入失败或者查询失败。  
因为向量字段的 `dims` 必须和 Embedding 模型输出维度完全一致。  
所以模型切换时，除了改接口地址和模型名，还要同步检查索引 mapping。

## 十三、分片上传、断点续传、Kafka 作用专项准备

### 1. 面试官会先怎么问

**Q：你详细讲讲你们项目里的分片上传、断点续传，以及 Kafka 在这里起什么作用。**

### 2. 一版完整回答

**A：** 这个项目的大文件上传是按 chunk 做的。前端先把文件切片并计算 `fileMd5`，每个分片单独上传。服务端接收分片后，会先做文件类型和参数校验，然后把 chunk 写到 MinIO，对应路径类似 `chunks/{fileMd5}/{chunkIndex}`。  
同时，服务端会用 Redis bitmap 记录每个分片的上传状态，并在数据库里保存文件记录和 chunk 元数据，比如 `chunkIndex`、`chunkMd5`、`storagePath`。  
断点续传的核心是客户端重新上传前先查状态接口，服务端根据 Redis bitmap 返回已上传的 chunk 列表，前端只补传缺失分片。  
当全部分片上传完成后，服务端再调用 MinIO 的合并能力，把所有 chunk 合并成完整文件，比如 `merged/{fileMd5}`。  
Kafka 的作用不是做上传，而是解耦上传完成后的后处理链路。也就是文件合并成功后，只发送一个文件处理任务到 Kafka，由消费者异步完成文档解析、切块、向量化和 Elasticsearch 建索引。这样做的好处是降低上传请求时延、削峰填谷，并且能通过重试和死信队列提高处理链路的可靠性。

## 十四、分片上传你要讲清的点

### Q1：为什么要分片上传？

**A：** 因为大文件如果一次性上传，会有几个明显问题：

- 请求时间太长，容易超时
- 网络抖动时，失败要整文件重传
- 浏览器和服务端的内存压力都更大
- 上传后还要继续解析、向量化，整条请求链路会非常重

所以分片上传的本质，是把“大文件传输”拆成多个可恢复的小任务，提高稳定性和可恢复性。

### Q2：你们的分片上传流程是什么？

**A：** 流程可以拆成 6 步：

1. 前端对文件切片，并计算整个文件的 `fileMd5`
2. 每个 chunk 单独调用上传接口，携带 `fileMd5`、`chunkIndex`、`totalSize`、`fileName` 等参数
3. 服务端把 chunk 写入 MinIO
4. 用 Redis 记录该 chunk 已上传
5. 用数据库保存文件记录和 chunk 元数据
6. 所有 chunk 上传完成后，再触发合并

### Q3：为什么还要保存 chunk 元数据，Redis 记录状态不就够了吗？

**A：** Redis 主要解决的是“快”，数据库解决的是“可追踪”。

- Redis bitmap 适合快速判断某个 chunk 是否上传成功
- 数据库里的 chunk 元数据适合做审计、补偿、排查和合并校验

只靠 Redis 状态不够稳，因为 Redis 更偏缓存和运行时状态。

## 十五、断点续传你要讲清的点

### Q4：断点续传的核心原理是什么？

**A：** 断点续传的核心不是记住文件传到了第几个字节，而是记住“哪些 chunk 已经传成功了”。  
也就是说，这个项目是按 chunk 粒度做续传，而不是按字节流做续传。

### Q5：你们项目里断点续传怎么实现？

**A：** 我们用 Redis bitmap 记录上传状态。

- key 形如 `upload:{userId}:{fileMd5}`
- 第 `chunkIndex` 位如果是 `1`，就表示该分片已经上传成功

客户端在重新上传之前，先请求上传状态接口。服务端返回已上传的分片列表后，前端只补传缺失的 chunk，这样就实现了断点续传。

### Q6：为什么 Redis bitmap 比 List 或 JSON 更适合？

**A：** 因为 bitmap 更节省空间，也更适合这种“某一位是否完成”的场景。

- 一个 chunk 只占 1 bit
- 查询单个分片是否上传完成是 O(1)
- 批量读取状态也方便

所以在大文件、多分片场景下，bitmap 是一种很轻量的状态存储方式。

### Q7：如果客户端断网、刷新页面甚至关闭浏览器，还能续传吗？

**A：** 只要服务端保存的状态还在，就可以续传。  
因为续传依赖的是服务端状态，不依赖前端内存状态。  
客户端重新发起上传时，先查询服务端记录的已上传 chunk，再从断点处继续补传。

### Q8：这套断点续传方案有没有局限？

**A：** 有。当前这套实现主要依赖 Redis bitmap 做状态记录。  
如果 Redis 状态丢了，但 MinIO 里的 chunk 还在，理论上文件内容没丢，但客户端可能无法精准知道哪些分片已上传，可能会出现重复上传。

更稳的优化方案是：

- 状态查询时除了 Redis，还可以结合数据库 chunk 元数据
- 甚至可以在必要时回退扫描 MinIO 对象

这样可以进一步提升续传准确性。

## 十六、Kafka 作用你要讲清的点

### Q9：Kafka 在这里到底解决了什么问题？

**A：** Kafka 解决的是“上传完成后的重任务不能和用户请求强耦合”的问题。

文件上传完成后，后面还有很多重操作：

- 文档解析
- 文本切块
- Embedding 向量化
- Elasticsearch 建索引

如果这些都放在上传接口里同步做，会导致：

- 请求耗时长
- 容易超时
- 高并发时拖垮服务线程
- 外部依赖抖动直接影响上传接口

所以我们在文件合并完成后，只发送一个文件处理任务到 Kafka，由消费者异步完成后续处理。

### Q10：Kafka 的好处有哪些？

**A：** 可以从四点讲：

1. 解耦：上传链路和解析/向量化链路分开
2. 削峰填谷：高峰时任务先进队列，后端慢慢消费
3. 失败隔离：Embedding 或 ES 抖动不会直接拖垮上传接口
4. 可恢复：结合重试和死信队列，失败任务可以继续处理或人工补偿

### Q11：为什么不直接同步解析？

**A：** 因为上传接口应该优先保证“文件可靠落盘”，而不是在一个 HTTP 请求里完成所有后续处理。  
后面的解析、切块、向量化和索引构建都属于重任务，更适合走异步消费链路。

### Q12：Kafka 失败了或者消费者处理失败了怎么办？

**A：** 这个项目里消费者失败后会抛异常，由 Kafka 的错误处理器接管。  
配置上支持：

- 自动重试
- 固定退避
- 死信队列 DLT

也就是说，处理失败不会直接吞掉，而是先重试；多次失败后进入死信队列，方便排查和补偿。

## 十七、你要能答出的进阶追问

### 追问1：Kafka 会不会重复消费？你怎么处理？

**A：** 会，所以消费者要做幂等。  
在文件处理场景里，可以用 `fileMd5` 作为天然幂等键。  
消费前先检查：

- 文件是否已经完成处理
- 向量或索引是否已经构建完成
- 当前状态是否已经是完成态

如果已经完成，就直接跳过，避免重复解析和重复建索引。

### 追问2：为什么 Kafka 比 RabbitMQ 更适合这里？

**A：** 这个场景是典型的异步任务流水线，更关注吞吐、顺序日志和消费恢复，而不是复杂路由。  
Kafka 更适合这种“文件上传完成后进入异步处理链路”的场景。  
如果只是简单通知，RabbitMQ 也能做，但从任务流水线角度看 Kafka 更顺。

### 追问3：如果某个 chunk 在 Redis 里已经标记成功，但 MinIO 实际没写进去，怎么办？

**A：** 服务端不能只信 Redis。  
更稳的顺序应该是：

1. 先写 MinIO
2. 写成功后再标记 Redis bitmap
3. 合并前再校验 chunk 元数据和 MinIO 对象是否都存在

这样可以避免缓存状态和对象存储状态不一致。

### 追问4：为什么 chunk 合并放在 MinIO，而不是放到本地磁盘？

**A：** 因为文件最终本来就要落在对象存储里。  
直接在 MinIO 层完成合并，可以减少本地磁盘中转和应用服务器 I/O 压力，也更适合多实例部署，不会把文件状态绑死在某一台机器上。

## 十八、最适合校招面试的 1 分钟口述版

**Q：你用 1 分钟讲一下分片上传、断点续传和 Kafka。**

**A：** 我们项目里大文件上传是按 chunk 做的，前端先切片并计算 `fileMd5`，每个分片单独上传。服务端把 chunk 写到 MinIO，同时用 Redis bitmap 记录每个分片的上传状态，并在数据库保存文件和分片元数据。  
断点续传的关键是客户端重新上传前先查状态接口，服务端返回已经上传成功的 chunk 列表，前端只补传缺失分片。  
当全部分片上传完成后，服务端再把所有 chunk 合并成完整文件。  
Kafka 的作用是在文件合并成功后，把后续的文档解析、切块、向量化和 Elasticsearch 建索引异步化，避免上传接口同步执行重任务。这样能降低请求时延，也能通过重试和死信队列提高处理链路的可靠性。

# 项目主链路

## 入库链路

1. 文件上传入口在 `UploadController.uploadChunk`。前端先按 5MB 对大文件分片，携带 `fileMd5`、`chunkIndex`、`totalSize`、`fileName`、`orgTag`、`isPublic` 等参数逐片上传。控制层先做文件类型校验、组织标签补全、组织上传大小限制校验。
    
2. 分片真正落地由 `UploadService.uploadChunk` 完成。第一次上传时会先在 MySQL 的 `file_upload` 表创建文件记录，保存 `fileMd5`、`fileName`、`totalSize`、`userId`、`orgTag`、`isPublic` 等元数据。
    
3. 断点续传依赖 Redis bitmap。服务端使用 `upload:{userId}:{fileMd5}` 作为 key，按 `chunkIndex` 标记分片上传状态；分片内容本身写入 MinIO，路径是 `chunks/{fileMd5}/{chunkIndex}`；同时把 `chunkMd5`、`storagePath` 等分片元数据写入 MySQL 的 `chunk_info` 表。前端查询上传状态时，后端通过 `getUploadedChunks` 返回已上传分片列表，只补传缺失分片。
    
4. 所有分片上传完成后，前端调用 `UploadController.mergeFile`。控制层先校验文件归属和分片完整性，再调用 `UploadService.mergeChunks` 使用 MinIO `composeObject` 将所有分片合并成 `merged/{fileMd5}`。合并成功后删除原始分片对象、清理 Redis 上传标记，并把 `file_upload.status` 更新为已完成。
    
5. 文件合并后，系统会先同步做一次 Embedding 用量预估。`UploadController.mergeFile` 通过 `uploadService.getMergedFileStream` 获取合并后的文件流，再调用 `ParseService.estimateEmbeddingUsage` 估算预计 token 数和预计 chunk 数，并把结果回写到 `file_upload` 表，供前端展示和配额控制使用。
    
6. 真正的知识入库不在上传请求里同步执行，而是由 `UploadController.mergeFile` 构造 `FileProcessingTask` 投递到 Kafka。这样把“上传成功”和“文档解析、向量化、建索引”解耦，避免上传接口阻塞在重任务上，同时也便于削峰、重试和死信处理。
    
7. Kafka 消费端是 `FileProcessingConsumer.processTask`。消费者收到任务后先从 MinIO 预签名 URL 下载完整文件，再调用 `ParseService.parseAndSave` 做文档解析和切块。这里对非 PDF 文档使用 Tika 流式解析，对 PDF 使用 PDFBox 按页提取文本，并通过段落、句子、分词三级策略切分成适合 RAG 的文本块。
    
8. 切块结果先写入 MySQL 的 `document_vectors` 表，保存 `fileMd5`、`chunkId`、`textContent`、`pageNumber`、`anchorText`、`userId`、`orgTag`、`isPublic` 等字段。随后 `VectorizationService.vectorizeWithUsage` 从库里取出 chunk 文本，调用 `EmbeddingClient` 访问 Embedding 模型生成向量，再通过 `ElasticsearchService.bulkIndex` 批量写入 ES 的 `knowledge_base` 索引，完成知识入库。
    
9. 这一段链路涉及的核心模块和技术栈包括：`Spring MVC`、`Spring Data JPA`、`MySQL`、`Redis bitmap`、`MinIO`、`Kafka`、`Apache Tika`、`PDFBox`、`HanLP`、`Embedding API`、`Elasticsearch`。
    

## 问答展示链路

1. 前端通过 WebSocket 建立会话，入口由 `WebSocketConfig` 注册到 `/chat/{token}`，实际连接处理在 `ChatWebSocketHandler.afterConnectionEstablished`。后端会校验 JWT，提取 `userId`，把 WebSocket 会话和用户绑定起来。
    
2. 用户发送问题后，由 `ChatWebSocketHandler.handleTextMessage` 接收消息，并转发给 `ChatHandler.processMessage`。`ChatHandler` 会先做限流校验，再从 Redis 中读取当前会话 ID 和历史对话记录，准备多轮问答上下文。
    
3. 检索阶段由 `HybridSearchService.searchWithPermission` 完成。系统先调用 `EmbeddingClient` 为用户问题生成查询向量，然后向 Elasticsearch 发起混合检索请求：`knn` 负责语义召回，`query.must(match)` 负责关键词匹配，`query.filter` 负责权限过滤，`rescore` 再用 BM25 风格做二阶段重排。
    
4. 权限控制贯穿在检索链路中。可见范围包括三类：用户自己的文档、公开文档、用户所属组织标签下的文档。这样保证最终召回的 chunk 与上传时写入的 `userId`、`orgTag`、`isPublic` 元数据一致，满足多租户和组织级知识隔离要求。
    
5. 检索结果返回后，`ChatHandler.buildContext` 会把命中的 chunk 组装成 RAG 上下文，并建立引用编号到文件名、页码、锚点文本、chunkId 的映射。这样模型回答时不仅有检索证据，前端后续也能展示引用出处。
    
6. 大模型调用由 `LlmProviderRouter.streamResponse` 完成。它会从 `ModelProviderConfigService` 读取当前激活的 LLM provider，把系统规则、检索上下文、多轮历史和当前问题拼装成 OpenAI 兼容格式的 `messages`，然后调用 `/chat/completions` 的流式接口。当前项目既支持云端模型，也保留了本地兼容 provider 的切换能力。
    
7. 模型流式返回后，`LlmProviderRouter` 逐块解析响应，把文本 chunk 回调给 `ChatHandler`。`ChatHandler` 一边累积完整回答，一边通过 WebSocket 向前端持续推送 `{"chunk":"..."}` 格式的数据，前端据此实现“打字机式”答案展示。
    
8. 当模型输出结束后，`ChatHandler` 会发送 completion 消息，通知前端本轮回答完成；同时把用户问题、完整回答和引用映射写回 Redis 会话历史，用于下一轮多轮问答。这样用户看到的是流式回答，系统内部维护的是“检索增强 + 多轮上下文”的完整问答链路。
    
9. 这一段链路涉及的核心模块和技术栈包括：`Spring WebSocket`、`Redis`、`Elasticsearch Hybrid Search`、`Embedding API`、`LLM Stream API`、`WebClient`、`JWT`、`RAG Prompt`、`多轮会话管理`。
